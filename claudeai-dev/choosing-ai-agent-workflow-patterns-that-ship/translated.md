# 选择真正能交付的AI智能体工作流模式

大多数团队在智能体项目上的失败并非因为模型质量，而是过早选择了错误的工作流模式：过度编排、组件过多，且复杂化缺乏明确理由。Anthropic最近发布的智能体常见工作流模式指南很有用，但本文针对构建生产系统的开发者进行了重构。

## 大多数团队忽视的核心原则

**从满足质量要求的最简模式开始**。实践中这意味着：

1. 首先尝试单智能体调用
2. 仅当依赖关系强制排序时才添加顺序步骤
3. 仅当任务真正独立且延迟敏感时才添加并行工作器
4. 仅当质量提升可量化时才添加评估器-优化器循环

如果跳过这个顺序，通常会在延迟、token成本和调试难度上付出代价。

## 模式1：顺序工作流

当步骤B依赖于步骤A时使用顺序工作流。可将其视为管道：

```
提取 → 转换 → 验证 → 路由
```

**适用场景**  
- 具有硬依赖关系的多阶段任务
- 每个阶段添加不同类型价值的数据管道
- 草稿→评审→润色流程

**不适用场景**  
- 单个智能体已能可靠完成任务时
- 步骤是虚假分离且可以合并时

**开发者权衡**  
- 更高延迟
- 更好的控制力和各阶段可观测性

**快速测试**  
如果移除某个阶段后输出质量几乎不变，则该阶段很可能不必要。

## 模式2：并行工作流

当子任务独立且可同时运行时使用并行工作流。用分布式系统术语来说就是扇出/扇入：

```
扇出 → 分发到多个智能体
扇入 → 使用聚合策略汇总
```

**适用场景**  
- 多维度评估（安全性、风格、正确性）
- 按类别进行的安防/代码审查
- 使用独立视角的文档分析

**不适用场景**  
- 当智能体需要共享动态演化的上下文时
- 没有稳健的聚合策略时
- API配额/并发限制抵消速度收益时

**开发者权衡**  
- 更快的完成速度
- 更高成本 + 聚合复杂性

**快速测试**  
如果聚合逻辑比每个并行工作器更复杂，说明过度并行化了。

## 模式3：评估器-优化器工作流

当初稿质量不足且质量标准明确时使用评估器-优化器模式。

**结构**：
1. 生成器产出草稿
2. 评估器根据具体标准评分
3. 生成器修订
4. 达到阈值或最大迭代次数时停止

**适用场景**  
- 具有严格标准的代码生成
- 语气和精确度至关重要的高风险文档/通信
- 带有安防/性能检查的SQL/查询生成

**不适用场景**  
- 需要即时响应的实时交互
- 评估器无法一致应用主观标准的任务
- 确定性工具已能解决验证的场景（代码检查器、模式验证器）

**开发者权衡**  
- 潜在的质量大幅提升
- 更多token消耗、更高延迟和无尽微迭代风险

**快速测试**  
如果第3次迭代后改进微乎其微，应降低迭代上限并继续推进。

## 实用决策树

在规划文档或设计评审中使用：

1. 单个智能体能可靠解决吗？→ 是则停止
2. 存在硬步骤依赖吗？→ 是则采用顺序模式
3. 子任务独立且延迟敏感吗？→ 是则添加并行分支
4. 首轮质量低于可衡量标准吗？→ 是则在需要处添加评估器-优化器

这能确保复杂度与实际需求成正比。

## 必须预先定义的故障处理

无论采用何种模式，上线前需定义：

- 各阶段重试策略
- 超时预算
- 回退行为
- 部分故障策略
- 矛盾输出解决规则

没有这些，你的“智能体架构”只是演示版。

## 编排前的基准测试

Anthropic的指导方向正确：先用简单方法建立基准。对团队而言，这意味着至少追踪：

- 任务成功率
- 延迟p50/p95
- 每次成功运行的token成本
- 人工修正率

仅当某项指标显著改善时，才保留增加的编排复杂度。

## 有效的模式组合

模式是构建块，而非互斥选择。两种常见组合：

1. **顺序+并行**：包含一个或两个并行密集型阶段的顺序管道
2. **并行+评估器**：多个生成器或评审器为集中评估环节提供输入

除非能用指标解释质量/成本影响，否则避免嵌套模式。

## 最终要点

工作流模式不是架构表演，而是成本/质量控制。要构建可交付且可维护的系统：

1. 从简单开始
2. 仅在瓶颈被证实时添加结构
3. 衡量每次复杂度增加

这才是将智能体实验转化为生产工作流的方法。
