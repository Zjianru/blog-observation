# 通过测试-衡量-优化构建更智能的Agent技能

来源：https://claudeai.dev/blog/build-better-agent-skills-with-tests

---

大多数Agent技能失败的原因都很乏味：我们修改提示词，重新运行一次，就宣称它“更好了”。

Anthropic最新的技能创建器更新推动了一种更工程化的循环模式：**先测试，衡量行为表现，再优化改进**。如果你正在构建内部Agent工作流，这才是真正重要的转变。

本文将官方公告重构为可供开发者每周执行的工作流程。

## 核心理念：停止盲目调整提示词​

本次更新不仅是“更好的提示词生成”，而是一个质量闭环：

  1. 预先定义可衡量的行为标准
  2. 根据行为标准生成测试套件
  3. 对新旧版本进行效果评估
  4. 利用评估反馈消除冲突指令和无效指令
  5. 基于证据迭代优化，而非主观感觉

如果这听起来像是提示词的持续集成流程，那就完全抓住了重点。

* * *

## 技能创建器的实际变化（开发者视角）​

根据官方公告，以下实际变化值得关注：

  * **先定标准提示词再实施** 你可以编写（或协同创建）代表真实任务的具体提示词，这些将成为你的基准案例。

  * **`test-creator`实现自动化技能测试** 技能创建器可根据需求自动生成测试用例，无需手动编写每个测试场景。

  * **集成化技能评估器** 评估反馈会突出显示指令冲突、功能重叠和薄弱环节，帮助你完善技能规范。

  * **基于质量信号的优化机制** 你需要循环迭代多个版本，根据测试输出结果衡量改进效果，而非依赖“感觉更好”的主观判断。

这种模式更适合工程师，因为它将技能视为可版本化的工件，而非一次性提示词。

* * *

## 可立即采用的实用工作流程​

### 步骤1）像定义API行为那样编写验收标准​

在编辑任何技能文本之前，以严格格式编写预期行为：

  * 输入格式规范
  * 输出结构定义
  * 必须执行规则
  * 禁止执行规则
  * 异常处理机制（上下文缺失时的应对方案）

如果你的标准模糊，你的测试也会模糊。

### 步骤 2) 从真实事件构建黄金提示集​

使用团队的真实日志、工单或请求。包括：

  * 正常情况
  * 嘈杂/模糊情况
  * 上下文缺失情况
  * 超出范围情况

这些就是你的回归测试集。保持其规模小但信息量高。

### 步骤 3) 生成并运行测试​

使用 Skill Creator + `test-creator` 生成结构化的技能测试。然后同时运行：

  * 当前生产环境技能
  * 候选更新技能

在相同测试集上比较输出质量。

### 步骤 4) 运行评估器反馈并精简指令​

寻找重复出现的失败模式：

  * 矛盾的指令
  * 过于宽泛的指令
  * 隐藏的假设
  * 输出格式不稳定

每次迭代仅优化一两个变量，以便能归因改进效果。

### 步骤 5) 仅当指标改善时才推广​

不要因为一个例子看起来不错就发布。仅当满足以下条件时才推广：

  * 在整个测试集上的通过率有所提高
  * 失败模式减少而非转移
  * 在边缘情况下输出格式保持稳定

## 团队通常遇到的瓶颈​

这些是我们在真实开发团队中常见的失败模式：

  * **单一技能包含过多规则** 一个试图同时完成摘要、规划、分类和政策解释的巨型技能通常会迅速退化。

  * **没有版本化的测试数据** 如果黄金提示集没有版本控制，你就无法信任趋势线。

  * **缺乏拒绝行为** 技能应定义在数据不足或超出范围时的处理方式。

  * **无假设的优化** 没有可衡量的假设就“调整措辞”会浪费周期。

* * *

## 一个更好的心智模型​

将技能视为：

  * 提示文本 = 实现
  * 黄金提示集 = 单元测试
  * 评估器 + 测试运行 = 回归检查
  * 发布说明 = 变更日志

一旦团队采用这个模型，技能质量就变得可预测。

## 你的下一个技能 PR 的最小化模板​

在你的内部 PR 描述中使用以下结构：

    ## 目标
    此技能解决的用户任务是什么。

    ## 行为契约
    - 输入
    - 输出
    - 防护栏
    - 拒绝策略

## 测试集
- 黄金提示词数量：N
- 是否包含边界案例：是/否

## 结果
- 基准通过率：X%
- 候选版本通过率：Y%
- 已知回归问题：……

## 决策
推荐上线 / 暂缓发布 / 回退版本

这使得技能变更能够以代码变更的同级标准进行评审。

## 核心启示​

Anthropic技能创建器更新的最大亮点并非“更智能的生成能力”。

而在于技能现在能够融入开发者原生的生命周期：

**设计 -> 测试 -> 评估 -> 优化 -> 发布**

如果你的团队正在构建严肃的智能体工作流，这正是杜绝提示词漂移、实现可靠技能交付的实践路径。
