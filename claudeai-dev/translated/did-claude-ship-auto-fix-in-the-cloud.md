Claude是否已在云端发布自动修复？

简短回答：不是作为一个单一的命名功能，但功能上，几乎是的。

Anthropic没有发布名为"云端自动修复"的东西。它实际发布的内容更有趣：Claude Code现在具备了在Anthropic管理的云基础设施上运行编码工作、监控拉取请求、自动尝试CI修复，甚至在检查通过时合并的组件。

对于开发者来说，这种区别很重要。这不仅仅是品牌细节。它告诉你工作流程实际是如何组合的，什么是自动的，什么仍然取决于您的设置。

Anthropic实际发布了什么

有两个需要结合的单独发布。

1. 网页上的Claude Code

2025年10月20日，Anthropic推出了网页版Claude Code：

- 编码任务在Anthropic管理的云基础设施上运行
- 任务可以跨仓库并行运行
- Claude可以创建PR并实时显示进度

这是"在云端"的部分。

2. PR的自动修复和自动合并

2026年2月20日，Anthropic宣布了新的Claude Code桌面功能：

- 预览运行中的应用
- 推送前审查本地差异
- 从应用程序监控PR状态
- 为CI失败启用自动修复
- 检查通过后启用自动合并

这是"自动修复"的部分。

Anthropic还明确表示，会话可以通过"继续使用网页版Claude Code"从桌面移动到云端。

真正的答案是什么

准确的答案是：

- 是的，Claude现在支持云支持的编码工作流程
- 是的，Claude可以自动尝试CI修复
- 是的，Claude可以跨桌面、网络和移动继续工作
- 不，Anthropic没有宣布一个 literally 名为"云端自动修复"的独立功能

更安全的表述是：Claude Code现在将云执行与PR自动修复和自动合并工作流程相结合。

为什么名字不如工作流程重要

名字不如工作流程转变重要。

之前，"AI编码助手"大多意味着两件事之一：在IDE或终端中的本地编辑帮助，或仍需要手动跟进的后台云任务。

Claude现在更接近闭环：

1. 在本地或网络上开始工作
2. 打开PR
3. 在后台监控CI
4. 尽可能自动修复失败
5. 准备好时自动合并

这不是完全的自主软件交付。但它确实是"生成代码并等待人工看管其余部分"之外的真正一步。

什么仍然不是自动的

Claude仍然不能神奇地成为一个通用的无人值守DevOps代理。

官方工作流程仍然取决于：

- GitHub托管的代码
- PR状态可见性
- Claude可以观察的CI信号
- 已在Claude Code中启动的会话
- 您关于自动修复和自动合并的批准和配置选择

正确的思维模式不是："Claude独立地在我的仓库中巡逻并修复一切。"

更接近的是："Claude可以更长时间地保持任务，跟随PR通过CI，并在每次失败时尝试闭环而不把我拉回来。"

最终结论

如果你问Claude是否发布了你可以合理描述为云端自动修复的东西，诚实的答案是：工作流程能力方面是，官方产品命名方面否。

Anthropic更早地发布了云运行器，后来发布了PR自动修复和自动合并，并连接了跨本地和云面的会话。

这种组合才是真正的故事。
