# 通过测试-衡量-优化构建更优的智能体技能

大多数智能体技能失败的原因很乏味：我们修改提示词，重新运行一次，就称之为“改进”。Anthropic最新的Skill Creator更新推动了一种更工程化的循环：先测试，衡量行为，再优化。如果你正在构建内部智能体工作流，这才是真正重要的转变。本文将官方公告改写为可供你每周执行的开发者工作流程。

## 核心理念：停止盲目调整提示词

官方更新不仅仅是“更好的提示生成”。它是一个质量闭环：
- 预先定义可衡量的行为
- 根据该行为生成测试套件
- 对新旧版本运行评估
- 利用评估反馈消除冲突和无效指令
- 基于证据而非感觉进行迭代

如果这听起来像是提示词的持续集成（CI），那就对了。

## Skill Creator的变更（开发者视角）

从官方文章中，以下是值得关注的实际变化：

**实施前的黄金提示词**
你编写（或协同创建）代表真实任务的具体提示词。这些成为你的基线用例。

**用于自动技能测试的test-creator**
Skill Creator可以根据你的需求生成测试，无需从头手动编写每个用例。

**集成的技能评估器**
评估反馈会突出指令冲突、重叠和薄弱环节，帮助你完善技能规范。

**基于质量信号的优化**
你需要循环多个版本，根据测试输出衡量变化，而不仅仅是“感觉更好”的判断。

这更适合工程师，因为它将技能视为版本化的工件，而非一次性提示词。

## 可用的实用工作流程

**步骤1）像定义API行为一样编写验收标准**
在编辑任何技能文本之前，以严格形式编写预期行为：
- 输入格式
- 输出模式
- 必须遵守的规则
- 禁止行为规则
- 失败行为（上下文缺失时的处理方式）

如果你的标准模糊，测试也会模糊。

**步骤2）从实际事件构建黄金提示词集**
使用团队的真实日志、工单或请求。包括：
- 正常情况
- 嘈杂/模糊情况
- 缺失上下文情况
- 超出范围情况

这些是你的回归测试套件。保持小而精。

**步骤3）生成并运行测试**
使用Skill Creator + test-creator生成结构化技能测试。然后运行：
- 当前生产技能
- 候选更新技能

针对同一测试集比较输出质量。

**步骤4）运行评估反馈并精简指令**
寻找重复出现的失败模式：
- 矛盾的指令
- 过于宽泛的指令
- 隐藏的假设
- 输出格式不稳定

每次迭代只优化一两个变量，以便归因改进效果。

**步骤5）仅当指标改善时才发布**
不要因为一个例子看起来不错就发布。仅当满足以下条件时才发布：
- 完整套件的通过率提高
- 失败模式减少而非转移
- 边缘情况下的输出格式保持稳定

## 团队常见的困境

以下是我们在真实开发团队中看到的常见失败模式：

**单一技能包含过多规则**
试图同时处理摘要、规划、分类和政策解释的单一巨型技能通常会迅速退化。

**没有版本化的测试数据**
如果黄金提示词没有版本化，你就无法信任趋势线。

**缺乏拒绝行为**
技能应定义数据不足或超出范围时的处理方式。

**无假设的优化**
没有可衡量假设的“让我们调整措辞”会浪费周期。

## 更好的心智模型

将技能视为：
- 提示文本 = 实现
- 黄金提示词 = 单元测试
- 评估器 + 测试运行 = 回归检查
- 发布说明 = 变更日志

一旦团队采用这种模型，技能质量就变得可预测。

## 下一个技能PR的最小化模板

在内部PR描述中使用此结构：

```markdown
## 目标
此技能解决的用户任务。

## 行为契约
- 输入
- 输出
- 防护措施
- 拒绝策略

## 测试集
- 黄金提示词数量：N
- 是否包含边缘情况：是/否

## 结果
- 基线通过率：X%
- 候选版本通过率：Y%
- 已知回归：...

## 决策
发布 / 暂缓 / 回滚
```

这使技能变更可以通过与代码变更相同的标准进行评审。

## 最终要点

Anthropic Skill Creator更新的最大亮点不是“更智能的生成”，而是技能现在适配了开发者原生的生命周期：设计 -> 测试 -> 评估 -> 优化 -> 发布

如果你的团队正在构建严肃的智能体工作流，这就是你停止提示漂移并开始交付可靠技能的方法。
