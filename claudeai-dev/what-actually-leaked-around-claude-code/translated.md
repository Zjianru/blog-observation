# 这次Claude Code到底泄露了什么？

来源：https://claudeai.dev/blog/what-actually-leaked-around-claude-code

---

起初，“Claude Code源代码泄露”这个说法听起来有些夸张。

但仔细审视社区证据后，这种怀疑需要重新评估。

这件事不仅涉及提取的提示词或模糊传闻。更可信的说法是：**Claude Code的npm包中附带的一个源码映射文件暴露了内部源文件的路径，社区利用该路径重建了一个比公开仓库大得多的代码库**。

这比社交媒体上常见的夸大其词更接近真正的源码泄露。

## 核心说法​

社区的说法很直接：

  1. 发布的Claude Code npm包包含一个JavaScript源码映射文件
  2. 该源码映射指向内部TypeScript源文件位置
  3. 这些源文件可被获取并重建
  4. 结果以反编译或重建形式重新发布到GitHub

如果这个链条准确，那么这就不仅仅是“提示词泄露”或“有人猜到了系统提示”。

这是一个具有源码级后果的打包错误。

## 为何社区认为这是真的​

这里有三个重要的公开证据。

### 1\. Reddit讨论串​

在**2026年3月30日**，`r/ClaudeAI`上的一个Reddit帖子声称Claude Code源码“通过NPM上的`.map`文件”泄露。

这个具体机制很重要，因为源码映射是已知的故障模式：

  * 可能暴露原始文件结构
  * 可能揭示未混淆的标识符
  * 根据构建和托管方式，可能指向原始源码内容

这已经比常见的“有人泄露了内部代码”传闻具体得多。

### 2\. Fried Rice的X推文​

社区持续引用的X推文提供了更详细的说法：

  * 据称Claude Code的npm包附带了源码映射
  * 据称该映射引用了Anthropic管理存储上的zip存档
  * 据称该存档包含比之前公开可见内容大得多的内部源码树

即使社区仍在重建某些细节，但所声称的机制是连贯的。

### 3\. `instructkr/claude-code`仓库​

最有力的公开证据是GitHub仓库：

  * `https://github.com/instructkr/claude-code`

该仓库明确表示自己是从泄露的源码映射重建的Claude Code，并声明：

  * 原始npm包仅暴露少量打包文件
  * 恢复的源码树规模大得多
  * 重建了数千个文件
  * 泄露暴露了内部实现细节、提示词和功能开发

这并不能自动证明仓库中的每一行代码都完美或完整，但已远超模糊传闻。

## 与早期提示词泄露事件的区别​

早期Claude Code争议通常涉及：

  * 提取的系统提示词
  * 工具模式
  * 行为指令
  * 对代理逻辑的公开逆向工程

这些已有重要意义，但与丢失源码不同。

这次声称的故障模式不同：

  * **不仅仅是提示词提取**
  * **不仅仅是公开仓库检查**
  * **不仅仅是CMS草稿暴露**
  * **而是似乎启用了代码重建的已发布构建产物**

这是关键区别。

如果社区重建基本准确，这就不仅仅是“人们了解了Claude Code的行为方式”，而是“人们获得了Claude Code实际构建方式的更多内容”。

## 可能发生了什么​

基于公开的社区证据，最可能的链条如下：

1. Anthropic向npm发布了Claude Code软件包。
2. 该软件包包含了一个本不应以该形式暴露的源码映射文件。
3. 该映射文件揭示或引用了原始源码信息。
4. 社区研究人员顺藤摸瓜，重构出了内部代码。
5. 重构后的代码随后通过GitHub和社交媒体传播开来。

这表明这是一次打包和发布流程的失误，而不仅仅是"有人泄露了提示词"的事件。

## 这对开发者为何重要​

这个现实教训严酷却非常熟悉：

**源码映射文件属于你的发布暴露面。**

开发团队常将其视为无害的调试残留物。但当它们出现以下情况时，绝非无害：

  * 保留原始路径元数据
  * 保留可读的符号名称
  * 引用私有构件存储
  * 或暴露足以重建内部源码的足够结构

对于AI产品，这种风险更为严重，因为一次泄露可能同时暴露多个层面：

  * 代码
  * 提示词
  * 工具合约
  * 功能开关
  * 安全控制机制
  * 未发布的产品工作

这正是为什么本次事件比普通的"助手提示词泄露"帖子更值得关注。

## 这并不意味着什么​

即使源码泄露的说法基本属实，团队也应避免夸大其词。

这**并不**自动意味着：

  * Anthropic的模型权重已泄露
  * 所有内部服务都已泄露
  * 所有Claude产品都已泄露
  * 或重构出的仓库必然是完美、权威的源码树

但这**确实**意味着某些严重情况：

  * 一个生产环境分发的构件暴露了足够信息，使得外部人员能够重建Claude Code的大部分内容

按照常规软件安全标准，这已然是重大失误。

## 更重要的安全启示​

真正的教训并非"AI公司格外脆弱"。

而是更普遍的道理：

  1. 构建流程会泄露信息。
  2. 调试构件若随产品发布，即成为生产构件。
  3. 一旦软件包元数据暴露内部存储引用，这些引用即变为公开。
  4. 在AI工具链中，提示词与代码已紧密关联，一次打包失误可能同时暴露实现逻辑和行为逻辑。

这正是为什么发布工程应当获得与模型安全宣传同等的重视。

## 最终结论​

在查阅Reddit讨论串、X平台对话以及重构的GitHub仓库后，更明确的结论是：

**Claude Code"源码泄露"事件基本属实，而源码映射文件是导致此事的关键原因。**

这个故事并非仅仅是人们猜中了提示词或过度解读公开代码。

更重要的主张是：Anthropic似乎通过npm发布了一个构建构件，该构件暴露了足够的源码级信息，使得社区能够重构Claude Code的主要部分。

如果这是真实的事件链条，那么这不仅是一则AI领域的戏剧性事件。

这是一次根源极其普通、后果却极不寻常的发布工程失误。
