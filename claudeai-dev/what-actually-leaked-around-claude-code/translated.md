# 这次Claude Code究竟泄露了什么？

来源：https://claudeai.dev/blog/what-actually-leaked-around-claude-code

---

"Claude Code源代码泄露"这个说法初听之下有些夸大其词。

但仔细审视社区提供的证据后，原先的怀疑态度需要调整了。

这次事件并非简单的提示词提取或模糊传闻。更可信的说法是：**Claude Code的npm包中附带了一个源码映射文件，该文件暴露了内部源码文件的路径，社区利用这个线索重建出了远比公开仓库更庞大的代码库**。

这比社交媒体上常见的夸张说法更接近真正的源码泄露事件。

## 核心主张

社区的主张非常明确：

  1. 发布的Claude Code npm包包含了一个JavaScript源码映射文件
  2. 该映射文件指向了内部TypeScript源码位置
  3. 这些源码可以被获取并重建
  4. 重建结果随后以反编译或重构形式发布到GitHub

如果这个链条属实，那么这就不仅仅是"提示词泄露"或"有人猜中了系统提示"那么简单。

而是一个具有源码级影响的打包失误。

## 社区为何认为此事属实

以下三个公开证据至关重要。

### 1. Reddit讨论串

在**2026年3月30日**，`r/ClaudeAI`板块的Reddit帖子声称Claude Code源码"通过NPM上的`.map`文件"泄露。

这个具体机制很重要，因为源码映射是已知的风险点：

  * 可能暴露原始文件结构
  * 可能揭示未混淆的标识符
  * 根据构建和托管方式，甚至可能指向原始源码文件

这已经比常见的"有人泄露内部代码"传闻具体得多。

### 2. Fried Rice的X平台推文

社区反复引用的X平台讨论串提供了更详细的说法：

  * 据称Claude Code的npm包附带了源码映射文件
  * 据称该映射引用了Anthropic管理存储中的zip归档文件
  * 据称该归档包含的内部源码树规模远超此前公开可见的内容

即便某些细节仍在社区重构中，所推测的机制在逻辑上是自洽的。

### 3\. `instructkr/claude-code` 代码仓库​

目前最有力的公开证据是 GitHub 仓库：

  * `https://github.com/instructkr/claude-code`

该仓库明确声称自身是基于泄露的源码映射（source maps）对 Claude Code 的重构，并指出：

  * 原始的 npm 包仅暴露了少量打包后的文件
  * 恢复出的源码树规模要大得多
  * 有数千个文件被重构还原
  * 此次泄露暴露了内部实现细节、提示词及功能开发内容

这并不能自动证明仓库中的每一行代码都完美或完整，但其可信度已远超模糊的传闻。

## 与早期提示词泄露事件的区别​

此前关于 Claude Code 的争议通常涉及：

  * 被提取的系统提示词
  * 工具模式定义
  * 行为指令
  * 对智能体逻辑的公开逆向工程

这些信息本身已有价值，但与源码丢失的性质不同。

本次事件中，推测的失效模式有所不同：

  * **不仅仅是提示词提取**
  * **不仅仅是公开仓库的审查**
  * **不仅仅是内容管理系统草稿的暴露**
  * **而是一个已发布的构建产物，似乎使得代码重构成为可能**

这才是关键区别。

如果社区的重构基本准确，那么这不仅仅是“人们了解了 Claude Code 的行为方式”，而是“人们获取了 Claude Code 实际构建过程的更多细节”。

## 可能的发生过程​

基于公开的社区证据，最合理的推测链条如下：

  1. Anthropic 向 npm 发布了一个 Claude Code 软件包。
  2. 该软件包包含了一个本不应以该形式暴露的源码映射。
  3. 该映射揭示或引用了原始源码信息。
  4. 社区研究人员循此线索重构了内部代码。
  5. 重构后的代码随后通过 GitHub 和社交媒体传播。

这使得本次事件成为一次打包和发布流程的失误，而不仅仅是“有人泄露了提示词”的事故。

## 对开发者的重要意义​

这个实际教训是严峻且非常熟悉的：

**源码映射是你发布内容的一部分。**

团队常将其视为无害的调试残留物。但当它们出现以下情况时，便不再无害：

* 保留原始路径元数据
* 保留可读的符号名称
* 引用私有构件存储
* 或暴露足以重建内部源码的结构

对于AI产品，这种风险更为严重，因为一次泄露可能同时揭示多个层面：

* 代码
* 提示词
* 工具契约
* 功能开关
* 安全控制
* 未发布的产品工作

这正是为什么此事比普通的"助手提示词泄露"帖子更值得关注。

## 这并不意味着​

即使源码泄露的说法基本属实，团队也应避免夸大其词。

这**绝不**自动意味着：

* Anthropic的模型权重已泄露
* 所有内部服务已泄露
* 所有Claude产品已泄露
* 重建的仓库必然是完美、权威的源码树

但这**确实**意味着某些严重问题：

* 一个生产分发的构件暴露了足够信息，使外部人员能够重建Claude Code的大部分内容

按照常规软件安全标准，这已是重大失误。

## 更重要的安全启示​

真正的教训并非"AI公司格外脆弱"。

而是更普遍的道理：

1. 构建流水线会泄露信息。
2. 如果你发布调试构件，它们就是生产构件。
3. 内部存储引用一旦被包元数据暴露，就会成为公开信息。
4. 在AI工具链中，提示词与代码已紧密关联，一个打包错误就可能同时暴露实现逻辑和行为逻辑。

这就是为什么发布工程值得获得与模型安全宣传同等的重视。

## 最终结论​

在查看Reddit讨论串、X平台对话和重建的GitHub仓库后，更明确的结论是：

**Claude Code"源码泄露"事件基本属实，而源码映射问题是其主要原因。**

此事并非仅是人们猜中提示词或过度解读公开代码那么简单。

更重要的说法是，Anthropic似乎通过npm发布了一个构建产物，其中暴露了足够的源码级信息，使得社区能够重构Claude Code的主要部分。

如果这确实是事件链条，那么这不仅仅是一则AI领域的戏剧性新闻。

这是一次发布工程上的失误，根源极其普通，后果却极不寻常。
