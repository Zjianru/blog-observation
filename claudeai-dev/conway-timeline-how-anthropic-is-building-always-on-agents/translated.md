# 康威时间线：Anthropic如何构建持续在线智能体

来源：https://claudeai.dev/blog/conway-timeline-how-anthropic-is-building-always-on-agents

---

关于**康威**最需要明确的一点是：

**康威并非Anthropic正式发布的产品。**

目前存在的信息混合了以下内容：

* Anthropic围绕**Cowork**、**Dispatch**、**计算机使用**、**定时任务**和**自动模式**的官方发布
* 当前帮助中心文档展示了这些功能模块如何协同工作
* 一份**2026年4月1日**的第三方报告披露了名为**康威**的未发布内部环境

如果只看泄露信息，你会错过整体架构。如果只看官方发布，你会错过Anthropic的发展方向。

技术真相需要结合两者来看。

## 简要回答​

从系统视角看，Anthropic显然正在向**持续在线智能体模式**迈进。

这并非因为康威已公布，而是因为已发布的产品栈已包含大部分必要基础组件：

* 跨设备同步的**持久化线程**
* 支持本地文件访问的**桌面运行时环境**
* **定时执行**功能
* 缺乏直接集成时的**计算机使用**能力
* **子智能体协同**机制
* 更安全的**长期运行权限模式**
* 通过**Dispatch**实现的移动端到桌面端任务交接

康威的重要性在于，它似乎将这些理念整合为更明确的持续在线智能体环境。

## 已确认信息与未确认信息​

在深入时间线前，先将证据分为三类。

### Anthropic官方确认​

Anthropic已正式发布或文档记录：

* Cowork作为基于Claude Code智能体架构的桌面代理环境
* 通过Dispatch实现手机与桌面间的持久化任务交接
* 定时任务与循环例程
* Cowork和Claude Code中的计算机使用功能
* Claude Code长时间会话的自动模式

### 当前Anthropic文档确认（但非单一产品发布）​

Anthropic当前帮助文档已描述如下工作流：

* 你保持**一个连续的对话线程**
* Claude 可将**开发工作路由至 Claude Code**
* Claude 可将**知识工作路由至 Cowork**
* 处理结果会返回至同一个持久化线程
* 计划任务可自动运行
* 计算机使用功能可操作你桌面上的应用程序

这已非常接近一个常驻智能体运行环境，即使 Anthropic 并未如此命名。

### 传闻披露，但未官方发布​

**Conway** 的名称源自 **2026年4月1日** 发布的第三方报告。根据该报告，Conway 似乎将提供：

* 专用的 **Conway 实例**
* 自带侧边栏，包含 **搜索**、**聊天** 和 **系统** 功能
* **扩展插件**
* **网络钩子**
* 从 **Chrome 版 Claude** 直接连接
* **通知系统**

这是目前最强烈的信号，表明 Anthropic 可能正在构建一个更明确的常驻智能体环境。

但这仍只是关于未发布产品的传闻，并非官方发布。

## 时间线​

以下是我根据官方信息及 Conway 报告梳理出的最清晰技术发展脉络。

### 2026年1月12日：Cowork 研究预览版发布​

Anthropic 推出 **Cowork**，称其为“面向其他工作的 Claude Code”。

关键不在于界面，而在于其运行模式：

* Claude 可直接操作你电脑上的文件夹
* Claude 能处理**多步骤任务**
* Cowork 基于与 Claude Code **相同的智能体技术基础**构建
* 用户可排队任务，让 Claude 更像协作者而非聊天线程

这是首个公开迹象，表明 Anthropic 希望构建更广泛的桌面智能体运行环境，而不仅是终端编码工具。

### 2026年1月至2月：Cowork 向多套餐和 Windows 系统扩展​

Anthropic 将 Cowork 推广至各付费套餐，随后扩展至 Windows 平台。

这在技术上具有重要意义，表明 Cowork 并非仅限高端用户的小范围实验。Anthropic 正在强化该运行环境以支持更广泛部署。

### 截至2026年3月下旬：Cowork 文档阐明完整的本地智能体运行架构​

Anthropic 当前的 Cowork 帮助文档现已比最初发布时更清晰地阐述了整体架构。

Cowork 的功能说明如下：

*   将 Claude Code 的**智能体能力**引入 Claude Desktop
*   支持**直接本地文件访问**
*   协调**多个子智能体并行工作**
*   支持长时间运行任务
*   支持**定时任务**
*   支持包含独立文件、指令和记忆的**项目**
*   在**虚拟机环境**中运行工作

这并非聊天机器人的功能集。这是一个智能体运行时。

### 2026年3月23日：Dispatch 与计算机使用功能在 Cowork 和 Code 中全面落地

这是迄今为止最重要的公开里程碑。

Anthropic 正式描述了一种产品状态，即 Claude 能够：

*   维持一个可随你从**手机到桌面**的连续对话
*   使用你的**计算机**
*   跨会话记住上下文
*   按**计划**运行任务
*   在**Cowork 和 Claude Code** 中协同工作

此刻，“始终在线”的发展方向已难以否认。

一个始终在线的智能体最需要两样东西：

1.  持久性
2.  在你未关注会话时采取行动的能力

Dispatch 加上计算机使用功能，正是这一组合。

### 2026年3月下旬：Dispatch 文档新增路由模型

Anthropic 的 Dispatch 支持文档增加了一个尤为重要的细节：

*   你拥有**一个连续持久的线程**
*   当你分配任务时，Claude 会判断其工作类型
*   **开发任务在 Claude Code 中运行**
*   **知识工作在 Cowork 中运行**

这是一个强烈的架构暗示。

Anthropic 不再将 Code 和 Cowork 呈现为孤立的产品，而是将其呈现为**同一任务线程下的专用执行界面**。

这正是你对一个始终在线智能体平台的预期划分：

*   一个面向用户的线程
*   多个执行后端
*   统一的记忆与交接

### 2026年3月24日：自动模式为 Claude Code 提供了更安全的长时运行路径

一天后，Anthropic 为 Claude Code 推出了**自动模式**。

其意义比初看起来更为重大。

如果每隔几分钟就需要请求许可，始终在线的智能体将毫无用处。但完全禁用权限又不够安全。自动模式是 Anthropic 的中间层方案：

*   Claude 可以代表您做出一些权限决策
*   分类器会在工具调用运行前进行筛查
*   安全的操作会自动执行
*   有风险的操作会被阻止或升级处理

这是实现无人值守或半无人值守工作所缺失的基础设施环节。

如果没有类似自动模式的功能，一个持续运行的编码代理大多只能停留在演示阶段。

### 2026年4月1日：第三方报告披露Conway​

TestingCatalog 发布报告称，Anthropic 正在测试一个名为 **Conway** 的内部持续运行代理环境。

报告的细节值得注意，因为它们与 Anthropic 已发布的产品方向完全吻合：

*   一个**独立实例**
*   扩展安装，包括 `.cnw.zip` 文件
*   **连接器和工具**
*   直接的 **Claude in Chrome** 连接
*   具有可唤醒实例的公共 URL 的 **webhooks**
*   通知功能

如果消息属实，Conway 并非一个随意的副业项目。它看起来像是 Anthropic 已在公开构建的运行环境之上的下一个封装层。

## Conway 为何在技术上成立​

如果抛开产品名称，Anthropic 已经具备了构建持续运行代理系统所需的大部分组件。

### 1. 持久控制线程​

Dispatch 提供了面向用户的线程，该线程能在设备更换时持续存在。

这个线程很重要，因为它成为了稳定的控制界面：

*   用户在此下达目标
*   结果返回此处
*   审批可在此进行
*   记忆可在此累积

没有这个线程，每个任务都是全新的启动。

### 2. 专用运行环境​

Anthropic 拥有两个不同的执行界面：

*   **Claude Code** 用于开发工作
*   **Cowork** 用于更广泛的桌面知识工作

这已经是一个便于调度的架构。系统可以决定任务需要哪种运行环境，而不是强迫一个界面处理所有事情。

### 3. 超越前台聊天的触发机制​

计划任务和 Dispatch 将模型推向了超越被动聊天的范畴。

这一点很重要，因为一个持续运行的代理需要外部触发器：

*   基于时间的触发器
*   来自其他设备的用户消息
*   可能还有服务触发的事件

康威报告在此处变得尤为有趣，因为**Webhook**恰恰是下一个合乎逻辑的触发界面。

### 4. 通过计算机使用实现后备执行​

直接集成始终优于屏幕控制。Anthropic在计算机使用文档中明确提到：Claude应优先选择连接器，其次是浏览器自动化，最后才是直接的屏幕交互。

这一顺序至关重要。它意味着Anthropic并非将“计算机使用”作为噱头来构建，而是将其视为更广泛行动栈中的**最后手段执行器**。

这正是严肃的智能体系统通常的设计方式：

* 首选结构化API路径
* 次选浏览器路径
* 最后考虑UI控制路径

### 5. 长期运行工作的治理机制​

自动模式是首个可见迹象，表明Anthropic认识到持续运行的智能体必须具备与前台助手不同的权限架构。

如果康威真实存在，它几乎必然需要强化版本的：

* 策略检查
* 事件过滤
* Webhook级别的信任边界
* 扩展沙箱化
* 可审计性
* 唤醒/休眠语义

这就是为什么康威报告中提及**扩展**和**Webhook**不仅仅是功能传闻——这些正是治理变得复杂的界面所在。

## 最值得关注的康威信号​

并非所有泄露细节都具有同等重要性。

从技术视角看，以下信号最为关键：

### 独立实例​

“康威实例”这一表述暗示Anthropic正在以**持久化智能体容器**的思维进行设计，而非仅仅是一个聊天标签页。

这引出了生命周期相关的问题：

* 实例何时唤醒
* 保持活跃的时长
* 哪些状态会持久保存
* 重启时哪些内容会重置
* 外部触发机制如何运作

这些都是典型的持续运行智能体必须面对的问题。

### 扩展机制​

若报告准确，康威正朝着包含以下要素的扩展模型演进：

* 工具集
* UI标签页
* 上下文处理器

这是重大转变。它表明Anthropic可能希望为持久化智能体构建一流的插件架构，而非仅提供静态的内置运行时环境。

### Webhook​

这是整份报告中最强烈的持续运行信号。

Webhook 意味着智能体不再只是在前台等待用户指令。它能够被外部系统事件唤醒。

这正是从以下模式的跨越：

*   “在我询问时执行此操作”

转变为：

*   “在世界变化时执行此操作”

如果 Anthropic 真的推出由 Webhook 触发的智能体实例，那将使 Conway 从助手领域迈入工作流自动化基础设施的范畴。

### Chrome 连接​

Anthropic 已经拥有 Claude for Chrome 和基于浏览器的操作路径。建立 Conway 到 Chrome 的直接桥梁在架构上是合理的，因为它能让这个常驻实例将浏览器视为一个持久的操作界面。

### 通知​

通知听起来微不足道，实则不然。

一个常驻智能体需要一个紧凑的信令通道来处理：

*   任务完成
*   需要批准
*   触发失败
*   环境离线
*   计划被跳过

没有通知，持久运行的智能体会变得不透明且令人厌烦。

## 真正的技术解读​

我的解读是：

**Conway 很可能并非一个全新的架构。它很可能是 Anthropic 将几个已经发布的子系统打包成一个更明确的持久智能体环境的尝试。**

这些子系统已经可见：

*   Cowork 运行时
*   Claude Code 运行时
*   调度线程
*   计划任务
*   计算机使用
*   移动设备切换
*   更安全的长期运行权限

如果 Conway 发布，它很可能将成为这样一个层面：将这些子系统转化为一个更明确的智能体实例模型，具备：

*   持久身份
*   事件触发器
*   扩展点
*   外部唤醒路径
*   统一的通知机制

这将是从“Claude 可以在我离开时继续工作”真正向前迈进的一步。

它将变成：

**Claude 可以作为软件系统常驻运行。**

## Anthropic 仍需解决的问题​

这部分是人们往往忽略的。

常驻智能体不仅仅是一个用户体验问题，它主要是一个系统和安全问题。

如果 Conway 公开推出，Anthropic 至少仍需解决五个难题。

### 1. 触发器信任​

Webhook 触发的智能体功能强大，但也带来了攻击面：

*   伪造事件
*   重放事件
*   载荷污染
*   通过外部服务进行权限提升

### 2. 扩展隔离​

只有当第三方逻辑无法悄无声息地破坏实例时，扩展模型才具有实际价值。

这意味着康威需要为以下问题提供强有力的解决方案：

*   软件包信任机制
*   安装权限控制
*   工具作用域限定
*   网络边界划分
*   更新行为规范

### 3. 状态健康度​

持久化智能体会不断积累上下文信息，而过多的上下文积累会导致信息劣化。

Anthropic需要建立强大的机制来处理：

*   记忆修剪
*   项目边界隔离
*   陈旧状态检测
*   特定触发器的上下文窗口管理

### 4. 人类控制权​

智能体的持久性越强，快速中断机制就越重要。

用户需要可靠的方式来实现：

*   暂停实例运行
*   撤销工具权限
*   切断连接器访问
*   禁用触发器
*   检查近期操作记录

### 5. 执行可见性​

前台对话模式便于观察操作行为，而持续运行模式则不然。

这意味着在康威式系统中，审计日志、执行摘要、调用链追踪和事件历史记录的重要性远超普通对话系统。

## 最终结论​

若问"Anthropic是否已正式发布康威？"答案是**没有**。

若问"Anthropic是否明确朝着持续运行智能体架构发展？"答案是**是的**。

官方产品发展轨迹已清晰指明方向：

*   **Cowork** 奠定了桌面智能体运行时基础
*   **Dispatch** 建立了跨设备持久化线程
*   **定时任务** 实现了基于时间的自动化
*   **计算机使用** 构建了备用执行层
*   **自动模式** 确立了更安全的长期运行机制
*   传闻中的**康威**环境则补充了缺失的实例、网络钩子、扩展和通知等关键语言

这才是真正的技术演进脉络。

康威的重要性不在于证明Anthropic已完成持续运行智能体的开发，而在于它让Anthropic的产品发展轨迹变得清晰可见。
