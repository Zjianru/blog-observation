Perplexity如何借助Realtime API为数百万用户带来语音搜索

Lessons from how Perplexity Computer's voice agent was built with the Realtime API.

作者：Paul Fryzel (Perplexity)、Charu Jaiswal (OpenAI)

在Perplexity，我们非常注重打造极致的产品体验。对于我们的智能浏览器Perplexity Comet，以及功能强大的通用数字工作者Perplexity Computer来说至关重要的一点是，让用户能够完全通过语音来使用这些产品。能直接说出你想要什么、然后交给AI执行、看着它完成任务，这种感觉有一种独特的满足感。我们非常看好语音作为界面，因为它让实际的交互体验更接近魔法。

我们在生产环境中使用Realtime-1.5，将这种魔法带到了Perplexity每月处理的数百万次语音会话中。通过Computer界面观察语音功能的增长，令人非常兴奋和学习良多。我们将分享一些我们学到的令人惊讶的经验。我们也鼓励你尝试Realtime-1.5，并与 我们分享你的发现。

1. 制定你的上下文管理策略

长篇内容，尤其是数小时的密集播客，是我们最清晰的上下文管理测试之一。我们想通过语音使播客转录稿变得可用，这样用户可以随时加入，询问两个半小时前发生的具体情况，并获得连贯的答案。

你无法将整个转录稿放入上下文。我们的第一版尝试是将其分成大块发送。我们很快发现，大块更新以一种全有或全无的方式失败。如果你试图发送一个10,000 token的更新，而窗口只剩下5,000 token的空间，模型将丢失所有前面的历史记录。这使得大块更新的风险更大，因为一次超量更新可能会抹掉整块上下文，而不是让系统更优雅地遗忘。

所以我们改变了方法。我们不再进行大块更新，而是开始将所有内容分解成更小的2,000 token块，并逐步输入。这样会有更多开销，但行为要稳定得多。当截断发生时，它会修剪一点历史，而不是抹掉一切。

我们学到的另一个微妙的事情是，并非所有上下文都应该以相同的方式进入模型。当使用conversation.item.create来更新上下文时，item.type: "message"有三个角色：system、user和assistant。这些角色告诉模型它正在查看什么类型的消息。system用于指令和行为塑造，user用于最终用户输入，assistant用于模型生成的输出。

当我们处理不当时，交互开始感觉就不对劲了。如果太多上下文以user角色进入，模型的行为就好像用户在大声朗读每个文本块，包括网页摘要和评论，而不是简单地基于这些材料提问。如果太多以system角色进入，情况则相反。模型失去了它固有"知道"的内容、作为上下文提供的内容，以及用户此刻实际在问什么之间的区别。

一个很好的例子是浏览流程。当有人在页面上滚动时，我们不断向模型更新屏幕上的内容。如果所有这些都被框定为用户输入，模型开始表现得好像用户大声朗读了每个段落。这不是正确的思维模型。我们希望系统感觉它在后台感知页面，然后在用户提问时自然地回答。这最终更多是关于正确理解对话语义的，而不是关于原始上下文量。

2. 在所有产品平台上标准化音频

Perplexity有多个产品平台，如Ask、Comet和Computer。每个平台都建立在不同的客户端堆栈上。Swift、TypeScript、Rust和C++都可能产生不同的原生音频缓冲区。当我们让每个客户端用自己的原始或原生音频格式向Realtime API发送时，这导致了性能不一致。

最终我们在Rust中构建了一个SDK，以抽象掉这些平台特定的差异，并确保每个客户端向API发送相同的音频契约。在实践中，这意味着在波形到达服务器之前就对其进行处理：重新采样到48 kHz单声道，匹配Opus编解码器偏好和WebRTC内部速率，通过WebRTC APM运行回声消除、自动增益控制、噪声抑制和高通滤波，然后编码传输。SDK让我们在一个地方标准化音频常量、重采样和设置完整的处理流水线，而不是逐个客户端地进行处理。

3. 在复杂环境中调优

根据用户实际生活的环境来调优VAD非常重要。这意味着要根据真实的麦克风、扬声器音量和背景噪音进行校准。我们其中一个内部测试案例是旧金山的一个嘈杂酒吧，因为这感觉像一个真实的产品时刻。有人说："你试过新的Perplexity应用了吗？"他的朋友拿出手机，如果语音失败了，我们刚刚就失去了两个用户。当它工作时，反应更接近于"哇靠"。这对我们来说是一个有用的推动力。在干净条件下工作的方法往往在现实世界中会出问题。首先为复杂环境进行调优会更好。

语音UX最难的部分之一是正确处理停顿。人们自然会停下来思考、在屏幕上查找内容，或准备大声朗读。模型很容易将那个停顿视为轮次结束并过早地插入。我们在某些情况下看到了这一点，比如有人请求帮助进行数学推导。他们会停下来找到公式，结果模型在他们完成问题之前就插入了。这就是我们开发语音锁定的原因。我们没有使用传统的按键说话方式——默认关闭语音，用户必须按下才能说话——而是反过来。交互默认保持环境音，但当用户想要暂时保持发言权时，他们可以锁定语音并拥有该轮次的控制权。我们也认为这不仅仅是一次性功能。随着语音界面进入更复杂的工作流程，我们相信这种交互模式的某种版本将成为标准。

4. 只使用核心工具并保持它们在分布内

将工具集缩小到最重要的少数工具，在我们的案例中意味着少于十个。我们专注于覆盖最高价值动作的小型核心集。这种权衡是有意义的，随着更新的快照改进，可能会变得更好。

我们在系统提示中添加了明确的指令，说明何时以及如何调用每个工具。我们小心翼翼地保持工具模式和使用输出都在模型的分布内。在实践中，这意味着将工具输出格式化为普通结构化工具数据，而不是像助手对话那样。我们返回带有清晰分离字段的结构化JSON，例如response_text用于面向用户的语句，以及require_repeat_verbatim等标志用于行为，而不是将口语内容与内联指令混合。这使得工具使用更稳定，并使交互模式更接近模型在训练中可能看到的模式。

// Good:
{
  "response_text": "我启动了创建市场研究仪表板的任务",
  "require_repeat_verbatim": true
}
// Avoid:

我启动了创建市场研究仪表板的任务
