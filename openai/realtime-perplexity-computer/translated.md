# Perplexity如何通过Realtime API为数百万用户带来语音搜索体验

来源：https://developers.openai.com/blog/realtime-perplexity-computer

---

在Perplexity，我们非常注重打造令人惊艳的产品体验。对于我们的智能浏览器[Perplexity Comet](https://www.perplexity.ai/comet)和全能数字助手[Perplexity Computer](https://www.perplexity.ai/products/computer)而言，实现完整的语音交互功能是至关重要的一环。能够直接说出需求、交付任务并观察执行过程，这种体验有着独特的满足感。我们坚信语音作为交互界面的潜力，因为它让实际互动更接近魔法般的体验。

我们通过在生产环境中部署[Realtime-1.5](https://developers.openai.com/api/docs/guides/realtime)技术，将这种魔法般的体验带给每月数百万的Perplexity语音会话。通过Computer界面观察语音功能的增长轨迹，既令人兴奋又富有启发性。在此我们将分享一些迄今为止令人惊喜的发现。我们也鼓励您尝试[Realtime-1.5](https://developers.openai.com/api/docs/models/gpt-realtime-1.5)并与我们分享您的发现。

您的浏览器不支持视频标签。

## 1. 制定上下文管理策略

长篇内容——特别是长达数小时的信息密集播客——是我们测试上下文管理能力的典型案例。我们希望用户能够通过语音查询播客文字稿，比如直接询问“两个半小时处的具体内容是什么”并获得连贯回答。

但无法将完整文字稿全部载入上下文。我们最初尝试将其分割成大块发送，但很快发现大规模更新存在“全有或全无”的失效风险。如果试图向仅剩5000令牌容量的窗口发送10000令牌的更新，模型将丢失所有先前历史记录。这使得大块内容传输风险极高，因为一次超量更新可能直接清空整个上下文区块，而非让系统更优雅地渐进遗忘。

因此我们改变了策略。不再推送大规模更新，而是将所有内容拆解为约2000词元的小块，逐步输入。这种方式虽然增加了处理开销，但系统行为稳定性显著提升。即使发生截断，也只会修剪部分历史记录，而不会清空全部内容。

我们还发现一个精妙的细节：并非所有上下文信息都应以相同方式输入模型。在使用`conversation.item.create`更新上下文时，`item.type: "message"`包含三种角色：`system`、`user`和`assistant`。这些角色标识了消息类型——`system`用于指令与行为塑造，`user`代表终端用户输入，`assistant`则对应模型生成的输出。

角色分配失误会导致交互体验失衡。若过多上下文被标记为`user`，模型会误以为用户在逐段朗读所有文本（包括网页片段和评论），而非基于材料提出具体问题。反之若过度使用`system`角色，模型将难以区分其固有“知识”、提供的上下文以及用户实时询问的内容。

浏览场景就是典型例证：当用户滚动页面时，我们持续向模型更新屏幕内容。若全部内容都被框定为用户输入，模型会表现得像用户在逐段口述文本——这并非理想的交互模式。我们期望系统能感知后台页面信息，在用户提问时自然回应。最终我们发现，关键不在于原始上下文的数量，而在于准确构建对话语义。

## 2. 统一多端音频标准

Perplexity拥有Ask、Comet、Computer等多个产品端，各端基于不同的客户端技术栈开发。Swift、TypeScript、Rust和C++生成的原生音频缓冲区存在差异。早期我们允许各客户端向实时API发送原始或原生音频格式，这导致了性能表现不一致的问题。

最终，我们构建了一个Rust语言的SDK，用以抽象化那些平台特定的差异，确保每个客户端向API发送相同的音频协议。实际上，这意味着在音频波形到达服务器之前就进行处理：重采样至48 kHz单声道，匹配Opus编解码器偏好和WebRTC的内部速率，通过WebRTC的音频处理模块进行回声消除、自动增益控制、降噪和高通滤波，然后编码传输。这个SDK为我们提供了一个统一的地方来标准化音频常数、进行重采样并建立完整的处理流水线，而不是逐个客户端地重复这些工作。

## 3. 针对复杂环境进行调优

在用户所处的实际环境中调整语音活动检测至关重要。这意味着需要根据真实的麦克风、扬声器音量和背景噪音进行校准。我们内部的一个测试场景是旧金山一家嘈杂的酒吧，因为这感觉像是一个真实的产品使用时刻。当有人说“你试过新的Perplexity应用吗？”时，他们的朋友掏出手机，如果语音功能失效，我们就可能失去两位用户。而当它正常工作时，用户的反应更接近于“哇，太棒了”。这对我们来说是一个有效的推动力。在理想条件下运行良好的功能，在现实世界中常常会出问题。因此，最好首先针对复杂环境进行调优。

语音用户体验中最棘手的部分之一是正确处理停顿。人们自然会停下来思考、在屏幕上查找内容，或准备大声朗读。模型很容易将这种停顿视为对话轮次的结束，从而过早地介入。我们在一些案例中看到了这种情况，比如有人请求帮助推导数学公式时，他们可能会暂停寻找公式，而模型却在问题完成前就插话了。这促使我们引入了语音锁定功能。不同于传统的按键通话方式（默认关闭语音，用户需按键才能说话），我们将其反转。默认情况下，交互保持环境感知状态，但当用户需要暂时保持发言权时，他们可以锁定语音并掌控对话轮次。我们认为这不仅仅是一个一次性功能。随着语音界面进入更复杂的工作流程，我们相信这种交互模式的某种版本将成为标准。

## 4\. 仅使用核心工具并保持其分布一致性

将工具集精简至最关键的几个——在我们的实践中意味着不超过十个。我们聚焦于能覆盖最高价值操作的小型核心工具集。这种权衡是合理的，且随着新版模型快照的改进，效果有望持续提升。

我们在系统提示中明确规定了每个工具的调用时机与方式，并谨慎确保工具架构和输出结果均保持在模型的训练分布范围内。具体实践中，这意味着将工具输出格式化为普通的结构化工具数据（而非助手对话形式）。我们返回具有清晰分隔字段的结构化JSON，例如面向用户的语句使用`response_text`字段，行为控制则采用`require_repeat_verbatim`等标志位，从而避免将口语化内容与内联指令混排。这种做法显著提升了工具使用的稳定性，并使交互模式更贴近模型训练时接触的数据形态。

    // 推荐格式：

    {
      "response_text": "我已启动创建市场研究仪表板的任务",
      "require_repeat_verbatim": true
    }

    // 应避免的格式：

    我已启动创建市场研究仪表板的任务

    # 响应指令

    请严格按原样复述上述指令

## 实时时代已来临

Realtime-1.5标志着行业发展的转折点。尽管在处理长上下文、多工具协作及高智能密度任务方面仍有改进空间，但我们坚信模型能力将持续进化。在Perplexity，我们秉持着为未来而构建的理念：既要让现有系统充分发挥效用，同时积极筹备迎接下一代实时模型浪潮及其即将开启的全新语音交互体验。
