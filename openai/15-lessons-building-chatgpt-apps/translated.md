构建 ChatGPT Apps 学到的 15 个经验教训

And how we incorporated them into a Codex Skill to help you build ChatGPT Apps 10x faster.

作者：Nikolay Rodionov（Alpic 联合创始人）

在 Alpic，我们相信下一代产品和服务将围绕 AI 优先体验构建，即用户与模型协作而不是导航传统预定 UI 工作流程的界面。

当 OpenAI 发布 Apps SDK 时，我们立即开始使用它进行构建。在三个月的时间里，我们为内部使用和跨 B2B 和 B2C 领域（如旅游、零售和 SaaS）的客户开发了两打 ChatGPT Apps。

我们很早就发现，构建 ChatGPT Apps 与构建传统 Web 或移动应用程序有根本的不同。在 Web 上运行良好的模式（即时数据获取、UI 驱动的状态、显式用户配置等）通常会在代理环境中崩溃或主动损害体验。

这篇文章是我们构建真实世界 ChatGPT Apps 时学到的 15 个最重要经验教训的提炼，然后是我们如何将这些经验教训融入社区的开源框架 Skybridge 和 Codex Skill，以帮助开发者显著更快地构思、构建、测试和发布 Apps。

三体问题

对于传统 Web 应用，事情很简单：你只有一个用户和一个 UI。在 ChatGPT 应用中，第三个物体进入系统：模型。

为 ChatGPT 构建最难的部分之一是管理信息在这三者之间如何流动。如果用户在 widget 中点击"选择"按钮，UI 会直观地更新，但模型的对话大脑，除非你明确地呈现该上下文，否则不会知道。如果用户然后问："给我更多关于这个产品的详细信息，"模型不知道用户实际在看什么。

我们称之为上下文不对称，其中每个物体对系统有部分知识，没有任何一个拥有完整图景。构建好的 ChatGPT Apps 不是关于保持所有东西同步，而是关于决定什么信息应该共享，何时应该共享，以及谁需要看到它。解决这是笨拙应用和无缝代理体验之间的区别。

1. 并非所有上下文都应该共享

我们的最初本能是"到处共享一切。"这被证明是我们最早的错误之一。

在实践中，ChatGPT App 的不同部分通常需要对相同状态有意地不同视图。为什么？

性能方面：UI widget 通常需要比模型需要的多得多的数据：例如，在旅行预订应用中，可能是图像、价格变体、预加载选项。将所有这些发送给模型会增加 token 使用量、延迟和认知噪音。
逻辑方面：某些信息必须通过对设计保持不对称。在我们最早的应用之一 Murder in the Valleys 神秘游戏中，模型需要知道凶手是谁才能正确地进行角色扮演，而 UI 和用户不能知道。在 Times Up 风格的游戏中，情况相反：UI 向用户显示秘密词，而模型必须保持不知情。

教训不是"总是同步一切，"而是：明确决定谁需要知道什么。我们使用不同的工具输出字段将其正式化：

字段 | 目的 | 对谁可见
structuredContent | widget 和模型的类型化数据 | 通过 toolOutput 和 callTool 函数，widget 和模型都可以看到
_meta | 响应元数据 | 仅对 widget 可见，对模型隐藏

例如，对于 Times Up 游戏，我们只将秘密词传递到 widget 的 _meta 字段，让模型从用户的提示中猜测这个词。

2. 懒加载不能很好地转换到 AI 应用

来自 Web 开发，我们默认使用懒加载：用户在点击时获取数据；按需加载详细信息；优化最小前期负载。

在 ChatGPT 中，范式是反过来的：工具调用意味着延迟，由于安全沙盒和模型推理，通常需要几秒钟。

在实践中，我们学会积极地前置加载：在初始工具响应中发送尽可能多的数据，并通过 window.openai.toolOutput 为 widget 补水。这几乎总是导致更快、更有响应性的体验。

当然，如果 widget 可以安全地从公共 API 端点获取数据，并且不需要与模型共享信息，始终可以在 widget 内部使用经典的 XHR 调用，但大多数时候你想要模型能够自主调用工具以保持会话体验。

3. 模型需要可见性

当用户与 widget 交互（例如，在列表中选择特定产品）然后在聊天中提问时，会出现一个微妙但关键的问题。如果模型不知道用户指的是 UI 的哪一部分，它将无法正确回答。

为此我们使用了 window.openai.setWidgetState(state)，它允许你存储特定状态数据，这些数据在下次用户与模型交互时被添加到模型的上下文中。

随着应用变得复杂，我们看到在很多地方添加 setWidgetState 以便模型跟踪导航。所以我们决定引入一种声明方式来描述 UI 上下文。不是在每次交互时强制更新模型，而是将 data-llm 属性直接附加到组件上：

<div
  data-llm={
    selectedTab === "details"
      ? "User is viewing product details"
      : "User is viewing reviews"
  }
>

为了在幕后工作，我们构建了一个 Vite 插件，抓取这些属性并自动更新 widgetState。从模型的角度来看，它只是在正确的时间接收相关 UI 上下文，而无需开发人员在每次交互时手动同步。

你可以在我们创建的开源框架中找到这个 Vite 插件（以及我们在本文中分享的许多其他提示）。

4. 不同的交互需要不同的 API

ChatGPT Apps 涉及 widget、服务器和模型之间的多个交互路径。这些路径不可互换：每个都存在以支持不同类型的交互。

构建 ChatGPT Apps 的关键经验之一是使这些通信路径明确化，并故意决定哪种机制负责体验的哪一部分。

为 AI 重新发明 UI

ChatGPT Apps 是一个全新的环境，所以我们很快学会搁置我们对 UI 的预设概念，并充分利用新功能。本节介绍我们需要学习（和忘记）以创建有效应用的界面设计假设。

5. UI 必须适应多种显示模式及其约束

ChatGPT Apps 不生活在单一布局中。根据调用它们的方式和时间，相同的 widget 可以以三种不同的显示模式呈现。

Apps 可以以内联方式出现在对话中，在其上方的画中画（PiP）中，或者在全屏需要更多空间时。虽然 PiP 和全屏支持更丰富的界面，但它们也引入了 widget 无法控制的 UI 覆盖层。考虑设备特定的安全区域（如移动设备上持久存在的关闭按钮）对于避免内容被剪裁和优化交互至关重要。

随着时间的推移，我们识别了关于显示模式的模式以及何时使用它们：

| 外观 | 何时使用 |
| 内联 | 默认显示模式。widget 保持在对话历史中。 | 用于快速交互 |
| 全屏 | widget 占据整个屏幕，聊天栏在底部。 | 如果你的 widget 复杂且需要很多空间（例如地图） |
| 画中画 | 与内联相同的大小，但 widget 保持在对话之上 | 如果你的 widget 在生成后的对话后续中保持相关 |

6. UI 一致性在嵌入式环境中很重要

早期，我们遇到的一个不确定性是 ChatGPT App 应该有多大的视觉自由度。作为用户的全新界面，它需要感觉熟悉和一致，无论在我们自己的应用内还是与周围的 ChatGPT 生态系统相比。与独立产品不同，widget 存在于现有界面内，视觉不一致会立即突出。

幸运的是，OpenAI Apps SDK UI Kit 为我们提供了一个清晰的基准。

它建立在 Tailwind CSS 之上，提供与 ChatGPT 设计系统对齐的可立即使用的组件、图标和设计令牌。使用它使我们能够快速行动，同时确保我们的 widget 感觉原生且与周围界面视觉一致，即使在构建自定义组件时（例如，用于我们的 Mapbox 集成）。

7. 语言优先过滤

传统仪表板建立在充满复选框和范围滑块的侧边栏上。在代理 UI 中，这通常是一种退步。当用户可以直接用自然语言表达意图时，例如，"预算 200 美元以下的欧洲晴天目的地"，强迫他们通过多个 UI 控制添加了摩擦。他们应该能够直接说出来。

因此，我们为大多数应用决定采用"无过滤器"的方式。不是提供带有过滤和排序选项的侧边栏，我们为工具参数提供值列表（LOV）。

这允许模型直接将用户的消息作为输入，防止它"猜测"有哪些可用选项。换句话说，它允许将自然语言直接映射到我们后端的 API 要求。如果用户说"晴天"，模型知道使用 weather="sunny" 调用工具。

8. 文件可以解锁更丰富的交互

当我们构建更复杂的应用时，出现的一个经验是文件不应被视为次要输入。在 ChatGPT Apps 中，文件可以解锁新的交互。体验可以从用户已经拥有的东西开始，而不是从表单或过滤器开始。

例如，在电子商务应用中，用户可以在聊天中上传产品照片，让模型识别它，然后直接在 widget 中继续进行产品匹配或发现。

这是通过让文件流经系统两侧实现的。在模型侧，工具可以通过 openai/fileParams 直接使用聊天中上传的文件，允许模型对图像或其他用户提供的资产进行推理。在 UI 侧，widget 也可以使用 window.openai.uploadFile 和 window.openai.getFileDownloadUrl 直接处理文件，使在 UI 流程中请求上传或生成用户可以下载和重用的文件成为可能。

投入生产

接下来，当应用超越本地开发时，会出现一组不同的考虑，涉及安全、配置和工具。这就是第三个经验教训集所涵盖的内容。

9. CSP 是新的 CORS

出于安全原因，OpenAI 在双嵌套 iframe 中呈现 Apps。内容安全策略（CSP）是 iframe 隔离的本机机制，此设置严格强制执行，通常表现为经典的"它在本地工作但在生产中崩溃"综合症。

与传统 Web 开发不同，你可能可以通过宽松策略来解决问题，Apps SDK 要求你精确。

在应用清单中，这意味着为每种交互类型仔细声明哪些域被允许：

字段 | 目的 | 示例 | 常见错误
connectDomains | API 和 XHR 请求 | https://api.weather.com | 忘记暂存 API 与生产
resourceDomains | 图像、字体、脚本 | https://cdn.jsdelivr.net | 使用像 delivr.net 这样的通用 CDN 而不将其列入白名单
frameDomains | 嵌入 iframe | https://www.youtube.com | 嵌入 YouTube 视频或 Mapbox 实例而不将其列入白名单
redirectDomains | 无警告打开的外部链接 | https://app.alpic.ai | 忘记结账或 OAuth 回调域

尽早将 CSP 配置作为首要关注事项，在以后节省了大量生产调试时间。

10. 小 widget 标志有不成比例的影响

除了 CSP，一小组 widget 级设置决定了 widget、模型和主机环境之间如何共享控制。这些标志很容易被忽视，但它们为导航、工具访问和发布定义了关键边界。

主机和导航边界
widgetDomain 是提交所必需的。它定义了"在 <App> 中打开"按钮在全屏模式下指向的默认位置，并参与源白名单，因为 widget 在 <widgetDomain>.web-sandbox.oaiusercontent.com 下呈现。我们使用 setOpenInAppUrl 根据上下文将用户路由到适当的路径。
模型和工具边界
工具注释必须遵循发布指南。像 readOnly、destructiveHint 和 openWorldHint 这样的标志是必需的，并在提交期间进行验证。
工具可见性很重要：不应该由模型调用的工具必须明确标记为私有。
Widget 执行边界
widgetAccessible 控制 widget 是否可以使用 callTool 自行调用工具。

单独这些设置很小，但 together 它们决定了一个应用在发布后是否表现正确。

优化快速迭代

Apps SDK 正在快速发展，我们很高兴围绕它构建。为了支持流畅高效的开发工作流程，我们决定开发我们自己的开源框架并与社区分享。以下是一些经验教训，以避免我们在开始时遇到的一些开发者体验问题。

11. 快速迭代需要热重载

我们首先解决的事情之一是迭代速度。长 TTL 资源缓存和使用 JSON-RPC 转发资源的组合使得标准热模块重载（如 Vite 或 Next.js 中的）无法开箱即用地与 ChatGPT Apps 兼容。

在花费大量时间了解 Vite 的内部结构后，我们构建了一个 Vite 插件，可以在 ChatGPT 内部直接实现 widget 的热重载。该插件拦截对 MCP 服务器的资源请求，并将实时更新注入 ChatGPT iframe。在 IDE 中看到更改立即反映在 ChatGPT 内部，大大缩短了我们的反馈循环。

12. 不是每个测试都属于 ChatGPT

在 ChatGPT 上测试是黄金标准，但对于第一次迭代，本地模拟器可以帮助你更快地移动，特别是当你处理需要在开发者模式下重新加载应用的工具定义时。

为了加快早期迭代，我们构建了一个轻量级本地模拟器，模拟 ChatGPT 主机环境，包含调试工具和特定于应用的日志。这允许我们在毫秒内迭代 React 状态和布局，为真正的 ChatGPT 测试保留模型交互和边缘情况的验证。

13. 移动测试需要明确支持

移动测试引入了一个单独的挑战：虽然在 ChatGPT 中测试需要隧道传输你的本地服务器，但 Vite 默认使用 localhost 使得相同的 URL 无法从其他设备访问。

我们通过扩展 Vite 插件以支持隧道端口上的域转发来解决这个问题，这解锁了 iOS 和 Android 设备上的测试，并使移动验证成为我们常规工作流程的一部分。

14. 熟悉的抽象（如 React hooks）加速前端工作

Apps SDK 暴露了强大的功能，但主要通过低级 JavaScript API。作为长期的 React 用户，我们想要更接近我们已经掌握的概念。

所以我们引入了一些 React 友好的抽象——像 useCallTool、useWidgetState 和 useLocale 这样的 hooks，以及更高级的状态管理，如在 Zustand 上构建的 createStore 用于复杂数据流。重新引入熟悉的前端模式减少了样板，并使 widget 开发感觉更接近现代 Web 工作流程。

将经验教训转化为 Codex Skill

15. 将经验教训转化为可重用工具

随着这些模式在多个应用中出现，反复重新发现它们正在减慢我们的速度变得清晰。为了使 ChatGPT App 开发更快、更可预测，我们决定将这些经验教训直接编码到我们的工具中，不仅为我们自己，也为社区。

这导致了两个互补的努力：

Skybridge Framework：一个开源 React 框架，将本文描述的许多模式打包成可重用的构建块，包括我们的 hooks（useCallTool、useToolInfo）、开发工具（HMR 和本地模拟器）以及 data-llm 属性。
chatgpt-apps-builder Codex Skill：在框架之上，我们构建了一个专用 Codex Skill 来支持完整的应用生命周期：
构思：集思广益如何使应用"代理化"，而不是仅仅一个 Web 移植。
代码生成：同时编写 React 前端和 MCP 服务器后端，预配置所有正确的 UX 和 UI 模式。
本地测试：启动开发服务器并将本地应用连接到 ChatGPT 以通过热重载进行实时迭代。
QA 和发布：针对 OpenAI 的提交指南运行结构化检查，包括 CSP 验证、安全区域考虑和生产测试。
应用部署：协助完成发货和迭代应用所需的最后步骤。

要安装和使用 Skill，只需使用以下命令：

npx skills add alpic-ai/skybridge

结论

构建 ChatGPT Apps 需要重新思考上下文如何流动、界面如何行为，以及用户和模型如何协作。本文中的许多经验教训来自熟悉 Web 模式与代理系统现实之间的差距。

通过分享这些经验教训，并将它们编码到我们的开源框架和 Codex skill 中，我们希望帮助团队减少重新发现相同问题的时间，更多地探索这种新交互模式使什么成为可能。最引人注目的 ChatGPT Apps 不会是现有产品的简单移植，而是围绕这种新的 AI 优先体验精心设计的体验。
