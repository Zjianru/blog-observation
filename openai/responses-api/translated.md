# 我们为何构建 Responses API

来源：https://developers.openai.com/blog/responses-api

---

随着 GPT-5 的问世，我们希望进一步阐述集成它的最佳方式——[Responses API](https://platform.openai.com/docs/api-reference/responses)，以及为何 Responses 专为推理模型和智能体未来量身打造。

每一代 OpenAI API 都围绕同一个核心问题构建：*开发者与模型交互最简单、最强大的方式是什么？*

我们的 API 设计始终以模型自身的工作原理为指导。最早的 `/v1/completions` 端点虽然简单但功能有限：用户输入提示词，模型仅完成后续文本。通过少量示例提示等技术，开发者尝试引导模型输出 JSON 或回答问题，但这些模型的能力远不及今日我们所习惯的水平。

随后迎来了 RLHF、ChatGPT 和后训练时代。模型不再只是续写未完成的文本——它们开始像对话伙伴一样*回应*。为适应这一变化，我们构建了 `/v1/chat/completions`（[据传仅用一个周末完成](https://x.com/athyuttamre/status/1899541474297180664)）。通过引入 `system`、`user`、`assistant` 等角色设定，我们提供了快速构建具备自定义指令和上下文的聊天界面的框架。

模型持续进化。很快，它们开始具备视觉、听觉和语音能力。2023 年末推出的函数调用功能成为最受喜爱的特性之一。同期我们发布了 Assistants API 测试版：这是我们首次尝试构建具备代码解释器和文件搜索等托管工具的完整智能体接口。部分开发者青睐此设计，但由于 API 结构相对聊天补全功能更受限且不易适配，未能实现大规模普及。

到 2024 年末，我们明确需要实现统一：既要具备聊天补全的易用性，又需拥有 Assistants 的强大功能，同时专为多模态和推理模型设计。于是 `/v1/responses` 应运而生。

## `/v1/responses` 是智能体循环

Chat Completions 为您提供了一个简单的回合制聊天界面。而 Responses 则为您提供了一个结构化的推理与行动循环。可以将其想象成与侦探合作：您提供证据，他们展开调查，可能会咨询专家（工具），最后向您汇报。侦探在步骤之间保留自己的私人笔记（推理状态），但从不将其交给客户。

这正是推理模型真正大放异彩之处：Responses 能够在多个回合间保留模型的**推理状态**。在 Chat Completions 中，推理过程在每次调用之间会被丢弃，就像侦探每次离开房间都会忘记所有线索。而 Responses 则让笔记本保持打开状态；逐步的思考过程实际上能够延续到下一个回合。这一点在基准测试中（TAUBench +5%）以及更高效的缓存利用和延迟方面都有所体现。

![responses vs chat completions](responses-api_00.webp)

Responses 还能够输出多个项目：不仅是模型**说了什么**，还包括它**做了什么**。您会收到凭证——工具调用、结构化输出、中间步骤。这就像同时获得完成的文章和草稿纸上的演算过程。这对于调试、审计以及构建更丰富的用户界面非常有用。

响应会发出一系列**多态条目**。模型采取的操作顺序是清晰的。作为开发者，您可以选择显示、记录或完全忽略其中的哪些部分。

### 通过托管工具提升技术栈

在函数调用的早期阶段，我们注意到一个关键模式：开发者利用模型既调用API，也搜索文档库以引入外部数据源——这现在被称为RAG（检索增强生成）。但如果您是刚起步的开发者，从头构建检索管道是一项艰巨且成本高昂的任务。通过Assistants，我们引入了首批**托管**工具：`file_search`（文件搜索）和`code_interpreter`（代码解释器），使模型能够执行RAG并编写代码来解决您提出的问题。在Responses中，我们更进一步，增加了网络搜索、图像生成和MCP（模型上下文协议）。由于工具执行通过托管工具（如代码解释器或MCP）在服务器端进行，您无需将每次调用都回传至自己的后端，从而确保了更低的延迟和往返成本。

### 安全地保留推理过程

那么，为何要如此大费周章地隐藏模型的原始思维链（CoT）呢？直接公开CoT，让客户像对待其他模型输出一样处理，不是更简单吗？简而言之，公开原始CoT存在诸多风险：例如产生幻觉、生成最终响应中不会出现的有害内容，并且对OpenAI而言，还会带来竞争风险。

去年年底我们发布o1-preview时，首席科学家雅库布·帕乔茨基在博客中写道：

> 我们认为，隐藏的思维链为监控模型提供了独特的机会。假设它是忠实且清晰的，隐藏的思维链使我们能够“读取”模型的思维，理解其思考过程。例如，未来我们可能希望监控思维链中是否存在操纵用户的迹象。然而，要实现这一点，模型必须能够自由地以未经修改的形式表达其思想，因此我们不能在思维链上施加任何策略合规性或用户偏好训练。同时，我们也不希望将未对齐的思维链直接暴露给用户。

Responses通过以下方式解决这一问题：

* 在内部以加密形式保存推理过程，对客户端隐藏。
* 允许通过`previous_response_id`或推理项安全地继续对话，而无需暴露原始CoT。

## 为何`/v1/responses`是最佳构建方式

我们将Responses设计为**有状态、多模态且高效的**。

*   **智能体工具调用：** Responses API 让增强智能体工作流变得轻而易举，轻松集成文件搜索、图像生成、代码解释器和 MCP 等工具。
*   **默认状态保持。** 对话与工具状态自动追踪。这使得推理和多轮工作流显著简化。通过 Responses 集成的 GPT-5，在 TAUBench 上的表现相比 Chat Completions 提升了 5%，这完全得益于其保持的推理状态。
*   **原生多模态支持。** 文本、图像、音频、函数调用——皆为一级公民。我们并非将模态功能简单附加于文本 API 之上，而是从一开始就设计了足够多“房间”的房子。
*   **更低成本，更优性能。** 内部基准测试显示，其缓存利用率相比 Chat Completions 提升了 40–80%。这意味着更低的延迟和更低的成本。
*   **更优设计：** 我们从 Chat Completions 和 Assistants API 中汲取了大量经验，并在 Responses API 和 SDK 中进行了多项细节优化，包括：
    *   语义流式事件。
    *   内部标记的多态性。
    *   SDK 中的 `output_text` 辅助函数（无需再使用 `choices.[0].message.content`）。
    *   多模态与推理参数更合理的组织方式。

## Chat Completions 将何去何从？

Chat Completions 不会消失。如果它适合您，请继续使用。但如果您需要持久化的推理、原生体验的多模态交互，以及无需“胶带”修补的智能体循环——那么 Responses 是您未来的方向。

## 展望未来

正如 Chat Completions 取代了 Completions 一样，我们预计 Responses 将成为开发者使用 OpenAI 模型进行构建的默认方式。它能在您需要时保持简洁，在您期望时展现强大，并且足够灵活，足以应对未来任何新范式带来的挑战。

这是我们未来数年将基于其进行构建的 API。
