使用 Codex 运行长期任务

作者：Derrick Choi

2025 年 9 月，OpenAI 推出了 GPT-5-Codex，这是第一个为代理编码优化的 GPT-5 版本。2025 年 12 月，我们推出了 5.2，那是人们开始相信使用自主编码代理可以可靠的时刻。特别是，我们看到模型可靠地遵循指令的时长有了巨大的飞跃。

我想对这个阈值进行压力测试。所以我给 Codex 一个空白仓库、完全访问权限和一个任务：从头开始构建一个设计工具。然后我让它使用 GPT-5.3-Codex 以"超高"推理模式运行。Codex 连续运行了大约 25 小时，使用了大约 13M token，生成了大约 30k 行代码。

这是一个实验，而不是生产部署。但它在长期工作的重要方面表现良好：遵循规范、保持任务专注、运行验证，以及在过程中修复故障。

真正的转变是时间跨度

这不仅仅是"模型变得更聪明"。实际变化是代理可以保持更长时间的一致性，端到端完成更大的工作块，并在不丢失线索的情况下从错误中恢复。

METR 关于时间跨度基准的工作为这一趋势提供了一个有用的框架：前沿代理能够以约 50% 和 80% 的可靠性完成的软件任务长度一直在快速攀升，大致有约 7 个月的翻倍时间。请参阅测量 AI 完成长期任务的能力 (METR)。

我们最近的 GPT-5.3-Codex 发布公告在两个方面进一步推动了代理工作：

它在多步执行方面更好（计划 → 实施 → 验证 → 修复）。
在飞行中更容易引导而无需重置整个运行（方向修正不会抹掉进度）。

我还受到 Cursor 关于长时间运行的自主编码系统的写作的启发，包括他们构建浏览器的实验：Cursor 如何构建 Web 浏览器（扩展代理）。

Cursor 团队写道，OpenAI 模型"在扩展自主工作方面要好得多：遵循指令、保持专注、避免漂移、精确完整地实现事物。"

为什么 Codex 可以在长期任务中保持一致

长期工作更多不是关于一个巨大的提示，而是关于模型运行的代理循环。

在 Codex 中，循环大致是：

计划
编辑代码
运行工具（测试/构建/lint）
观察结果
修复失败
更新文档/状态
重复

这个循环很重要，因为它给代理：

真实反馈（错误、diff、日志）
外部化状态（仓库、文件、文档、工作树、输出）
随时间的可引导性（你可以根据结果进行方向修正）

这也是 Codex 模型在 Codex 表面上比通用聊天窗口感觉更好的原因：工具链提供结构化上下文（仓库元数据、文件树、diff、命令输出）并强制执行有纪律的"完成时"例程。

我们最近发表了一篇关于 Codex 代理循环的文章，其中有更多细节。

最重要的是，我们还推出了 Codex 应用程序，使该循环可以日常使用：

跨项目的并行线程（长期工作不会阻塞你的日常工作）
技能（标准化计划/实施/测试/报告）
自动化（后台例程工作）
Git 工作树（隔离运行，保持 diff 可审查，减少混乱）

我的测试设置

我为这个"实验"选择了一个设计工具，因为它是一个无情的测试：UI + 数据模型 + 编辑操作 + 大量边缘情况。你不能忽悠它。如果架构错了，它会很快崩溃。

我给 GPT-5.3-Codex 一个扎实的规范，以"超高"推理运行，它最终连续运行了约 25 小时，能够保持一致并提供高质量的代码。该模型还为每个完成的里程碑运行了验证步骤（测试、lint、类型检查）。

关键理念：持久项目内存

最重要的技术是持久项目内存。我将规范、计划、约束和状态写在 Codex 可以反复访问的 markdown 文件中。这防止了漂移并保持了"完成"的稳定定义。

仓库链接如下，文件堆栈如下：

Prompt.md（规范 + 可交付成果）

目的：冻结目标，这样代理就不会"构建令人印象深刻但错误的东西"。

文件中的关键部分：

目标和非目标
硬约束（性能、确定性、UX、平台）
可交付成果（完成时必须存在的内容）
"完成时"（检查 + 演示流程）

初始提示告诉 Codex 将提示/规范文件视为完整项目规范，并生成基于里程碑的计划：

Plan.md（里程碑 + 验证）

目的：将开放式工作转化为代理可以完成和验证的检查点序列。

文件中的关键部分：

足够小以在一个循环中完成的里程碑
每个里程碑的验收标准和验证命令
停止和修复规则：如果验证失败，在继续之前修复
决策笔记以避免振荡
代码库的预期架构

请注意，我们最近为 Codex 应用、CLI 和 IDE 扩展添加了原生计划模式。这有助于在做出更改之前将较大的任务分解为清晰的、可审查的步骤序列，这样你就可以预先对齐方法。如果需要额外澄清，Codex 会提出后续问题。要打开它，使用 /plan 斜杠命令。

Implement.md（引用计划的执行说明）

目的：这是运行手册。它准确告诉 Codex 如何操作：遵循计划，保持 diff 有范围，运行验证，持续更新文档。

文件中的关键部分：

Plans markdown 文件是真理来源（逐里程碑）
每个里程碑后运行验证（立即修复失败）
保持 diff 有范围（不要扩大范围）
持续更新文档 markdown 文件

Documentation.md（状态 + 决策，因为它已发货）

目的：这是共享内存和审计日志。这就是我如何可以离开几个小时仍然理解发生了什么。

文件中的关键部分：

当前里程碑状态（什么是完成的，什么是下一步）
做出的决策（以及为什么）
如何运行 + 演示（命令 + 快速冒烟测试）
已知问题 / 后续事项

长期 Codex 任务的要点

使这次运行成功的原因不是单个聪明的提示。它是以下组合：

清晰的目标和约束（规范文件）
带有验收标准的检查点里程碑（plans.md）
代理应该如何操作的手册（implement.md）
持续验证（测试/lint/类型检查/构建）
实时状态/审计日志（documentation.md），以便运行保持可检查

这就是长期编码工作的方向：更少的看管，更多的委托和护栏。

在你自己的长期任务上尝试 Codex

这次 25 小时的 Codex 运行是编码构建方向的预览。我们正在超越单次提示和紧密的结对编程循环，走向可以端到端承担真正工作片的长期队友，而你可以在里程碑处引导，而不是微观管理每一行。

我们对 Codex 的方向很简单：更强的队友行为、更紧密的与真实上下文的集成，以及保持工作可靠、可审查和易于发货的护栏。我们已经看到开发者在代理吸收常规实施和验证时移动得更快，让人类可以专注于最重要的事情：设计、架构、产品决策和没有模板的全新问题。

而且这不会仅限于开发人员。随着 Codex 在捕获意图和提供安全脚手架（计划、验证、预览、回滚）方面变得更好，更多的非开发人员将能够在不生活在 IDE 中的情况下构建和迭代。Codex 表面和模型会有更多的东西，但北极星保持不变：让代理感觉不像你照看的工具，而更像你可以信任长期工作的队友。
