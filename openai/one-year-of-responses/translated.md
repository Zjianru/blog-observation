# 从提示到产品：Responses API 一周年回顾

来源：https://developers.openai.com/blog/one-year-of-responses

---

一年前，我们推出了 [Responses API](/api/reference/responses/overview) —— 为开发者和企业构建实用可靠智能体奠定了基石。通过为模型配备一系列托管工具，人工智能得以从聊天助手演变为能够代表用户执行任务的系统。如今，Responses API 已支持多种工具以驱动智能体工作流，并推出一系列专为更强大模型构建而设计的新功能和基础组件。

目前，已有数千名开发者正在使用 Responses API 来提升各行各业的效率，涵盖[客户支持](https://openai.com/index/klarna/)、[法律](https://openai.com/index/hebbia/)、[生命科学](https://openai.com/index/gpt-5-amgen/)、[旅游](https://openai.com/index/booking-com/)等领域。在分享了这些行业的众多成功案例后，今天我们聚焦五位开发者过去一年基于 Responses API 构建的、较少被讲述的创新故事。

## 检测与修复 AI 智能体故障

_作者：Raindrop AI 的 Alexis Gauba 与 Ben Hylak_

**使用工具：** 自定义构建工具  
**模型版本：** GPT-5.2（正在测试 GPT-5.4）

Raindrop 是全球最具雄心的 AI 公司所采用的监控平台，用于捕捉其智能体在生产环境中出现的异常行为。随着智能体日益复杂，这类故障的影响也愈发关键。

> 如果没有 Responses API，构建此类监控系统将困难得多，且可靠性会大幅降低。

该系统通过 Responses API（借助 Vercel AI SDK）运行后台分析，在不同模型提供商间共享工具，并保持系统在多环境间的可移植性。这些工作流能揭示异常行为。一旦出现问题，系统会立即向开发者发出警报，并协助诊断根本原因。

您的浏览器不支持视频播放标签。

该平台聚焦三大核心系统：

  1. 智能体行为监控  
  2. 故障检测与警报  
  3. 开发者调查与调试工具

这些系统共同帮助团队在AI智能体影响生产系统之前，发现、追踪并修复问题。

### 监控架构

![Raindrop监控架构](one-year-of-responses_00.jpg)

该架构使团队能够持续监控智能体行为，并在问题发生时迅速响应。

### 1. 智能体行为监控

系统持续评估智能体行为，以判断其是否按预期运行。

开发人员可以为不良结果设置条件，当这些条件满足时，平台会发出警报。

### 2. 故障检测与告警

一旦检测到异常，Raindrop会通知开发人员，并提供调查问题所需的相关上下文。

该平台提供以下工具：

  * 追踪不同智能体版本间的行为变化
  * 识别导致故障的具体提示或系统变更
  * 检查推理轨迹和工具调用记录

这使得开发人员能够快速定位故障根源并部署修复方案。

### 3. 调查与调试工具

Raindrop还提供帮助开发人员诊断智能体工作流问题的工具。这些功能让团队能够将故障检测与系统改进联系起来。

Raindrop AI利用Responses API驱动所有长期运行的背景分析工作流。没有它，实现这些监控系统将变得异常困难。

## 面向复杂数据的深度推理工作流

_作者：[Repo Prompt](https://repoprompt.com)的Eric Provencher_

**使用工具：** Codex（配合应用服务器 + MCP）、网络搜索  
**使用模型：** GPT-5.3 Codex

> 我们不再让推理模型在规划或审查过程中消耗上下文窗口来导航信息，而是利用独立的智能体预先整理上下文，让推理模型能将其大部分推理能力专注于解决任务本身。

Eric Provencher构建的系统帮助开发者和研究人员对大量文档、代码库及数据集进行深度分析。

[Repo Prompt](https://repoprompt.com) 专注于语境工程——自动收集、整理并构建相关信息，使推理模型能够高效分析。

尽管许多智能体系统侧重于数据收集，但埃里克的架构将语境收集与深度推理分离。该系统利用智能体工作流来整合相关语境，然后将这些精心整理的信息交给专门负责分析的推理模型。

该平台运用 OpenAI Responses API 来编排长时间运行的智能体工作流和推理任务，涵盖以下场景：

  * 大型代码库分析与架构规划
  * 深度代码审查工作流
  * 海量文档集合的研究分析
  * 医学与科学文献分析

系统围绕三大核心组件构建：语境构建智能体工作流、深度推理模型（“先知”工作流）以及迭代式研究与分析循环。

### 1. 语境构建智能体工作流

您的浏览器不支持视频标签。

系统的第一阶段是语境构建智能体。该工作流通过分析大规模数据仓库，确定与特定查询相关的信息。

借助工具与模型推理能力，结合 Responses API，智能体能识别相关文件、文档间的关联性以及关键信息段落。

此阶段的输出是一个结构化的语境包，它将作为推理阶段的输入。

### 2. “先知”深度推理工作流

![Repo Prompt 先知工作流示意图](one-year-of-responses_01.png)

与语境构建智能体不同，“先知”模型（即深度推理模型）不执行工具调用或额外信息检索。它完全专注于分析所提供的精炼语境。

通过将研究与推理分离，该模型能够将全部推理能力集中于问题理解。在许多工作流中，推理阶段可持续运行较长时间，深入分析所提供语境中的复杂关联。

### 3. 迭代式研究与分析循环

该系统还支持迭代推理循环。在推理模型生成输出后，另一智能体可对结果进行审查，并判断是否需要进一步调查。

如有必要，系统将启动新一轮上下文收集与推理流程。这种循环机制使得系统能够开展长期调查任务，并持续优化分析结果。

#### 迭代工作流

![Repo Prompt 迭代工作流](one-year-of-responses_02.jpg)

该系统依托响应式API的多项核心能力：

  * 后台任务：支持运行持续数分钟乃至数小时的长期推理任务
  * 智能体编排：协调上下文收集、推理与验证的智能体循环流程
  * 可观测性：实时监控并管理执行中的长期推理工作流

平台采用Codex模型收集并结构化相关上下文，随后将精炼后的上下文传递给高性能推理模型进行深度分析。这些能力共同构成了平台融合智能体工作流与深度推理模型的混合架构。

## 黑胶唱片收藏家的对话式交互界面

_作者：Ash Ryan Arnwine（来自[Collxn](https://www.collxn.com)）_

**工具：** 网络搜索及16款定制工具  
**模型：** GPT-5.4、GPT-5 nano

> 相较于构建完整的检索增强生成系统等替代方案，响应式API显著减轻了我的工作负担。

Ash Ryan Arnwine 创建的 [“Collxn”](https://www.collxn.com)（意为“收藏”）是一项小而精的服务，其宏大的使命是：帮助黑胶收藏家重新发现架上已有的唱片，并与他们的藏品进行互动。

收藏家们常在Discogs平台管理庞大的唱片库，有时藏品规模高达数千张。Collxn通过接入这些收藏数据，每日发送名为“每日推荐”的邮件，重点介绍一张特定唱片及其艺术家详情，帮助收藏家重温已拥有的音乐。

由于翻阅唱片时若能随时提问会更有趣味，Collxn运用OpenAI响应式API驱动聊天界面，让用户能够直接与自己的唱片对话。

您的浏览器不支持视频标签。

### 对话式工具调用界面

该应用通过响应式API提供名为“询问此掉落”的聊天界面，用户可针对每日掉落中的唱片记录提出问题。

模型配置了Discogs API工具访问权限，能在回答问题时直接从Discogs获取信息。

例如，用户可以询问：

*   这张唱片当前市场价格是多少？
*   这位艺术家还发行过哪些专辑？
*   这个压制版本有多稀有？

![询问此掉落界面](one-year-of-responses_03.png)

_“询问此掉落”为Collxn用户提供与黑胶唱片收藏的对话界面。_

收藏者只需提出问题，即可获得结合实时Discogs数据与个人唱片收藏背景生成的答案。

这种方式将静态的唱片收藏转化为连接更广阔音乐生态的对话式体验。

### 每日掉落与艺术家动态

Collxn还运用OpenAI智能体SDK，为每日掉落邮件中的推荐艺术家生成“近期动态”板块。

![Collxn每日掉落艺术家动态板块](one-year-of-responses_04.png)

_OpenAI智能体SDK驱动着Collxn每日掉落的艺术家动态板块。_

该功能通过部署支持网络搜索的智能体，查找艺术家近期文章或动态，并将这些背景信息整合至每日邮件。在测试用户中，动态功能因其能以动态方式将唱片收藏体验与外部世界连接，迅速成为产品最受欢迎的组成部分。

最终Ash将Collxn迁移至响应式API以推出“询问此掉落”。此举使应用能够支持对话式工作流中的多步推理，以及内置与自定义工具调用。Collxn的响应式API实施方案除运用内置网络搜索工具实现聊天框内艺术家动态查询外，还配备了16个自定义工具，用于操作Discogs API、查询用户Collxn账户等功能。

![Collxn每日掉落艺术家动态板块](one-year-of-responses_05.png)

_响应式API的网络搜索工具为Collxn“询问此掉落”中的实时艺术家动态查询提供支持。_

Responses API中的有状态对话也让多轮聊天交互的处理变得更简单、更快速。总体而言，Ash指出，与构建完整的检索增强生成（RAG）系统相比，使用Responses API简化了架构。

## 将屏幕录制转化为交互式产品演示

_来自[Arcade](https://www.arcade.software)团队的Nick Sorrentino和Pawel Wszola_

**工具：** 计算机使用  
**模型：** GPT-5.2、computer-use-preview  

> 集成API驱动的内容生成将发布演示所需的步骤减少了50%，显著提高了发布率和采用率。

Arcade将大多数团队已经在做的事情——录制屏幕——转化为精美的交互式产品演示。团队无需现场演示产品或编写逐步操作文档，只需录制一次工作流程，其余工作由Arcade处理。

在底层，平台分析录制内容并自动生成引导式演练，解释每个步骤中发生的情况。

您的浏览器不支持视频标签。

### 演示生成工作流程

在录制过程中：

  1. 用户在执行工作流程时录制屏幕。
  2. 在桌面端或浏览器中，Arcade直接捕获结构化交互，如点击、输入和滚动。
  3. 在移动端，由于iOS沙盒机制阻止应用捕获系统级交互，用户改为录制应用的纯屏幕视频。
  4. 录制内容通过computer-use工具发送至OpenAI Responses API，该工具分析视觉帧并推断发生的交互。
  5. 系统将这些推断出的操作转换为结构化步骤。
  6. Arcade生成叙述文本和交互式热点，引导观众浏览演示。

这些步骤自动转化为用户看到的交互式演练。

结构化操作随后传递给Chat Completions API，生成演示中显示的标题和热点描述。用户可以使用内置的AI编辑工具调整生成的文本，例如缩短或重写内容。

### 将演示制作时间减半

自动化演示旁白显著降低了发布产品导览所需的工作量。

集成API驱动的工作流程后：

* 发布前所需操作次数的中位数下降了50%
* P80操作次数从约230次降至约120次
* 发布率和产品采用率均实现增长

通过消除演示制作流程中的阻力，Arcade让团队能够更快速地将原始录制内容转化为精美的交互式演示。

## 衡量并提升AI输出中的品牌可见度

_作者：[Hexagon](https://joinhexagon.com)的Tunde Adeyinka与Ramon Silva_

**使用工具：** 网络搜索  
**使用模型：** GPT-5.2 Chat

Tunde Adeyinka与Ramon Silva创立Hexagon，旨在为零售商解答一个新问题：*AI助手如何描述您的产品？*

随着AI助手日益影响产品发现过程，Hexagon帮助企业监控其品牌在AI生成答案中的呈现方式，并持续优化这些结果。

该平台基于OpenAI Responses API构建了三大核心系统：

### 1. 响应模拟架构

Hexagon每日运行模拟流程，以衡量AI助手如何回答产品相关问题。系统每天生成数千条真实的消费者提问、产品推荐提示和购物查询，并通过Responses API发送。通过分析返回的输出结果，追踪品牌在AI生成答案中的可见度。

零售客户可据此了解其产品的出现频率，以及这些回答随时间的变化趋势。

![Hexagon响应模拟架构](one-year-of-responses_06.png)

### 2. 多智能体内容生成流程

除分析功能外，Hexagon还运用Responses API生成优化内容，以提升品牌在AI回答中的可见度。

系统采用四智能体架构，每个智能体负责流程中的特定环节，将输出传递至下一阶段，直至最终内容生成并发布。智能体通过非确定性循环进行迭代优化，确保内容在发布前持续完善。

![Hexagon多智能体内容生成流程](one-year-of-responses_07.jpg)

### 3. 仪表板与客户工具

该平台还包含名为“Hexi”的聊天机器人，它通过Responses API的函数调用功能构建。借助Hexi，客户能以对话方式探索分析数据，并以自助服务形式生成AI可见性数据摘要。Hexagon通过零售商仪表板呈现分析结果，该仪表板可追踪产品在AI生成答案中的呈现情况。

![Hexagon仪表板截图](one-year-of-responses_08.png)

Hexagon依托Responses API的几项关键能力，确保其产品中的模拟既真实又实用：

  * 网络搜索：复现类似ChatGPT的联网浏览式回答
  * 用户位置参数：模拟不同地区的查询以测试地理差异
  * 推理强度：控制回答的深度与复杂度
  * 最大输出标记数：限制长文本输出的回答长度
  * 上下文持久性：在多次调用间保持上下文，实现多智能体工作流

> Responses API提供了更优的响应质量与更强的跨调用上下文持久性，这对支撑Hexagon平台的多步骤流程至关重要。

## 总结

一年来，[Responses API](/api/reference/responses/overview)已成为开发者构建智能体软件的核心基石。

这五个开发者案例展示了实践中的具体应用：多智能体系统协调工具、检测漏洞、运行工作流，以及交付由AI驱动的产品。

平台本身正在快速演进——通过新增功能实现更优的编排和[更丰富的工具生态](/api/docs/guides/tools)，例如具备网络功能的OpenAI托管容器和[命令行工具](/api/docs/guides/tools-shell)。
