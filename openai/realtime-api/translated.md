# 关于实时API的开发者笔记

来源：https://developers.openai.com/blog/realtime-api

---

我们最近[宣布](https://openai.com/index/introducing-gpt-realtime/)了最新的语音转语音模型`gpt-realtime`，同时实时API和一系列新API功能也已全面开放。实时API和语音转语音（s2s）模型现已正式进入通用可用性（GA）阶段，在模型质量、可靠性和开发者体验方面均有显著提升。

虽然您可以在[文档](https://platform.openai.com/docs/guides/realtime)和[API参考](https://platform.openai.com/docs/api-reference/realtime)中了解新的API功能，但我们希望重点介绍一些您可能遗漏的功能，并提供使用场景建议。如果您正在集成实时API，希望这些笔记能为您带来启发。

## 模型改进

新模型包含多项改进，旨在更好地支持生产级语音应用。本文主要关注API变更。为了更好地理解和使用该模型，我们推荐阅读[发布博客文章](https://openai.com/index/introducing-gpt-realtime/)和[实时提示指南](/cookbook/examples/realtime_prompting_guide)。不过，我们仍将指出一些具体细节。

使用该模型的几点关键建议：

*   在[实时交互平台](https://platform.openai.com/playground/realtime)中尝试提示词设计。
*   使用 `marin` 或 `cedar` 语音可获得最佳助手音质。
*   针对新模型重写提示词。由于指令遵循能力的提升，具体指令现在效果显著增强。
    *   例如，旧模型可能将“当Y时始终说X”这类提示视为模糊指引，而新模型可能会在意料之外的情境中严格执行该指令。
    *   请注意您提供的具体指令。默认所有指令都将被遵循。

## API 结构变更

随着正式版发布，我们更新了实时API结构，这意味着当前存在测试版接口和正式版接口。建议客户端迁移至正式版接口进行集成，以获取新功能特性，测试版接口最终将停止维护。

迁移所需完整变更清单可在[测试版至正式版迁移文档](https://platform.openai.com/docs/guides/realtime#beta-to-ga-migration)中查阅。

您仍可通过测试版接口访问新的 `gpt-realtime` 模型，但部分功能可能不受支持。详见下文说明。

### 功能可用性

实时API正式版包含多项新功能。其中部分功能在旧模型上可用，部分则不可用。

功能特性| GA 模型| Beta 模型
---|---|---
图像输入| ✅| ❌
长上下文| ✅| ✅
异步函数调用| ✅| ❌
提示词| ✅| ✅
MCP| ✅ _配合异步函数调用效果最佳_|  ✅ _无异步函数调用时功能受限*_
音频令牌转文本| ✅| ❌
欧盟数据驻留| ✅| ✅ _仅限06-03版本_
SIP| ✅| ✅
空闲超时| ✅| ✅

*由于 Beta 模型不支持异步函数调用，未返回输出的待处理 MCP 工具调用可能无法被模型妥善处理。建议搭配 GA 模型使用 MCP。

### 温度参数调整

GA 接口已移除 `temperature` 作为模型参数，Beta 接口将温度值限制在 `0.6 - 1.2` 范围内，默认值为 `0.8`。

您可能会问：“为何不允许用户自由设置温度参数，例如通过调低温度使输出更确定？” 这是因为在此模型架构中，温度参数的作用机制有所不同，将温度设为建议值 `0.8` 几乎总能获得最佳效果。

根据我们的观察，降低温度参数无法使音频响应变得确定，而提高温度则会导致音频畸变。建议通过调整提示词来探索控制模型行为表现的方法。

## 新增功能

除了从 Beta 版到 GA 版的变更外，我们还为实时 API 新增了多项功能。

所有功能详情均可在[文档](https://platform.openai.com/docs/guides/realtime)和[API参考](https://platform.openai.com/docs/api-reference/realtime)中查阅，但在此我们将重点说明在集成与迁移过程中如何理解新功能。

### 对话空闲超时

对于某些应用场景，用户长时间无输入可能不符合预期。以电话通话为例——若长时间未听到对方声音，我们会主动询问对方状态。可能是模型未捕捉到用户发言，也可能是用户不确定模型是否仍在讲话。为此我们新增了一项功能，可自动触发模型说出类似“您还在吗？”的提示语。

通过在语音端点检测（VAD）的`server_vad`设置中配置`idle_timeout_ms`即可启用此功能。超时计时将从最后一条模型响应音频播放完毕后开始计算——即超时值设定为`response.done`时间加上音频播放时长再加上超时等待时间。若在该时段内未触发VAD，则启动超时机制。

超时触发时，服务器将发送[`input_audio_buffer.timeout_triggered`](https://platform.openai.com/docs/api-reference/realtime-server-events/input_audio_buffer/timeout_triggered)事件，随后将空白音频片段提交至对话历史记录并触发模型响应。提交空白音频可使模型有机会检测VAD是否失效，以及相关时段内是否存在用户语音输入。

客户端可以按如下方式启用此功能：

    {
      "type": "session.update",
      "session": {
        "type": "realtime",
        "instructions": "你是一个有用的助手。",
        "audio": {
          "input": {
            "turn_detection": {
              "type": "server_vad",
              "idle_timeout_ms": 6000
            }
          }
        }
      }
    }

### 长对话与上下文处理

我们已调整了Realtime API处理长会话的方式。请注意以下几点：

  * Realtime会话现在最长可持续60分钟，此前为30分钟。
  * `gpt-realtime`模型的令牌窗口为32,768个令牌。响应最多可消耗4,096个令牌。这意味着模型的最大输入为28,672个令牌。
  * 会话指令及工具的最大长度为16,384个令牌。
  * 当会话达到28,672个令牌时，服务将自动截断（丢弃）消息，但此设置可配置。
  * GA服务将在有转录文本时自动丢弃部分音频令牌以节省令牌使用。

#### 配置截断设置

当对话上下文窗口达到令牌限制时，Realtime API 会在超过限制后自动开始截断（丢弃）会话中最早的消息。您可以通过设置 `"truncation": "disabled"` 来禁用此截断行为，这样当输入令牌过多时，系统会直接返回错误。然而，截断功能非常有用，因为它能确保即使输入内容超出模型处理范围，会话仍可继续进行。Realtime API 不会对丢弃的消息进行摘要或压缩处理，但您可以自行实现这一功能。

截断的一个负面影响是，改变对话开头的消息会破坏[令牌提示缓存](https://platform.openai.com/docs/guides/prompt-caching)。提示缓存的工作原理是识别提示前缀中完全相同的匹配内容。在后续的每一轮对话中，只有未发生变化的令牌会被缓存。当截断操作改变了对话的开头部分时，可缓存的令牌数量就会减少。

我们已实施一项功能来减轻这种负面影响，具体做法是在进行截断时，采取比必要程度更激进的截断策略。将保留比例设置为`0.8`，意味着将截去上下文窗口20%的内容，而非仅截取刚好使输入标记数低于上限的部分。其核心理念是：与其每次小幅截断，不如一次性截除更多上下文窗口内容，从而减少缓存失效的频率。这种缓存友好型方法有助于在长时间会话达到输入限制时控制成本。

    {
      "type": "session.update",
      "session": {
        "truncation": {
          "type": "retention_ratio",
          "retention_ratio": 0.8
        }
      }
    }

### 异步函数调用

响应API要求在函数调用后立即返回函数响应，而实时API则允许客户端在函数调用待处理期间继续会话。这种延续性优化了用户体验，使实时对话能自然流畅地进行，但模型有时会幻觉出并不存在的函数响应内容。

为解决此问题，GA响应API新增了经过实验评估与调优的占位响应机制，确保模型在等待函数响应期间仍能保持优雅表现。若向模型询问函数调用结果，它将回复诸如“我仍在等待该结果”之类的提示。该功能已在新模型中自动启用——您无需进行任何更改。

### 欧盟数据驻留

现已专门为 `gpt-realtime-2025-08-28` 和 `gpt-4o-realtime-preview-2025-06-03` 模型提供欧盟数据驻留支持。数据驻留功能需在组织层面明确启用，并通过 `https://eu.api.openai.com` 域名进行访问。

### 追踪功能

Realtime API 将实时会话中的关键事件记录为追踪日志，并保存至[开发者控制台](https://platform.openai.com/logs?api=traces)，便于问题排查与调试。随着正式版发布，我们新增了以下事件类型：

  * 会话更新（当 `session.updated` 事件发送至客户端时触发）
  * 输出文本生成（记录模型生成的文本内容）

### 托管式提示词

现在您可以通过[在 Realtime API 中使用提示词](https://platform.openai.com/docs/guides/realtime-models-prompting#update-your-session-to-use-a-prompt)，便捷地实现应用代码与可独立编辑的提示词解耦。提示词既包含指令内容，也涵盖会话配置（如对话轮次检测设置）。

您可以在[实时交互平台](https://platform.openai.com/audio/realtime)创建提示词，根据需求进行迭代更新和版本管理，随后客户端可通过提示词ID进行调用，示例如下：

{
  "type": "session.update",
  "session": {
    "type": "realtime",
    "prompt": {
      "id": "pmpt_123", // 您存储的提示ID
      "version": "89", // 可选：锁定特定版本
      "variables": {
        "city": "Paris" // 示例变量，供您的提示使用
      }
    },
    // 您仍可直接设置会话字段；若与提示字段重叠，这些字段将覆盖提示字段：
    "instructions": "清晰简洁地表达。在采取行动前确认理解。"
  }
}

如果提示设置与传递给会话的其他配置发生重叠，如上例所示，会话配置将优先生效。因此，客户端既可使用提示的配置，也可在会话时进行调整。

### 旁路连接

Realtime API允许客户端通过WebRTC或SIP直接连接到API服务器。然而，您很可能希望将工具使用及其他业务逻辑保留在应用服务器上，以确保这些逻辑的私密性和客户端无关性。

通过旁路控制通道连接，将工具使用、业务逻辑及其他细节安全地保留在服务器端。我们现在为SIP和WebRTC连接均提供了旁路选项。

旁路连接意味着同一实时会话中存在两个活跃连接：一个来自用户客户端，另一个来自您的应用服务器。服务器连接可用于监控会话、更新指令以及响应工具调用。

如需了解更多信息，请参阅[边带连接文档](https://platform.openai.com/docs/guides/realtime-server-controls)。
