# Codex 如何助力 OpenAI DevDay 2025

本周，我们在旧金山圆满结束了第三届也是规模最大的一届 OpenAI DevDay。这次盛会是公司全体员工辛勤工作的成果。但随着 DevDay 的临近，讨论中反复出现一句话：“没有 Codex，我根本做不到这些。”今年是首次有 Codex 参与的 DevDay。在我们构建的所有环节中都用到了它：从舞台演示（甚至那些与 Codex 无关的演示），到社区大厅的街机游戏机，再到产品本身，Codex 都是打造 2025 DevDay 的关键部分。以下简要介绍 Codex 在幕后如何帮助我们节省时间、解决问题、处理多任务、确定优先级以及进行组织协调的几种方式。

## 控制摄像头与创建场地灯光 MCP

让我们从最引人注目的项目开始：Romain Huet 的 Codex 主题演讲演示。如果你错过了，可以在这里查看。正如 Romain 提到的，除了使用我们的 Realtime agents 入门应用之外，你在演示中看到的一切都是由 Codex 构建的。

这个演示实际上源于一个想法：希望展示 Realtime 如何控制观众席的摄像头和灯光。但当 Romain 开始深入研究这个项目时，他面临着以编程方式控制摄像头和灯光的挑战。Codex 能够找到使用 VISCA 协议（一种 90 年代初的协议！）控制支持网络摄像头的解决方案，完全独立地实现了该协议，甚至继续构建了一个 MCP 服务器来控制灯光的协议。

使用 Codex CLI，Romain 能够并行处理这两个问题，并在一个下午就让初始版本启动并运行，而无需触碰键盘——避免了原本可能需要的大量研究和调试工作。

## 带来节拍

DevDay 上的一大发布是 Apps SDK，它让你可以直接在 ChatGPT 内构建丰富的应用体验。对于 Katia Gil Guzman 的“开发者国情咨文”演示，想法是在 Codex 为 Romain 构建的灯光 MCP 服务器基础上，构建一个丰富的节拍垫界面。这意味着要构建一个视觉上吸引人且功能正常的界面，包括处理与灯光 MCP 服务器的连接以控制灯光并允许其演奏不同的乐器。

得益于 Codex Cloud 和 best-of-N 功能，Katia 不仅能够快速获得一个功能正常的应用，还能并行迭代多个不同的设计。她尝试了从更具未来感的现代外观到更符合 OpenAI DevDay 品牌形象的 UI 等各种方案，甚至尝试了不同的功能，所有这些都没有浪费时间和精力。

## 多任务游戏设计

如果你在 DevDay 的走廊里漫步，可能会看到 ArcadeGPT，这是两个街机游戏机，让你可以通过使用 GPT-5 混音现有视频游戏集合来自定义自己的视频游戏。当 Kevin Whinnery 开始构建基础时，他需要一系列供 GPT-5 混音的初始游戏——而且他需要快速获得。

为了快速创建并迭代这些游戏，他打开了七个（！！）不同的终端，每个终端都有一个 Codex CLI 实例，分别处理一个单文件的 Phaser 游戏实现。得益于 Codex CLI，他可以异步迭代每个游戏，同时测试它们，为与会者提供广泛的游戏进行游玩和混音。

## 重建演示应用

就我个人而言，在 DevDay 前的几乎所有任务中，我都使用了 Codex。很难一一列举我感谢 Codex 的每一个时刻，但有一个特别突出。我一直在为我的 Open Models 演讲准备微调演示，并全程使用了 Streamlit。但 Streamlit 应用感觉复杂，观众难以理解，并且存在一些不易修复的行为错误。

在用 v0 截取了一些屏幕截图并创建了一个快速的初始设计后，我下载了模拟的 Next.js 应用，并让 Codex IDE 扩展开始工作。我要求它获取我的 Streamlit 应用，创建一个执行相同工作并将其连接到我的 Next.js 前端的 FastAPI 服务器。发出任务后，我去吃午饭，回来时已经有一个完全实现且正常工作的应用程序。

从那里开始，我能够让 Codex 处理额外的任务，创建额外的页面，帮助我更好地说明演示。没有 Codex，这个演示永远无法按时完成。

## 实现构想

Erika Kettleson 通过使用 Codex IDE 扩展，将整个展位演示变为现实，从而节省了时间。她从一个草图开始，将其输入 Codex 以创建初始 UI，甚至让 Codex 编写评估程序，帮助在权衡速度和质量的同时确定用于生成 SVG 的最佳模型。

Codex 帮助 Erika 评估了在演示中使用单代理或多代理架构的权衡，然后重构了整个代码库以转向单代理架构。在构建完所有内容后，Codex 创建了详细的 Mermaid 图表，Erika 在展位上用这些图表向人们解释应用的工作原理。

## 大规模审查

AgentKit 发布的一部分是我们新的 Python 和 TypeScript Guardrails SDK 的发布。这些 SDK 旨在与我们的 Python 和 TypeScript Agents SDK 以及 Agent Builder 配合使用。

为了确保开发者在使用这些 SDK 时获得良好体验，Kazuhiro (Kaz) Sera 加入项目以帮助项目完成最后阶段。他使用 Codex 快速熟悉了两个 SDK 的代码库，识别了他和 Codex 发现的一些错误的根本原因，使用 Codex CLI 和 IDE 扩展修复了它们，并利用 Codex 代码审查来识别任何未解决的错误。

得益于 Codex，他能够完成所有这些工作，帮助团队发布 SDK，同时使用相同的工具来完善我们在同一天发布的 ChatKit 示例应用。

## 同时处理多个项目

在 DevDay 前夕，我们很多人都在同时处理越来越多的项目。Codex 允许我们使用 IDE 扩展和 CLI 在本地和云端任务之间进行委派，从而同时处理多项任务。你经常会看到我们同时运行 3-4 个完全独立的任务。

例如，就我自己的情况而言，我让 Codex 同时执行以下任务：将 Jupyter notebook 支持构建到 gpt-oss 服务器中，重构并修复我的代理演示中的一些错误，重组一些 Codex 文档，以及调试我的微调运行。

为了在我们这边快速切换上下文，我们不会花很多时间精心设计正确的提示——相反，我们会用简短的句子向 Codex 描述问题，发出任务，立即切换到下一个任务，稍后再回来查看 Codex 的状态。甚至快速离开办公桌时，也养成了“让我再发送一个 Codex 任务”的习惯。

## 组织协调

为开发者发布多个新产品会带来大量新文档，在早期阶段，这些文档可能散落在各处：无论是在 GitHub 仓库内部、Google Docs 中，还是在 Notion 里。通常，这些文档会一直迭代到最后一刻。这次发布也不例外。

得益于 Codex Cloud，团队能够获取零散的文档，将它们交给 Codex，并粗略描述我们希望如何将它们拆分并组织到我们的文档中，然后让 Codex 处理其余的工作。Codex 分割了文件，将它们转换为 MDX 文件，设置了必要的导航结构，并开启了一个 PR，由于部署预览功能，我们可以与团队分享以进行审查和迭代。没有 Codex，这在 DevDay 前夕通常需要数小时（甚至数天）才能完成。

## 处理支线任务

最后，我们都经历过这种情况——你正在处理最重要的任务，但突然想起你一直计划要做的一件事，但你总是分心。DevDay 前夜也不例外。在彩排间隙，我们都在努力为这个大日子做好一切准备。

Katia 正准备上台彩排她的演示时，她意识到自己还没有像计划那样发布更新后的 404 页面。她快速在 Codex Web 上打开了另一个标签页，发送了一个任务，要求 Codex 实现一个新的 developers.openai.com/404 页面，同时使用 best-of-n 功能让 Codex 同时创建两个尝试。

在 Katia 五分钟后上台之前，她能够借助 Codex 中的预览截图审查这两个选项，快速查看页面，使用 IDE 扩展进行一些编辑，然后发布新设计的 404 页面。

## 仅仅触及表面

我们可能可以花几个小时谈论 Codex 如何帮助我们塑造 DevDay，更不用说它如何在日常工作中帮助我们每个人了——但这只是我们在 OpenAI 内部使用 Codex 的一个缩影。如果你想了解更多关于我们如何使用 Codex 以及一些最佳实践，请查看我们关于 Codex 的 DevDay 演讲或查阅我们的文档。
