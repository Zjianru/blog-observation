Shell + 技能 + 压缩：进行实际工作的长期运行代理的提示

Practical patterns for building with skills, hosted shell, and server-side compaction in the Responses API.

作者：Charlie Guo

我们正在从单轮助手转向处理真实知识工作的长期运行代理：读取大型数据集、更新文件和编写应用程序。

基于开发者反馈和我们自己构建 Codex 和内部代理的经验，我们正在发布一套新的代理原语，使长期工作更加实用：

技能（与代理技能开放标准对齐）：可重用的版本化指令，你可以挂载到容器中，以便代理更可靠地执行任务。
升级的 shell 工具：OpenAI 托管的容器，具有受控的互联网访问，代理可以在其中安装依赖项、运行脚本和写入输出（例如，报告和产物）。
服务器端压缩：自动压缩长期代理运行的一种简单方式，这样你就永远不会达到上下文限制。

文档和 API 参考分别涵盖了上述每个项目。这篇文章专注于我们目前在 OpenAI 的工作和 Glean（早期技能客户）的生产中看到的最佳实践的非显而易见提示和模式。

一个简短的思维模型

技能："程序"，模型可以按需加载

技能是文件束加上包含前言和指令的 SKILL.md 清单。想象：一本版本化的 playbook，模型可以在做真实工作时参考。

当技能可用时，平台将每个技能的名称、描述和路径暴露给模型。模型使用该元数据来决定是否调用技能。如果它调用了，它会读取 SKILL.md 获取完整工作流程。

Shell 工具：代理的"执行"

Shell 工具让模型在真实终端环境中工作，可以是：

OpenAI 托管的容器。
你自己执行的本地 shell 运行时（相同的工具语义，但你控制机器）。

托管 shell 通过 Responses API 运行，这意味着你的请求带有有状态工作、工具调用、多轮继续和产物。

压缩：保持长期运行移动

随着工作流程变长，它们会遇到上下文窗口限制。服务器端压缩通过管理上下文窗口和自动压缩对话历史来保持长期运行移动。

Responses API 中的压缩为你提供了两种处理方式：

服务器端压缩（新）：当上下文跨越阈值时，压缩在流中自动运行，因此没有单独的压缩调用。
独立压缩端点：当你想明确控制压缩发生的时间时，使用 /responses/compact。

为什么它们在一起更好

技能通过将稳定的过程和示例移动到可重用束中来减少提示意大利面。
Shell 提供了一个完整的执行环境，让你安装代码、运行脚本和写入输出。
压缩在长期运行中保持连续性，因此相同的工作流程可以继续执行而无需手动上下文操作。
 together，你获得具有真实执行的可重复工作流程，而不会将系统提示变成一个脆弱的巨大文档。

提示和技巧

1) 编写技能描述像路由逻辑（而不是营销文案）

你技能的描述实际上就是模型的决策边界。它应该回答：

我什么时候应该使用这个？
我什么时候不应该使用这个？
输出和成功标准是什么？

一个实用的模式是在描述中直接包含一个简短的"何时使用 vs. 何时不使用"块，并保持具体（输入、涉及的工具、预期的产物）。

2) 添加负面示例和边缘情况以减少误触发

一个令人惊讶的失败模式是让技能可用最初可能会降低正确触发。我们看到有效的修复之一是负面示例加上边缘情况覆盖。

在实践中，这意味着写一些明确的"不要在……时调用此技能"案例（以及应该怎么做）。这有助于模型更干净地路由，特别是当你有多个粗略看起来相似的技能时。

Glean 直接看到了这一点：在定向评估中，基于技能的路由最初将触发降低了约 20%，然后在他们添加了描述中的负面示例和边缘情况覆盖后恢复了。

3) 在技能内部放置模板和示例（未使用时基本上是免费的）

如果你一直在将模板塞进系统提示，停下来。

技能内部的模板和工作示例有两个优势：

它们在需要时正好可用（当技能被调用时）。
它们不会为不相关的查询膨胀 token。

这对知识工作输出特别有效，例如：

结构化报告。
升级分类摘要。
客户计划。
数据分析撰写。

Glean 报告说，这种模式在生产中推动了一些最大的质量和延迟收益，因为那些示例只在技能触发时加载。

4) 在设计初期就考虑长期运行与容器重用和压缩

长期代理很少作为单次提示成功。在开始时计划连续性：

当你想要稳定依赖项、缓存文件和中间输出时，在步骤之间重用相同的容器。
传递 previous_response_id 以便模型可以在同一线程中继续工作。
将压缩作为长期运行的默认原语使用，而不是紧急回退。

这种组合减少了重启行为，并在线程增长时保持多步作业一致。

5) 当你需要确定性时，明确告诉模型使用技能

默认行为是模型决定何时使用技能。这通常是你想要的。

但当你运行具有明确合约的生产工作流程时（你宁愿确定性而不是聪明），就说：

"使用 <skill name> 技能。"

这是你可以拉动的最简单可靠性杠杆。它将模糊路由变成明确合约。

6) 将技能加网络视为高风险组合（为 containment 设计）

这是很容易被忽略的安全提示，以后很难修复。

将技能与开放网络访问结合会产生高风险的数据泄露路径。如果你使用网络，保持网络允许列表严格，假设工具输出是不可信的，并避免在消费者面向的流程中开放互联网加强大程序，用户期望强有力的确认控制。

一个强大的默认姿势：

技能：允许
Shell：允许
网络：仅在请求时为狭窄范围的任务启用最小允许列表

7) 使 /mnt/data 成为产物的交接边界

对于托管 shell 工作流程，将 /mnt/data 视为写入你要检索、审查或传递到后续步骤的输出的标准位置。示例包括报告、清理的数据集和完成的电子表格。

一个好的思维模型：工具写入磁盘，模型推理磁盘，开发者从磁盘检索。

8) 理解允许列表作为一个两层系统（组织级别和请求级别）

网络在两个地方控制：

组织级允许列表（由管理员配置），设置最大允许目的地。
请求级 network_policy，必须是组织允许列表的子集。

两个在操作上重要的含义：

保持组织允许列表小而稳定（"你信任的已批准目的地"集）。
保持请求允许列表更小（"这个工作需要的目的地"集）。

如果请求包含组织允许列表之外的网络域，它将出错。

9) 使用 domain_secrets 进行身份验证调用（避免凭据泄露）

如果允许的域需要 auth 头，请使用 domain_secrets 以便模型永远不会看到原始凭据。

在运行时，模型看到占位符（例如 $API_KEY），sidecar 仅针对批准的目的地注入真实值。每当你的代理需要从容器内调用受保护的 API 时，这是一个强大的默认设置。

10) 在云和本地使用相同的 API

你可以使用这两个原语而不必承诺托管一切：

技能与托管 shell 和本地 shell 模式一起工作。
Shell 有一个本地执行模式，你可以在其中自己执行 shell_call 并将 shell_call_output 返回给模型。
如果你使用的是 Agents SDK，你也可以插入你自己的 shell 执行器。

一个实用的开发循环看起来像：

开始本地（快速迭代，访问内部工具，易于调试）。
当想要可重复性、隔离和部署一致性时，移动到托管容器。
跨两种模式保持技能相同（工作流程在执行移动时保持稳定）。

三种构建模式

虽然你应该自由地尝试这些新的代理原语，但这里有三个如何组合它们来构建有用应用程序的示例。

模式 A：安装 -> 获取 -> 写入产物

这是从托管 shell 中受益的最简单方式：代理安装依赖项、获取外部数据并产生具体可交付成果。

例如：

安装几个库。
抓取或调用 API。
将报告写入 /mnt/data/report.md。

这种模式是真实工作代理的基础，因为它创建了一个清晰的审查边界：你的应用可以向用户显示产物、记录它、diff 它，或将其反馈到后续步骤。

模式 B：技能 + shell 用于可重复工作流程

一旦你构建了一两个成功的 shell 工作流程，你会注意到下一个问题：这有效，但当提示漂移时可靠性会下降。

这就是技能的作用所在。遵循一个持久的结构：

在工作流程（步骤、护栏、模板）中编码技能。
将技能挂载到你的 shell 环境中。
让代理遵循技能确定性地产生产物。

这对于以下工作流程特别有效：

电子表格分析或编辑。
数据集清理加上摘要生成。
 recurring 业务流程的标准化报告生成。

模式 C（高级）：技能作为企业工作流程载体

我们看到的一个早期模式是在单工具调用和多工具编排之间的差距中丢失准确性。技能可以通过使工具推理更加程序化来弥合这一差距，而不膨胀系统提示。

来自 Glean 的一个具体例子：

面向 Salesforce 的技能将评估准确性从 73% 提高到 85%，并将首个 token 的时间减少了 18.1%。
实用策略包括仔细路由、负面示例和在技能内部嵌入模板/示例。
Glean 还使用技能来编码企业工作流程中的 recurring 任务，包括客户计划、升级分类和品牌对齐的内容生成。

这就是事物变得强大的形态。技能成为活的 SOP（标准操作程序）：随着你的组织发展而更新，并由代理一致地执行。

构建一次，随处运行

当长期代理能够遵循程序并在计算机上做真实工作时，它们会变得非常有用了。技能、托管 shell 和压缩创造了那个基础。回顾：

使用技能编码如何（过程、模板、护栏）。
使用 shell 执行做（安装、运行、写入产物）。
使用压缩保持长期运行一致（无需手动管理上下文）。
当快速迭代时在本地开始。
当想要可重复、隔离的执行时移动到托管容器。
通过组织级和请求级允许列表保持网络锁定，并使用 domain secrets 进行身份验证调用。

在你的应用程序中开始。看技能文档、shell 文档和压缩文档来学习如何做。
