# Shell + 技能 + 压缩：为执行实际工作的长期运行智能体提供技巧

来源：https://developers.openai.com/blog/skills-shell-tips

---

我们正从单轮对话助手转向能够处理实际知识工作的长期运行智能体：读取大型数据集、更新文件以及编写应用程序。

基于开发者反馈和我们构建Codex及内部智能体的经验，我们发布了一套新的智能体基础组件，使长期任务变得更加可行：

  * [**技能**](/api/docs/guides/tools-skills)（符合Agent Skills开放标准）：可复用、版本化的指令集，可挂载到容器中，使智能体能够更可靠地执行任务。
  * 升级版[**Shell工具**](/api/docs/guides/tools-shell)：一个由OpenAI托管的容器，具备受控的网络访问权限，智能体可在其中安装依赖、运行脚本并输出成果（例如报告和制品）。
  * [**服务端压缩**](/api/docs/guides/context-management)：一种自动压缩长周期智能体运行的简便方法，确保永远不会触及上下文长度限制。

文档和API参考分别涵盖了上述各项内容。本文重点介绍我们在OpenAI内部工作以及早期技能客户Glean的生产实践中，迄今为止最有效的非显性技巧和模式。

## 简要心智模型

### 技能：模型可按需加载的“流程”

技能是一个文件包，包含一个带有前言和指令的`SKILL.md`清单。可以理解为：一个版本化的操作手册，模型在执行实际工作时可随时查阅。

当技能可用时，平台会向模型公开每个技能的`名称`、`描述`和`路径`。模型利用这些元数据决定是否调用某个技能。如果调用，模型将读取`SKILL.md`以获取完整工作流程。

### Shell工具：智能体的“执行环境”

Shell工具让模型能够在真实的终端环境中工作，可选择：

  * 由OpenAI管理的托管容器。
  * 自行执行的本地Shell运行时（工具语义相同，但由您控制机器）。

托管式Shell通过响应API运行，这意味着您的请求具备状态化工作、工具调用、多轮次延续和生成物等特性。

### 压缩机制：保障长流程持续运行

当工作流程变长时，往往会触及上下文窗口的限制。服务端压缩功能通过管理上下文窗口并自动压缩对话历史，确保长流程持续运行。

响应API的压缩机制提供两种处理方式：

  * **服务端自动压缩（新增功能）：** 当上下文超过阈值时，压缩流程会在数据流中自动执行，无需单独调用压缩接口。
  * **独立压缩端点：** 若需精确控制压缩时机，可使用`/responses/compact`接口。

### 协同优势解析

  * **技能模块**将稳定流程和示例封装为可复用单元，避免提示词结构混乱。
  * **Shell环境**提供完整的执行环境，支持代码安装、脚本运行和结果输出。
  * **压缩机制**保障长流程的连续性，使同一工作流程无需人工干预上下文即可持续执行。
  * **三者结合**既能实现可复现的工作流程与真实执行环境，又避免将系统提示词变成脆弱臃肿的巨型文档。

## 实用技巧

### 1）技能描述应遵循路由逻辑（而非营销文案）

技能描述实质上是模型的决策边界，需明确回答：

  * 何时应调用此技能？
  * 何时不应调用此技能？
  * 输出结果与成功标准是什么？

建议在描述中直接包含简明的“适用场景/禁用场景”说明块，并保持具体性（涉及输入参数、工具类型、预期生成物）。

### 2）添加反例与边界案例降低误触发率

一个值得注意的现象是：开放技能调用权限初期反而可能导致正确触发率下降。实践证明，补充反例与边界案例是有效的解决方案。

具体操作时，建议编写若干明确的“禁止调用场景”说明（并标注替代方案）。这能帮助模型更精准地进行路由决策，尤其当存在多个表面相似的技能时效果显著。

Glean直接观察到了这一点：基于技能的路径选择在定向评估中最初使触发率下降了约**20%**，但在他们为描述添加了负面示例和边缘情况覆盖后，触发率得以恢复。

### 3) 将模板和示例置于技能内部（未使用时基本不占用资源）

如果你一直把模板塞进系统提示中，请停止这样做。

将模板和已处理的示例放在技能内部有两个优势：

  * 它们仅在需要时（技能被调用时）可用。
  * 它们不会为不相关的查询增加令牌负担。

这对于知识型工作输出尤其有效，例如：

  * 结构化报告。
  * 升级分诊摘要。
  * 账户计划。
  * 数据分析报告。

Glean报告称，这种模式在生产环境中带来了质量和延迟方面的一些最大提升，因为这些示例仅在技能触发时才会加载。

### 4) 通过容器重用和压缩，早期为长时运行设计

长周期智能体很少能通过一次性提示成功。从一开始就规划连续性：

  * 当需要稳定的依赖项、缓存文件和中间输出时，在多个步骤中重用同一个容器。
  * 传递`previous_response_id`，以便模型可以在同一线程中继续工作。
  * 将压缩作为默认的长时运行原语，而非紧急备用方案。

这种组合减少了重启行为，并随着线程增长保持多步骤作业的连贯性。

### 5) 当需要确定性时，明确指示模型使用技能

默认行为是模型自行决定何时使用技能。这通常是你想要的。

但当你在运行具有明确契约的生产工作流时（并且你宁愿要确定性而非巧妙性），只需说：

“使用`<技能名称>`技能。”

这是你可以使用的最简单的可靠性杠杆。它将模糊的路径选择转变为明确的契约。

### 6) 将技能加网络视为高风险组合（设计时考虑隔离）

这是一个容易被当前忽视、日后难以修复的安全提示。

**将技能与开放网络访问相结合，为数据外泄创造了高风险路径。** 若使用网络功能，请保持网络允许列表严格，假设工具输出不可信，并避免在面向用户的流程中同时使用开放互联网和强大操作——这些场景中用户通常期望严格的确认控制。

**强默认安全策略：**

*   **技能：** **允许**
*   **Shell：** **允许**
*   **网络：** **仅在最小化允许列表下启用**，按需为范围严格限定的任务开放。

### 7) 将 `/mnt/data` 作为工作产物的交接边界

对于托管的 Shell 工作流，请将 `/mnt/data` 视为写入待检索、审查或传递至后续步骤的输出的标准位置。例如报告、清洗后的数据集和最终生成的电子表格。

一个良好的思维模型是：工具写入磁盘，模型基于磁盘内容进行推理，开发者从磁盘检索结果。

### 8) 将允许列表理解为一个双层系统（组织级与请求级）

网络控制在两个层面进行：

*   **组织级允许列表**（由管理员配置），设定允许访问的最大目标集合。
*   **请求级的 `network_policy`**，其必须是组织允许列表的一个子集。

这在操作上有两个重要含义：

1.  保持组织允许列表小而稳定（即“你信任的已批准目标”集合）。
2.  保持请求允许列表更小（即“此单次任务所需的目标”集合）。

如果请求包含组织允许列表之外的域名，将会出错。

### 9) 使用 `domain_secrets` 进行身份验证调用（避免凭据泄露）

如果某个允许的域名需要身份验证头信息，请使用 `domain_secrets`，这样模型就永远不会看到原始凭据。

在运行时，模型看到的是占位符（例如 `$API_KEY`），而一个辅助组件仅会为已批准的目标注入真实值。当你的代理需要在容器内调用受保护的 API 时，这是一个强大的默认安全措施。

### 10) 在云端和本地使用相同的 API

你可以在无需托管所有内容的情况下，同时使用这两类基础功能：

*   技能支持托管Shell与本地Shell两种模式。
*   Shell提供本地执行模式，您可自行执行`shell_call`并将`shell_call_output`返回给模型。
*   若使用Agents SDK，您还可接入自定义的Shell执行器。

实际开发流程通常如下：

  1. 从本地开始（快速迭代、访问内部工具、便于调试）。
  2. 当需要可复现性、隔离性和部署一致性时，迁移至托管容器环境。
  3. 保持两种模式下技能的一致性（即使执行环境迁移，工作流仍保持稳定）。

## 三种构建模式

在自由探索这些新型智能体基础能力的同时，以下提供三种组合运用它们构建实用应用的示例。

### 模式A：安装→获取→生成交付物

这是利用托管Shell的最简方式：智能体安装依赖项、获取外部数据并生成具体交付成果。

例如：

  * 安装若干程序库。
  * 抓取数据或调用API。
  * 将报告写入`/mnt/data/report.md`。

此模式是构建实用型智能体的基础，因为它创建了清晰的审查边界：您的应用可向用户展示交付物、记录日志、进行差异比对，或将其输入后续处理环节。

### 模式B：技能+Shell实现可复现工作流

当成功构建一两个Shell工作流后，您会发现新的挑战：方案虽可行，但提示词漂移会导致可靠性下降。

这正是技能的用武之地。以下推荐一种稳健的实施结构：

  1. 将工作流（步骤、防护机制、模板）编码为技能。
  2. 将该技能挂载至Shell环境。
  3. 引导智能体遵循技能要求，确定性地生成交付物。

此模式特别适用于以下工作流：

  * 电子表格分析或编辑。
  * 数据集清洗及摘要生成。
  * 面向周期性业务流程的标准化报告生成。

### 模式C（进阶）：作为企业级工作流载体的技能

我们早期观察到的一个模式是，在单一工具调用与多工具协同操作之间存在准确性下降的问题。技能可以通过使工具推理更具流程性来弥合这一差距，同时避免系统提示过度膨胀。

来自 Glean 的一个具体示例：

  * 一项面向 Salesforce 的技能将评估准确率从 **73% 提升至 85%**，并将 **首词生成时间** 缩短了 **18.1%**。
  * 实际采用的策略包括精细的路由控制、反例运用，以及在技能内部嵌入模板和示例。
  * Glean 还利用技能来编码企业工作流中的重复性任务，包括客户规划、升级分级处理以及符合品牌调性的内容生成。

这正是能力得以强化的体现。技能成为动态的标准操作程序：随着组织发展而更新，并由智能体一致执行。

## 一次构建，随处运行

当长时运行的智能体既能遵循流程，又能在计算机上执行实际工作时，其效用将大幅提升。技能、托管式命令行和上下文压缩共同构成了这一基础。简要回顾：

  * 使用技能来编码“如何做”（流程、模板、防护规则）。
  * 使用命令行来执行“做什么”（安装、运行、生成产物）。
  * 使用上下文压缩来保持长时运行的一致性（无需手动管理上下文）。
  * 快速迭代时，从本地环境开始。
  * 需要可重复、隔离的执行环境时，迁移至托管容器。
  * 通过组织级和请求级白名单严格控制网络访问，并使用域密钥进行身份验证调用。
