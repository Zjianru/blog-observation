由Tristan Hume撰写，他是Anthropic性能优化团队的负责人。Tristan设计——并重新设计——帮助Anthropic招聘数十名性能工程师的带回家测试。

评估技术候选人随着AI能力的改进变得更加困难。今天能很好地区分人类技能水平的带回家测试可能明天就会被模型 trivially解决——使其对评估无用。

自2024年初以来，我们的性能工程团队使用了一个带回家测试，候选人优化模拟加速器的代码。超过1,000名候选人完成了它，数十人现在在这里工作，包括将我们的Trainium集群启动并运送了自Claude 3 Opus以来的每个模型的工程师。

但每个新的Claude模型都迫使我们重新设计测试。当给予相同的时间限制时，Claude Opus 4超过了大多数人类申请者。那仍然允许我们区分最强的候选人——但后来Claude Opus 4.5甚至匹配了那些。在无限时间内，人类仍然可以超过模型，但在带回家测试的约束下，我们不再有办法区分我们顶级候选人和我们最有能力模型之间的输出。

我已经通过三个版本的带回家测试进行了迭代，试图确保它仍然携带信号。每次，我都学到了一些关于什么使评估对AI辅助具有鲁棒性、什么不具有鲁棒性的新知识。

这篇文章描述了原始带回家设计，每个Claude模型如何击败它，以及我为确保我们的测试保持在顶级模型能力之上而采取的越来越不寻常的方法。虽然我们做的工作随着模型发展，但我们仍然需要更多强大的工程师——只是越来越需要创造性的方式来找到他们。

为此，我们将原始带回家作为开放挑战发布，因为在无限时间内最好的人类表现仍然超过Claude所能达到的。如果您能击败Opus 4.5，我们希望收到您的来信——详情请见本文底部。

带回家的起源

2023年11月，我们准备训练和发布Claude Opus 3。我们已经获得了新的TPU和GPU集群，我们的大型Trainium集群即将到来，我们在加速器上的花费比过去多得多，但我们没有足够的性能工程师来适应我们的新规模。我在Twitter上发帖要求人们给我们发电子邮件，这带来了比通过我们的标准面试管道评估更多的有前途的候选人，这个过程消耗了员工和候选人大量时间

我们需要一种更有效地评估候选人的方法。所以，我花了两周设计一个带回家测试，可以充分捕捉角色需求并识别最有能力的申请者。

设计目标

带回家有坏名声。通常它们充满工程师觉得无聊的通用问题，这使得它们成为糟糕的过滤器。我的目标不同：创造一些真正引人入胜的东西，让候选人兴奋参与，并允许我们在高水平分辨率上捕获他们的技术技能。

该格式在评估性能工程技能方面也提供了优于现场面试的优势：

更长的时间范围：工程师很少在不到一小时的截止日期下编码。4小时窗口（后来减少到2小时）更好地反映了工作的实际性质。它仍然比大多数真实任务短，但我们需要平衡它与它有多繁重。

现实环境：没有人观看或期望叙述。候选人在他们自己的编辑器中工作，没有干扰。

理解和使用工具的时间：性能优化需要理解现有系统，有时还需要构建调试工具。两者在正常的50分钟面试中都很难现实地评估。

与AI辅助的兼容性：Anthropic的一般候选人指南要求候选人在没有指示的情况下完成带回家。对于这个带回家，我们明确指示可以。

时间跨度更长的问题对AI来说更难完全解决，所以候选人可以使用AI工具（就像他们在工作中会做的那样），同时仍然需要展示他们自己的技能。

除了这些特定于格式的目标外，我在设计任何面试时都应用相同的原则来制作带回家：

代表真实工作：问题应该让候选人品尝工作实际涉及的内容。

高信号：带回家应该避免取决于单一见解的问题，并确保候选人有更多机会展示他们的全部能力——尽可能减少偶然性。它还应该有足够广泛的分数分布，并确保即使是最强的候选人也不会完成所有事情。

没有特定的领域知识：具有良好基础的人可以在工作中学习具体内容。不必要地要求狭窄的专业知识会不必要地限制候选人池。

有趣：快速开发循环、有深度的问题，以及创意空间。

模拟机器

我为假的加速器构建了一个Python模拟器，其特征类似于TPU。候选人优化在这个机器上运行的代码，使用热重载的Perfetto跟踪显示每条指令，类似于我们在Trainium上的工具。

该机器包括使加速器优化有趣的特性：手动管理的便笺式内存（与CPU不同， accelerators经常需要显式内存管理）、VLIW（多个执行单元在每个周期并行运行，需要高效的指令打包）、SIMD（每条指令对多个元素进行向量操作）和多核（跨核心分配工作）。

任务是一个并行的树遍历，故意不深入学习风格，因为大多数性能工程师还没有从事深度学习工作，可以在工作中学习领域具体内容。这个问题受到无分支SIMD决策树推理的启发，这是一种经典的ML优化挑战，作为对过去的致敬，只有少数候选人事先遇到过。

候选人从一个完全串行的实现开始，逐步利用机器的并行性。热身是多核并行性，然后候选人选择是处理SIMD向量化还是VLIW指令打包。原始版本还包括一个候选人需要首先调试的bug，行使他们构建工具的能力。

早期结果

初始带回家效果很好。Twitter批次中有一个人的分数明显高于其他人。他在2月初开始工作，比我们通过标准管道聘用的第一批人晚两周。测试被证明是有预测力的：他立即开始优化内核，并发现了一个涉及张量索引数学溢出32位的发布阻塞编译器bug的解决方法。

在接下来的一年半中，约1,000名候选人完成了带回家，它帮助我们聘用了目前大多数性能工程团队。它对于纸上经验有限的候选人特别有价值：几位表现最高的工程师直接来自本科，但展示了足够的带回家技能让我们有信心雇用。

反馈是积极的。许多候选人因为喜欢而超过4小时限制工作。最强的无限时间提交包括完整的优化微型编译器和几个我没有预料到的聪明优化。

然后Claude Opus 4击败了它

到2025年5月，Claude 3.7 Sonnet已经悄悄爬升到超过50%的候选人如果完全委托给Claude Code会更好。然后我在带回家上测试了预发布的Claude Opus 4。它在4小时限制内提出了比几乎所有人更优化的解决方案。

这不是我的第一次面试被Claude模型击败。2023年我设计了一个现场面试问题，具体是因为我们当时的问题基于早期Claude模型有大量知识并可以轻松解决常见任务。我尝试设计一个需要更多问题解决技能而不是知识的问题，仍然基于我在工作中解决的真实（但小众）问题。Claude 3 Opus击败了那个问题的第1部分；Claude 3.5 Sonnet击败了第2部分。我们仍然使用它，因为我们其他的现场问题也不是抗AI的。

对于带回家，有一个直接的修复。问题有比任何人在4小时内可以探索的更多深度，所以我使用Claude Opus 4来确定它在哪里开始挣扎。那成为版本2的新起点。我写了更干净的起始代码，为更深度添加了新的机器特性，并移除了多核（Claude已经解决了，而且只会减慢开发循环而不增加信号）。

我还将时间限制从4小时缩短到2小时。我最初选择4小时是基于候选人反馈喜欢较少的风险，如果他们在bug或困惑上被卡住一会儿会沉没，但调度开销导致我们的管道中数周的延迟。两小时更容易适应周末。

版本2强调聪明的优化见解而不是调试和代码量。它为我们服务了好几个月。

然后Claude Opus 4.5击败了那个

当我测试预发布Claude Opus 4.5检查点时，我看到Claude Code处理这个问题2小时，逐步改进其解决方案。它解决了初始瓶颈，实现了所有常见的微优化，并在不到一小时内达到我们的通过阈值。

然后它停下来，相信它遇到了不可逾越的内存带宽瓶颈。大多数人类达到相同的结论。但是有一些聪明的技巧可以利用问题结构来绕过那个瓶颈。当我告诉Claude可能的周期计数时，它思考了一会儿并找到了那个技巧。然后它调试、调优并实现了进一步的优化。到2小时标记时，它的分数匹配了在该时间限制内最好的人类表现——而那个人大量使用了带 steering的Claude 4。

我们在内部测试时计算推理harness上尝试了它以获得更严格的结果，并确认它可以在2小时内击败人类，并在时间上继续攀升。发布后，我们甚至以通用方式改进了我们的harness并获得了更高的分数。

我遇到了一个问题。我们即将发布一个模型，在我们的带回家上最好的策略将是委托给Claude Code。

考虑选项

一些同事建议禁止AI辅助。我不想这样做。除了执行挑战，我有一种感觉，鉴于人们继续在我们的工作中发挥重要作用，我应该能够想出一些方法让他们在有AI的环境中区分自己——就像他们在工作中会做的那样。我还不想屈服于人类只在超过几小时的任务上才有优势的想法。

其他人建议将标准提高到"显著超过Claude Code单独实现的性能。"这里的担忧是Claude工作得很快。人类通常在开始优化之前花一半的2小时阅读和理解问题。试图 steering  Claude的人类可能总是落后，只能在事后理解Claude做了什么。主导策略可能变成坐视和观察。

如今，Anthropic的性能工程师仍然有很多工作要做，但它看起来更像是艰难的调试、系统设计、性能分析、弄清楚如何验证我们系统的正确性，以及弄清楚如何使Claude的代码更简单更优雅。不幸的是这些事情在没有大量时间或共同背景的情况下难以客观地测试。设计代表工作的面试一直很难，但现在比以往任何时候都更难。

但我也担心如果我投入设计一个新的带回家，要么Claude Opus 4.5也会解决它，要么它会变得对人类在两小时内不可能完成。

尝试1：不同的优化问题

我意识到Claude可以帮助我快速实现我设计的任何东西，这促使我尝试开发一个更难带回家。我选择了一个基于我在Anthropic做过的更棘手的内核优化之一的问题：2D TPU寄存器上的高效数据转置同时避免bank冲突。我将其提炼为模拟机器上的更简单的问题，并让Claude在一天内实现更改。

Claude Opus 4.5发现了一个我甚至没有想到的伟大优化。通过仔细分析，它意识到可以转置整个计算而不是弄清楚如何转置数据，并相应地重写了整个程序。

在我的真实案例中，这不会起作用，所以我修补了问题以删除该方法。然后Claude取得进展但找不到最有效的解决方案。它似乎我有我的新问题，现在我只需要希望人类候选人能足够快地完成它。但我有一些挥之不去的怀疑，所以我使用Claude Code的"ultrathink"功能与更长的思考预算 double-checked ...它解决了。它甚至知道修复bank冲突的技巧。

事后看来，这不是正确的问题尝试。工程师们跨许多平台与数据转置和bank冲突斗争过，所以Claude有大量训练数据可以借鉴。虽然我从第一性原理找到了我的解决方案，但Claude可以借鉴更大的经验工具箱。

尝试2：走得更奇怪

我需要一个人类推理可以战胜Claude更大经验库的问题：足够超出分布的东西。不幸的是，这与像我工作中那样可识别的目标相冲突。

我想到了我最享受的不寻常优化问题，并想到了Zachtronics游戏。这些编程益智游戏使用不寻常的、高度受限的指令集，迫使你以非传统方式编程。例如，在深圳I/O中，程序跨多个通信芯片拆分，每个芯片只保存大约10条指令，带有一两个状态寄存器。聪明的优化通常涉及将状态编码到指令指针或分支标志中。

我设计了一个新的带回家，使用微小、高度受限的指令集来优化最少指令数的解决方案。我实现了一个中等难度的益智游戏并在Claude Opus 4.5上测试。它失败了。我充实了更多益智游戏并让同事验证比我少浸泡在问题中的人仍然可以超过Claude。

与Zachtronics游戏不同，我故意不提供可视化或调试工具。起始代码只检查解决方案是否有效。构建调试工具是被测试的一部分：你可以插入精心制作的打印语句，或在几分钟内让编码模型生成一个交互式调试器。如何投资工具的判断是信号的一部分。

我对新的带回家相当满意。它可能比原始的有更低的方差，因为它由更多独立的子问题组成。早期结果是有希望的：分数与候选人过去工作的素质很好地相关，我最有能力的同事之一得分比任何候选人迄今都高。

我对失去原始的真实感和变化的深度感到悲伤。但真实感可能是我不再拥有的奢侈品。原始工作是因为它像真实工作。替代工作是因为它模拟了新颖的工作。

一个公开挑战

我们发布原始带回家供任何人尝试，时间无限。人类专家在足够长的时间范围内仍比当前模型有优势。提交的最快人类解决方案大大超过了Claude即使有大量测试时计算实现的结果。

发布的版本从头开始（像版本1）但使用版本2的指令集和单核设计，所以周期计数与版本2相当。

性能基准（从模拟机器测量的时钟周期）：

或者您可以通过我们 typical流程申请，这使用我们（现在是）抗Claude的带回家。我们很好奇它能持续多久。
