# 用并行Claude团队构建C编译器

来源：https://www.anthropic.com/engineering/building-c-compiler

---

_作者：Nicholas Carlini，我们安全防护团队的研究员。_

我一直在尝试一种新的语言模型监督方法，我们称之为"智能体团队"。

通过智能体团队，多个Claude实例可以在无需人工干预的情况下并行处理共享代码库。这种方法极大地扩展了大型语言模型智能体所能实现的任务范围。

为了进行压力测试，我让16个智能体从头开始编写一个基于Rust的C编译器，要求能够编译Linux内核。经过近2,000次Claude Code会话和20,000美元的API成本，这个智能体团队最终产出了一个包含10万行代码的编译器，能够在x86、ARM和RISC-V架构上构建Linux 6.9。

[这个编译器本身就是一个有趣的成果](https://github.com/anthropics/claudes-c-compiler)，但在此我将重点分享关于为长期运行的自主智能体团队设计控制框架的经验：如何编写测试以确保智能体在无人监督的情况下保持正轨，如何组织工作结构以使多个智能体能够并行推进，以及这种方法在哪些方面会遇到瓶颈。

## 实现长期运行的Claude

现有的智能体框架（如Claude Code）需要操作员在线并能够协同工作。如果你要求解决一个长期且复杂的问题，模型可能会解决其中一部分，但最终它会停止并等待进一步的输入——可能是一个问题、状态更新或澄清请求。

为了激发持续、自主的进展，我构建了一个控制框架，将Claude置于一个简单的循环中（如果你见过Ralph-loop，应该会感到熟悉）。当它完成一项任务时，会立即开始下一项。_（请在容器中运行此代码，而不是在你的实际机器上）。_

    #!/bin/bash

    while true; do
        COMMIT=$(git rev-parse --short=6 HEAD)
        LOGFILE="agent_logs/agent_${COMMIT}.log"

        claude --dangerously-skip-permissions \
               -p "$(cat AGENT_PROMPT.md)" \
               --model claude-opus-X-Y &> "$LOGFILE"
    done

复制

在智能体提示中，我会告诉Claude需要解决的问题，并要求它通过将问题拆解为小块、追踪当前工作内容、确定下一步任务，并持续优化直至完美的方式来处理问题。（关于最后一点，Claude其实别无选择。这个循环会永远运行——不过有一次我确实看到Claude意外执行了`pkill -9 bash`指令，导致自我终止并结束了循环。哎呀！）。

##
并行运行Claude

并行运行多个实例可以解决单智能体框架的两个弱点：

  * 单个Claude Code会话一次只能处理一项任务。尤其当项目范围扩大时，并行调试多个问题会高效得多。
  * 运行多个Claude智能体可以实现专业化分工。部分智能体负责解决核心问题，其他专业智能体则可以（例如）负责维护文档、监控代码质量或处理特定子任务。

我实现的并行Claude框架非常基础：新建一个空的git仓库，为每个智能体启动一个挂载了该仓库至`/upstream`路径的Docker容器。每个智能体将仓库克隆到本地的`/workspace`目录，完成任务后从各自容器推送更改至上游仓库。

为防止多个智能体同时处理相同任务，该框架采用简单的同步算法：

  1. Claude通过向current_tasks/目录写入文本文件来"锁定"任务（例如：一个智能体可能锁定current_tasks/解析if语句.txt，另一个则锁定current_tasks/生成函数定义.txt）。若两个智能体试图申领相同任务，git的同步机制会强制第二个智能体选择其他任务。
  2. Claude处理任务时，会先拉取上游更新，合并其他智能体的修改，推送自身更改，最后解除锁定。合并冲突时有发生，但Claude足以智能地解决这些问题。
  3. 无限智能体生成循环会在新容器中启动新的Claude Code会话，如此周而复始。

这是一个非常早期的研究原型。我尚未实现任何其他智能体间通信方法，也没有强制执行任何高层目标管理流程。我没有使用协调智能体。

相反，我让每个Claude智能体自行决定如何行动。在大多数情况下，Claude会选择处理"下一个最明显"的问题。当遇到程序错误陷入困境时，Claude通常会维护一份持续更新的文档，记录失败的尝试方法和剩余任务。在该项目的[Git代码库](https://github.com/anthropics/claudes-c-compiler)中，你可以查阅完整的历史记录，观察它如何锁定各项任务进行处理。

## 使用Claude智能体团队编程的经验教训

脚手架让Claude在循环中运行，但只有当Claude能够判断如何取得进展时，这种循环才有意义。我的大部分精力都投入到了围绕Claude的环境设计——测试框架、运行环境、反馈机制——使其能够在没有我干预的情况下自主定位。以下是我在协调多个Claude实例时发现最有效的方法：

### 编写极其高质量的测试

Claude会自主解决我交给它的任何问题。因此任务验证器必须近乎完美，否则Claude可能会解决错误的问题。改进测试框架需要：寻找高质量的编译器测试套件、为开源软件包编写验证器和构建脚本、密切关注Claude犯下的错误，并根据识别出的故障模式设计新的测试用例。

例如，在项目接近尾声时，Claude每次实现新功能时经常破坏现有功能。为了解决这个问题，我构建了持续集成流水线，并实施了更严格的约束机制，使Claude能够更好地测试其工作成果，从而确保新的提交不会破坏现有代码。

### 设身处地为Claude着想

我必须不断提醒自己：这个测试框架是为Claude而非我自己编写的。这意味着需要重新思考关于测试结果传达方式的许多固有假设。

例如，每个智能体都被置于一个全新的容器环境中，没有任何上下文，需要花费大量时间进行自我定位，尤其是在大型项目中。在我们开始测试之前，为了帮助Claude更好地自助，我加入了维护详尽README文档和进度文件的指令，这些文件应频繁更新以反映当前状态。

我还考虑到语言模型固有的局限性，并围绕这些限制进行了设计。这些限制包括：

* **上下文窗口污染：** 测试框架不应输出数千字节的无用信息。最多只应输出几行内容，并将所有重要信息记录到文件中，以便Claude在需要时查找。日志文件应便于自动处理：如果出现错误，Claude应标记ERROR并将原因写在同行，方便grep命令检索。预先计算汇总统计数据有助于避免Claude重复计算。
* **时间感知缺失：** Claude无法感知时间，若放任不管，它会持续数小时运行测试而忽视进度推进。测试框架会低频输出增量进度（避免污染上下文），并内置默认的`--fast`选项，用于运行1%或10%的随机样本。该子样本对每个智能体具有确定性，但在不同虚拟机间随机分布，因此Claude仍能覆盖所有文件，同时每个智能体都能精准识别回归问题。

### 简化并行处理

当存在多个独立失败的测试时，并行化处理变得轻而易举：每个智能体选择不同的失败测试进行处理。当测试套件通过率达到99%后，每个智能体转而致力于编译不同的小型开源项目（如SQLite、Redis、libjpeg、MQuickJS、Lua）。

但当智能体开始编译Linux内核时，它们陷入了困境。与包含数百个独立测试的测试套件不同，编译Linux内核是一项巨型任务。每个智能体都会遇到相同的错误，修复该错误后却会相互覆盖修改。即使同时运行16个智能体也无济于事，因为每个智能体都被困在解决相同的任务上。

修复方案是使用[GCC](https://gcc.gnu.org/)作为在线已知可靠的编译器参照基准进行比较。我编写了一个新的测试框架，随机使用GCC编译内核的大部分文件，仅对剩余文件使用Claude的C编译器进行编译。如果内核能正常运行，则问题不在Claude负责的文件子集中；如果出现故障，就可以通过用GCC重新编译部分文件来进一步定位问题。这使得每个智能体能够并行工作，在不同文件中修复不同的错误，直至Claude的编译器最终能够编译所有文件。（此方法成功后，仍需应用增量调试技术来识别那些单独编译正常但组合编译会失败的成对文件。）

### 多智能体角色分工

并行机制也催生了专业化分工。由于LLM生成的代码经常重复实现已有功能，我安排一个智能体专门负责合并发现的重复代码。另一个智能体负责提升编译器自身的性能，第三个则专注于生成高效的编译后代码。我还指派一个智能体以Rust开发者的视角审视项目设计，对项目结构进行调整以提升整体代码质量，另设一个智能体专门负责文档工作。

## 压力测试智能体团队的极限

本项目被设计为能力基准测试。我致力于压力测试当前LLM技术**勉强能够实现**的极限能力，这有助于我们为未来模型将稳定实现的功能做好准备。

我一直将C编译器项目作为整个Claude 4模型系列的基准测试工具。与先前项目类似，我首先起草了目标需求：一个从零开始、无依赖项的优化编译器，需兼容GCC、能够编译Linux内核，并设计为支持多后端架构。虽然我指定了部分设计要素（例如应采用SSA中间表示以实现多轮优化），但并未具体说明实现方式。

之前的Opus 4模型几乎无法生成可用的编译器。Opus 4.5首次突破了一个门槛，能够生成可通过大型测试套件的功能性编译器，但仍无法编译任何实际的大型项目。我开发Opus 4.6的目标是再次挑战极限。

### 评估过程

在近两周的2000次Claude Code会话中，Opus 4.6消耗了20亿输入词元，生成了1.4亿输出词元，总成本略低于2万美元。即使与最昂贵的Claude Max方案相比，这也是个极其昂贵的项目。但这个总额仅相当于我个人完成这项工作所需成本的一小部分——更不用说整个团队了。

这是一个净室实现（Claude在开发过程中全程无网络访问权限），仅依赖Rust标准库。这个10万行代码的编译器能够在x86、ARM和RISC-V架构上构建可启动的Linux 6.9系统。它还能编译QEMU、FFmpeg、SQLite、PostgreSQL、Redis等软件，在包括[GCC折磨测试套件](https://gcc.gnu.org/onlinedocs/gccint/Torture-Tests.html)在内的大多数编译器测试套件中通过率达到99%。它还通过了开发者的终极试金石测试：能够编译并运行《毁灭战士》。

不过，该编译器仍存在一些局限性：

* 缺少启动Linux实模式所必需的16位x86编译器。为此需要调用GCC（其自带的x86_32和x86_64编译器可正常工作）
* 没有独立的汇编器和链接器——这两部分是Claude最后开始自动化的功能，目前仍存在较多缺陷。演示视频中使用的是GCC的汇编器和链接器
* 虽然能成功构建许多项目，但尚未实现全面兼容，还不能完全替代实际编译器
* 生成代码的效率较低。即使启用所有优化选项，其输出代码的效率仍低于禁用所有优化的GCC
* Rust代码质量尚可，但远未达到专业Rust程序员的水准

由此产生的编译器几乎已经触及了Opus能力的极限。我努力尝试修复上述几个限制，但并未完全成功。新功能和错误修复常常会破坏现有功能。

一个特别具有挑战性的例子是，Opus无法实现启动16位实模式所需的16位x86代码生成器。虽然编译器可以通过66/67操作码前缀输出正确的16位x86代码，但生成的编译输出超过60kb，远远超出了Linux强制执行的32k代码限制。因此，Claude在这里直接"作弊"，调用GCC来完成这一阶段（这仅适用于x86架构。对于ARM或RISC-V，Claude的编译器可以完全独立编译。）

[该编译器的源代码已公开](https://github.com/anthropics/claudes-c-compiler)。您可以下载它，阅读代码，并在您喜欢的C项目上尝试使用。我一直认为，理解语言模型能力的最佳方式是将其推向极限，然后研究它们开始失效的边界。在接下来的几天里，如果您想关注Claude持续尝试解决这些限制的过程，我将继续推动Claude进行新的改进。

## 展望未来

每一代语言模型都开辟了与之协作的新方式。早期模型在IDE中用于代码补全很有帮助。不久之后，模型已经能够根据函数文档字符串完成函数体。Claude Code的推出将智能体带入主流，使开发人员能够与Claude进行结对编程。但这些产品都基于一个假设：用户定义任务，大语言模型运行几秒或几分钟后返回答案，然后用户提供后续反馈。

智能体团队展示了自主实施整个复杂项目的可能性。这使我们作为这些工具的用户，能够设定更加宏大的目标。

我们仍处于早期阶段，完全自主的开发伴随着切实的风险。当开发过程中有人类与Claude协同工作时，他们能够确保质量稳定并实时发现错误。而对于自主系统，人们很容易看到测试通过就认为任务已完成，但实际情况往往并非如此。我曾从事渗透测试工作，专门挖掘大公司产品的安全漏洞，想到程序员可能部署从未亲自验证过的软件，这确实令人担忧。

因此，尽管这个实验让我兴奋，同时也让我感到不安。构建这个编译器是我近期最有趣的经历之一，但我没料到在2026年这么早的阶段就能实现接近这样的成果。语言模型及其交互框架的飞速发展，为我们开启了编写海量新代码的大门。我预计积极应用将超过负面影响，但我们正在进入一个需要全新安全策略来驾驭的新世界。
