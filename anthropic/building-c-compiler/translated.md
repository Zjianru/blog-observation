由Nicholas Carlini撰写，他是我们的安全保障团队的研究员。

我一直在尝试一种我们称为"智能体团队"的新方法来监督语言模型。

使用智能体团队，多个Claude实例在共享代码库上并行工作，无需主动人工干预。这种方法显著扩展了使用LLM智能体可以实现的范围。

为了对其进行压力测试，我给16个智能体分配了从头编写Rust-based C编译器的任务，能够编译Linux内核。近2,000个Claude Code会话和20,000美元的API成本，智能体团队产生了100,000行代码的编译器，可以在x86、ARM和RISC-V上构建Linux 6.9。

编译器本身是一个有趣的工件，但我在此专注于为长时运行自主智能体团队设计harness的经验教训：如何编写测试来保持智能体正常轨道而无需人工监督，如何结构化工作以便多个智能体可以并行取得进展，以及这种方法在哪里达到极限。

实现长时运行Claude

现有的智能体scaffold如Claude Code需要一个操作员在线并可用来共同工作。如果你要求一个长期复杂问题的解决方案，模型可能会解决部分问题，但最终会停止并等待持续输入——一个问题、状态更新或澄清请求。

为了引出持续的、自主取得的进展，我构建了一个将Claude放入简单循环的harness（如果你见过Ralph-loop，这看起来很熟悉）。当它完成一个任务时，它立即选择下一个。（在容器中运行，而不是在你的实际机器上。）

#!/bin/bash

while true; do
    COMMIT=$(git rev-parse --short=6 HEAD)
    LOGFILE="agent_logs/agent_${COMMIT}.log"

    claude --dangerously-skip-permissions \
           -p "$(cat AGENT_PROMPT.md)" \
           --model claude-opus-X-Y &> "$LOGFILE"
done

使用Claude智能体团队编程的经验教训

scaffold让Claude在循环中运行，但只有当Claude能告诉如何取得进展时那个循环才有用。我大部分精力都花在了围绕Claude设计环境——测试、环境、反馈——以便它可以在没有我的情况下定位自己。这些是我在编排多个Claude实例时发现最有帮助的方法。

编写极高质量的测试

Claude将自主工作来解决我给它的任何问题。所以任务验证器 nearly perfect是很重要的，否则Claude会解决错误的问题。改进测试harness需要寻找高质量的编译器测试套件，为开源软件包编写验证器和构建脚本，并观察Claude犯的错误，然后设计新的测试来识别那些失败模式。

例如，在项目接近尾声时，Claude开始经常在每次实现新功能时破坏现有功能。为了解决这个问题，我构建了一个持续集成管道并实现了更严格的 enforcement，允许Claude更好地测试其工作，以便新提交不能破坏现有代码。

把自己放在Claude的角度

我不得不不断提醒自己，我为Claude编写这个测试harness而不是为我自己，这意味着要重新思考我对测试如何传达结果的许多假设。

例如，每个智能体被放入一个 fresh容器，没有上下文，会花费大量时间定位自己，尤其是在大项目上。在我们甚至到达测试之前，为了帮助Claude帮助自己，我包含了维护 extensive READMEs和进度文件的指令，这些文件应该用当前状态 frequent更新。

使并行化容易

当有许多 distinct failing tests时，并行化很简单：每个智能体 choose a different failing test来处理。测试套件达到99% pass rate后，每个智能体致力于让不同的小型开源项目（例如SQLite、Redis、libjpeg、MQuickJS、Lua）编译。

但当智能体开始编译Linux内核时，它们 stuck了。与具有数百个独立测试的测试套件不同，编译Linux内核是一项巨大的任务。每个智能体会 hit the same bug，修复那个bug，然后 overwrite each other's changes。运行16个智能体没有帮助，因为每个都 stuck解决相同的任务。

修复方法是使用GCC作为在线已知良好编译器 oracle进行比较。我编写了一个新的测试harness，随机用GCC编译大部分内核，只用Claude的C编译器编译剩余文件。如果内核工作，那么问题不在Claude的文件子集中。如果它坏了，那么可以通过用GCC重新编译其中一些文件进一步细化。这样每个智能体可以并行工作，修复不同文件中的不同bug，直到Claude的编译器最终可以编译所有文件。（这 work之后，仍然需要应用delta debugging技术来找到一起失败但独立工作的文件对。）

多个智能体角色

并行性还支持专门化。LLM编写的代码经常 re-implements现有功能，所以我 assign了一个智能体来合并它发现的任何重复代码。我让另一个负责改进编译器本身的性能，第三个负责输出高效的编译代码。我让另一个从Rust开发者的角度批评项目设计，并进行结构性更改以改善整体代码质量，还让另一个处理文档。

对智能体团队极限的压力测试

这个项目被设计为一个能力基准。我有兴趣对LLM今天勉强能达到的极限进行压力测试，以帮助我们为模型未来可靠达到的目标做准备。

我一直在使用C编译器项目作为整个Claude 4模型系列的基准。像之前的工作一样，我首先起草我想要的东西：一个从头开始的优化编译器，无依赖，GCC兼容，能够编译Linux内核，并设计为支持多个后端。虽然我指定了设计的某些方面（例如它应该有一个SSA IR来启用多个优化 passes），但我没有详细介绍如何做到这一点。

之前的Opus 4模型只能勉强产生一个功能编译器。Opus 4.5是第一个跨越能够产生可以通过大测试套件的功能编译器的阈值，但它仍然无法编译任何真正的大型项目。我的Opus 4.6目标是再次测试极限。

评估

在近2,000个Claude Code会话横跨两周，Opus 4.6消耗了20亿输入令牌并生成了1.4亿输出令牌，总成本刚刚超过20,000美元。与即使最昂贵的Claude Max计划相比，这是一个极其昂贵的项目。但总数是我自己生产这个的一小部分——更不用说整个团队了。

这是一个干净室实现（Claude在其开发过程中从未访问过互联网）；它只依赖Rust标准库。这个100,000行编译器可以在x86、ARM和RISC-V上构建可启动的Linux 6.9。它还可以编译QEMU、FFmpeg、SQLite、postgres、redis，并且在大多数编译器测试套件（包括GCC torture test suite）上达到99% pass rate。它还通过了开发者的 ultimate litmus test：可以编译和运行Doom。

由此产生的编译器几乎达到了Opus能力的极限。我（hard!）尝试修复上述一些限制，但没有完全成功。新功能和bug修复经常破坏现有功能。

作为一个特别具有挑战性的例子，Opus无法实现引导进入16位实模式所需的16位x86代码生成器。虽然编译器可以通过66/67操作码前缀输出正确的16位x86，但 resulting compiled output超过60kb，远远超过Linux enforcement的32k代码限制。相反，Claude在这里简单地cheats并为这个阶段调用GCC（这只适用于x86。对于ARM或RISC-V，Claude的编译器可以完全独立编译。）

编译器的源代码可用。下载它，阅读代码，并在你最喜欢的C项目上尝试。我一直发现理解语言模型能做什么的最好方法是推动它们达到极限，然后研究它们在哪里开始崩溃。在接下来的几天里，如果你们想跟随，Claude将继续推动新的更改尝试解决这些限制。

展望未来

每一代语言模型都开辟了与它们合作的新方式。早期模型在IDE中用于tab补全有用。没多久，模型就可以从一个函数的docstring完成函数体。Claude Code的发布将智能体带入主流，并使开发者能够与Claude进行结对编程。但这些产品都在假设用户定义任务、LLM运行几秒钟或几分钟并返回答案、然后用户提供 follow-up下运作。

智能体团队展示了实现整个复杂项目 autonomously的可能性。这使我们作为这些工具的用户，能够将我们的目标变得更具雄心。

我们仍然处于早期，完全自主开发带有真正的风险。当人类在开发期间与Claude坐在一起时，他们可以确保一致的质量并实时捕获错误。对于自主系统，很容易看到测试通过并假设工作完成了，而这种情况很少发生。我曾经做过渗透测试，利用大公司生产产品中的漏洞，程序员部署他们从未亲自验证过的软件的想法是一个真正的担忧。

所以，虽然这个实验让我兴奋，但它也让我感到不安。构建这个编译器是我最近最有乐趣的事情之一，但我没想到这会在2026年这么早成为可能。语言模型和我们用来与它们交互的scaffold的快速进步打开了编写大量新代码的大门。我期望积极的应用程序超过 negative的，但我们正在进入一个将需要新策略来安全导航的新世界。
