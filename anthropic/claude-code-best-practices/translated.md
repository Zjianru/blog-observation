Claude Code是一个智能体编码环境。与需要等待问答的聊天机器人不同，Claude Code能够读取您的文件、执行命令、进行修改，并在您监督、调整或完全离开的情况下自主解决问题。这彻底改变了您的工作方式——您无需亲自编写代码再请Claude审查，只需描述需求，Claude便会自主构建解决方案。Claude会进行探索、规划与实施。但这种自主性仍存在需要理解的学习曲线，Claude在特定约束条件下运行。本指南汇集了Anthropic内部团队及各类工程师在不同代码库、编程语言和环境中使用Claude Code验证有效的实践模式。关于智能体循环的底层工作原理，请参阅[Claude Code运行机制](/docs/en/how-claude-code-works)。

大多数最佳实践都基于一个约束条件：Claude的上下文窗口会迅速填满，且性能会随着填充程度而下降。Claude的上下文窗口存储着整个对话内容，包括每条消息、Claude读取的每个文件以及每个命令输出。然而，这个窗口可能很快被填满。一次调试会话或代码库探索就可能生成并消耗数万个令牌。这一点至关重要，因为随着上下文填充，LLM的性能会下降。当上下文窗口接近满载时，Claude可能会开始"遗忘"先前的指令或出现更多错误。上下文窗口是需要管理的最重要资源。要了解会话在实践中如何被填充，请[观看交互式演示](/docs/en/context-window)，了解启动时加载的内容以及读取每个文件的成本。通过[自定义状态栏](/docs/en/statusline)持续跟踪上下文使用情况，并参考[减少令牌使用](/docs/en/costs#reduce-token-usage)了解降低令牌使用量的策略。

##

​

为Claude提供验证其工作的途径

包含测试、截图或预期输出，以便Claude能够自我检查。这是您可以采取的最高效措施。

当Claude能够验证自身工作时（例如运行测试、比对截图、验证输出），其表现会显著提升。若缺乏明确成功标准，它可能生成看似正确但实际无法工作的内容。您将成为唯一的反馈循环，每个错误都需要您亲自处理。

策略| 改进前| 改进后
---|---|---
**提供验证标准**| _“实现一个验证电子邮箱地址的函数”_| _“编写validateEmail函数。示例测试用例：[user@example.com](mailto:user@example.com)返回true，invalid返回false，[user@.com](mailto:user@.com)返回false。实现后运行测试”_
**可视化验证UI变更**| _“让仪表板看起来更美观”_| _“[粘贴截图] 请实现此设计。完成后截图并与原图对比，列出差异项并修正”_
**解决根本原因而非表象**| _“构建失败了”_| _“构建失败并报错：[粘贴错误信息]。请修复并验证构建成功。需解决根本原因，而非掩盖错误”_

UI变更可通过[Chrome版Claude扩展](/docs/en/chrome)进行验证。该扩展会在浏览器中新建标签页测试UI，并持续迭代直至代码生效。验证方式也可采用测试套件、代码检查工具或能检测输出的Bash命令。请着力构建坚如磐石的验证机制。

##

​

先探索，再规划，后编码

将调研规划与实施阶段分离，避免解决错误的问题。

若直接让Claude开始编码，可能产生解决错误问题的代码。请使用[规划模式](/docs/en/common-workflows#use-plan-mode-for-safe-code-analysis)将探索与执行阶段分离。推荐工作流包含四个阶段：

1

探索

进入规划模式。Claude将读取文件并回答问题，期间不进行任何修改。

claude（规划模式）

    请阅读/src/auth目录，理解会话管理与登录机制。
    同时查看我们如何处理包含密钥的环境变量。

2

规划

要求Claude制定详细的实施方案。

claude（规划模式）

    需要添加Google OAuth功能。哪些文件需要修改？
    会话流程是怎样的？请制定实施方案。

按`Ctrl+G`将方案在文本编辑器中打开，便于在Claude执行前直接编辑。

3

实施

切换回普通模式，让Claude根据方案进行编码并同步验证。

claude（普通模式）

实施你计划中的OAuth流程。为回调处理器编写测试，运行测试套件并修复所有失败。

4

提交

请Claude用描述性信息提交并创建PR。

claude（普通模式）

    用描述性信息提交并开启PR

计划模式很有用，但也会增加开销。对于范围明确且改动较小的任务（比如修复拼写错误、添加日志行或重命名变量），可以直接让Claude执行。当你对方法不确定、需要修改多个文件或不熟悉要修改的代码时，计划模式最有用。如果你能用一句话描述改动内容，就跳过计划步骤。

##

​

在提示中提供具体上下文

你的指示越精确，需要的修正就越少。

Claude能够推断意图，但无法读心。请引用具体文件、提及限制条件，并指向示例模式。

策略| 之前| 之后
---|---|---
**明确任务范围。** 指定哪个文件、什么场景以及测试偏好。| _“为foo.py添加测试”_| _“为foo.py编写测试，覆盖用户已登出的边界情况。避免使用模拟对象。”_
**指向信息来源。** 引导Claude查找能解答问题的来源。| _“为什么ExecutionFactory的API这么奇怪？”_| _“查阅ExecutionFactory的git历史记录，总结其API是如何形成的”_
**参考现有模式。** 指向代码库中的现有模式。| _“添加日历组件”_| _“查看首页现有组件的实现方式以理解模式。HotDogWidget.php是个好例子。遵循该模式实现一个新的日历组件，让用户选择月份并能前后翻页选择年份。不使用代码库已有库以外的库从头开始构建。”_
**描述问题现象。** 提供现象、可能的位置以及“修复后”的状态。| _“修复登录错误”_| _“用户报告会话超时后登录失败。检查src/auth/中的认证流程，特别是令牌刷新。编写一个能复现该问题的失败测试，然后修复它”_

模糊的提示在探索阶段可能很有用，此时你还有调整的余地。像 `"这个文件里有什么可以改进的地方？"` 这样的提示可能会揭示出你未曾想到要询问的问题。

###

​

提供丰富的内容

使用 `@` 引用文件、直接粘贴截图/图片，或通过管道传输数据。

你可以通过多种方式为 Claude 提供丰富的数据：

  * **使用 `@` 引用文件**，而不是描述代码的位置。Claude 会在回复前读取文件内容。
  * **直接粘贴图片**。将图片复制/粘贴或拖放到提示框中。
  * **提供文档和 API 参考的 URL**。使用 `/permissions` 将常用域名加入允许列表。
  * **通过管道传输数据**，例如运行 `cat error.log | claude` 来直接发送文件内容。
  * **让 Claude 自行获取所需内容**。指示 Claude 使用 Bash 命令、MCP 工具或读取文件来拉取上下文。

配置你的环境

进行一些设置步骤，可以让 Claude Code 在所有会话中显著提高效率。有关扩展功能的完整概述以及何时使用每个功能，请参阅[扩展 Claude Code](/docs/en/features-overview)。

###

​

编写有效的 CLAUDE.md

运行 `/init` 命令，根据你当前的项目结构生成一个初始的 CLAUDE.md 文件，然后随时间逐步完善。

CLAUDE.md 是一个特殊的文件，Claude 在每次对话开始时都会读取它。在其中包含 Bash 命令、代码风格和工作流程规则。这为 Claude 提供了它无法仅从代码中推断出的持久化上下文。`/init` 命令会分析你的代码库，以检测构建系统、测试框架和代码模式，为你提供一个可以进一步完善的坚实基础。CLAUDE.md 文件没有固定的格式要求，但请保持其简短且易于人类阅读。例如：

CLAUDE.md

    # 代码风格
    - 使用 ES 模块（import/export）语法，而非 CommonJS（require）
    - 尽可能解构导入（例如：import { foo } from 'bar'）

    # 工作流程
    - 完成一系列代码更改后，务必进行类型检查
    - 出于性能考虑，优先运行单个测试，而非整个测试套件

CLAUDE.md 文件会在每次会话中加载，因此只应包含广泛适用的内容。对于仅偶尔相关的领域知识或工作流程，请改用[技能](/docs/en/skills)。Claude 会按需加载这些技能，避免每次对话都变得臃肿。保持内容简洁。针对每一行，请自问：“删除这一条会导致 Claude 犯错吗？”如果不会，就删掉它。臃肿的 CLAUDE.md 文件会导致 Claude 忽略你真正的指令！

✅ 应包含 | ❌ 应排除
---|---
Claude 无法猜测的 Bash 命令 | Claude 通过阅读代码就能推断出的内容
与默认值不同的代码风格规则 | Claude 已经知道的标准语言惯例
测试说明和偏好的测试运行器 | 详细的 API 文档（改为链接到文档）
仓库规范（分支命名、PR 约定） | 频繁变化的信息
特定于项目的架构决策 | 冗长的解释或教程
开发环境特殊要求（必需的环境变量） | 对代码库逐文件的描述
常见陷阱或非显而易见的行为 | 不言自明的实践，如“编写整洁的代码”

如果 Claude 持续做出你不希望的行为，尽管已有相关规则禁止，那很可能是因为文件太长，规则被淹没了。如果 Claude 询问的问题在 CLAUDE.md 中已有答案，可能是表述存在歧义。将 CLAUDE.md 视为代码：在出错时审查它，定期修剪，并通过观察 Claude 的行为是否实际改变来测试修改。你可以通过添加强调（例如“重要”或“你必须”）来调整指令，以提高遵循度。将 CLAUDE.md 提交到 git 中，以便你的团队可以共同维护。该文件的价值会随时间累积。CLAUDE.md 文件可以使用 `@路径/到/导入文件` 语法导入其他文件：

CLAUDE.md

    项目概述请参阅 @README.md，可用 npm 命令请参阅 @package.json。

    # 附加说明
    - Git 工作流程：@docs/git-instructions.md
    - 个人覆盖设置：@~/.claude/my-project-instructions.md

你可以将 CLAUDE.md 文件放置在以下几个位置：

*   **主文件夹 (`~/.claude/CLAUDE.md`)**：适用于所有 Claude 会话
*   **项目根目录 (`./CLAUDE.md`)**：可提交至 git 以便与团队共享
*   **项目根目录 (`./CLAUDE.local.md`)**：个人项目特定笔记；请将此文件添加到 `.gitignore` 中，以免与团队共享
*   **父目录**：适用于 monorepo，可自动引入 `root/CLAUDE.md` 和 `root/foo/CLAUDE.md`
*   **子目录**：当处理这些目录中的文件时，Claude 会根据需要引入子目录的 CLAUDE.md 文件

配置权限

使用[自动模式](/docs/en/permission-modes#eliminate-prompts-with-auto-mode)让分类器处理审批，使用 `/permissions` 来允许特定命令，或使用 `/sandbox` 进行操作系统级隔离。每种方式都能在保持您控制权的同时减少干扰。

默认情况下，Claude Code 会请求可能修改您系统的操作权限：文件写入、Bash 命令、MCP 工具等。这很安全但繁琐。在第十次批准之后，您就不再是真正审查，而只是点击通过。有三种方法可以减少这些干扰：

*   **自动模式**：由独立的分类器模型审查命令，仅阻止看起来有风险的操作：权限提升、未知基础设施或恶意内容驱动的操作。当您信任任务的大方向但不想点击通过每个步骤时最佳
*   **权限允许列表**：允许您知道安全的特定工具，如 `npm run lint` 或 `git commit`
*   **沙盒隔离**：启用操作系统级隔离，限制文件系统和网络访问，允许 Claude 在定义的边界内更自由地工作

了解更多关于[权限模式](/docs/en/permission-modes)、[权限规则](/docs/en/permissions)和[沙盒隔离](/docs/en/sandboxing)的信息。

###

​

使用 CLI 工具

在与外部服务交互时，告诉 Claude Code 使用 CLI 工具，如 `gh`、`aws`、`gcloud` 和 `sentry-cli`。

CLI 工具是与外部服务交互时最高效的方式。如果您使用 GitHub，请安装 `gh` CLI。Claude 知道如何使用它来创建议题、开启拉取请求和读取评论。即使没有 `gh`，Claude 仍可使用 GitHub API，但未经身份验证的请求常会触发速率限制。Claude 也能高效学习它原本不熟悉的 CLI 工具。尝试使用类似这样的提示：`使用 'foo-cli-tool --help' 了解 foo 工具，然后用它解决 A、B、C 问题。`

​

连接 MCP 服务器

运行 `claude mcp add` 以连接外部工具，如 Notion、Figma 或您的数据库。

通过 [MCP 服务器](/docs/en/mcp)，您可以要求 Claude 实现来自议题跟踪器的功能、查询数据库、分析监控数据、集成 Figma 的设计并自动化工作流程。

###

​

设置钩子

对于每次都必须执行且绝无例外的情况，请使用钩子。

[钩子](/docs/en/hooks-guide) 会在 Claude 工作流程的特定节点自动运行脚本。与建议性的 CLAUDE.md 指令不同，钩子是确定性的，能确保动作必然发生。Claude 可以为您编写钩子。尝试使用类似这样的提示：“编写一个在每次文件编辑后运行 eslint 的钩子” 或 “编写一个阻止写入 migrations 文件夹的钩子。” 您可以直接编辑 `.claude/settings.json` 来手动配置钩子，并运行 `/hooks` 来浏览已配置的内容。

###

​

创建技能

在 `.claude/skills/` 目录下创建 `SKILL.md` 文件，以赋予 Claude 领域知识和可复用的工作流程。

[技能](/docs/en/skills) 通过您项目、团队或领域特有的信息来扩展 Claude 的知识。Claude 会在相关时自动应用它们，您也可以直接使用 `/技能名称` 来调用。通过在 `.claude/skills/` 下添加一个包含 `SKILL.md` 的目录来创建技能：

.claude/skills/api-conventions/SKILL.md

    ---
    name: api-conventions
    description: 我们服务的 REST API 设计规范
    ---
    # API 规范
    - URL 路径使用短横线命名法
    - JSON 属性使用驼峰命名法
    - 列表端点必须包含分页
    - 在 URL 路径中进行 API 版本控制（/v1/、/v2/）

技能也可以定义可直接调用的可重复工作流：

.claude/skills/fix-issue/SKILL.md

    ---
    name: fix-issue
    description: 修复GitHub问题
    disable-model-invocation: true
    ---
    分析并修复GitHub问题：$ARGUMENTS。

    1. 使用 `gh issue view` 获取问题详情
    2. 理解问题描述中的核心问题
    3. 在代码库中搜索相关文件
    4. 实施修复问题所需的必要更改
    5. 编写并运行测试以验证修复
    6. 确保代码通过代码规范检查和类型检查
    7. 创建描述性提交信息
    8. 推送更改并创建拉取请求

运行 `/fix-issue 1234` 来调用此技能。对于具有副作用且希望手动触发的工作流，请使用 `disable-model-invocation: true` 设置。

###

​

创建自定义子代理

在 `.claude/agents/` 目录中定义专门的助手，Claude可以委托其处理独立任务。

[子代理](/docs/en/sub-agents)在独立的上下文中运行，拥有自己的一套允许使用的工具。它们适用于需要读取多个文件或需要专注处理特定任务而不干扰主对话的场景。

.claude/agents/security-reviewer.md

    ---
    name: security-reviewer
    description: 审查代码中的安全漏洞
    tools: Read, Grep, Glob, Bash
    model: opus
    ---
    你是一名高级安全工程师。审查代码中的以下问题：
    - 注入漏洞（SQL注入、XSS、命令注入）
    - 身份验证和授权缺陷
    - 代码中的密钥或凭据泄露
    - 不安全的数据处理方式

    提供具体的行号引用和修复建议。

明确指示Claude使用子代理：_“使用子代理审查这段代码的安全问题。”_

安装插件

##

​

有效沟通

您与Claude Code的沟通方式会显著影响结果质量。

###

​

询问代码库问题

向Claude提出您会向资深工程师咨询的问题。

当接手新代码库时，使用Claude Code进行学习和探索。您可以向Claude提出与询问其他工程师相同类型的问题：

  * 日志系统如何工作？
  * 如何创建新的API端点？
  * `foo.rs`第134行的`async move { ... }`有什么作用？
  * `CustomerOnboardingFlowImpl`处理哪些边界情况？
  * 为什么这段代码在第333行调用`foo()`而不是`bar()`？

以这种方式使用Claude Code是一种高效的入职工作流程，能缩短熟悉时间并减轻其他工程师的负担。无需特殊提示技巧：直接提问即可。

###

​

让Claude采访您

对于较大型的功能开发，先让Claude采访您。从最简提示开始，要求Claude使用`AskUserQuestion`工具进行采访。

Claude会询问您可能尚未考虑的问题，包括技术实现、UI/UX、边界情况和权衡取舍。

    我想构建[简要描述]。请使用AskUserQuestion工具详细采访我。
    
    询问技术实现、UI/UX、边界情况、关注点和权衡取舍。不要问显而易见的问题，深入挖掘我可能未考虑的难点。
    
    持续采访直到涵盖所有内容，然后将完整规范写入SPEC.md。

规范完成后，开启新会话进行实施。新会话将获得专注于实现的纯净上下文，同时您还有书面规范可供参考。

##

​

管理会话

对话具有持久性和可回溯性。请善用这一特性！

###

​

尽早并频繁调整方向

一旦发现Claude偏离轨道，请立即纠正。

最佳结果源于紧密的反馈循环。虽然Claude偶尔能在首次尝试时就完美解决问题，但快速纠正通常能更快地产生更好的解决方案。

* **`Esc`键**：使用`Esc`键可随时中断Claude的操作。上下文会被保留，方便您重新引导对话方向。
* **`Esc + Esc`或`/rewind`**：按两次`Esc`键或执行`/rewind`命令可打开回溯菜单，恢复之前的对话和代码状态，或从指定消息开始重新生成内容。
* **`"撤销操作"`**：让Claude撤回其做出的更改。
* **`/clear`命令**：在不同任务间重置上下文。包含无关内容的长会话会降低模型表现。

若在同一会话中针对同一问题纠正Claude超过两次，说明上下文中已堆积过多失败尝试。此时请执行`/clear`命令，并运用已掌握的技巧重新开始更精准的提问。相比累积多次修正的长会话，拥有清晰上下文和优化提示的新会话几乎总能获得更佳效果。

###

​

主动管理上下文

在不同任务间使用`/clear`命令重置上下文。

当接近上下文容量限制时，Claude Code会自动压缩对话历史，在保留关键代码和决策的同时释放空间。在长时间会话中，Claude的上下文窗口可能被无关对话、文件内容和操作指令填满，这将影响其表现并可能导致注意力分散。

* 频繁使用 `/clear` 在任务间彻底重置上下文窗口
* 当自动压缩触发时，Claude 会总结最关键的内容，包括代码模式、文件状态和关键决策
* 如需更多控制，可运行 `/compact <指令>`，例如 `/compact 聚焦于API变更`
* 若仅需压缩部分对话，使用 `Esc + Esc` 或 `/rewind`，选择消息检查点后点击 **从此处总结**。这将压缩该点之后的对话内容，同时保留先前的完整上下文
* 在 CLAUDE.md 中通过指令自定义压缩行为，例如 `"压缩时始终保留完整的修改文件列表及所有测试命令"`，以确保关键信息在总结过程中得以保留
* 对于无需保留在上下文中的快速提问，请使用 [`/btw`](/docs/en/interactive-mode#side-questions-with-btw)。答案将显示在可关闭的浮层中且不会进入对话历史，方便您在不增加上下文负担的情况下查询细节

###

​

使用子代理进行调查

通过 `"使用子代理调查 X"` 委派研究任务。子代理将在独立上下文中探索，保持您的主对话界面整洁以专注于实施。

由于上下文是根本性限制，子代理是最强大的工具之一。当 Claude 研究代码库时会读取大量文件，这些都会消耗您的上下文。子代理在独立的上下文窗口中运行并返回总结报告：

    使用子代理调查我们的认证系统如何处理令牌刷新，
    以及是否存在可复用的现有 OAuth 实用工具。

子代理将探索代码库、阅读相关文件并反馈发现结果，整个过程不会干扰主对话。您也可以在 Claude 实施后使用子代理进行验证：

    使用子代理审查此代码的边缘情况

###

​

通过检查点回退

Claude 的每个操作都会创建检查点。您可以将对话、代码或两者恢复到任意先前的检查点状态。

Claude 会在执行变更前自动创建检查点。双击 `Escape` 键或运行 `/rewind` 命令即可打开回溯菜单。您可以选择仅恢复对话、仅恢复代码、同时恢复两者，或从指定消息开始生成总结。详见[检查点功能](/docs/en/checkpointing)。您无需精心规划每个步骤，可以指示Claude尝试高风险操作。若未达预期，只需回溯并尝试其他方案。检查点支持跨会话保存，即使关闭终端后仍可进行回溯。

检查点仅追踪 _Claude执行_ 的变更，不记录外部进程的改动。此功能并非git的替代方案。

###

​

恢复对话

运行 `claude --continue` 可接续最近会话，或使用 `--resume` 从近期会话列表中选择。

Claude Code 会将对话内容本地保存。当任务需要跨多轮会话时，您无需重复说明上下文：

    claude --continue    # 恢复最近进行的对话
    claude --resume      # 从近期对话列表中选择

使用 `/rename` 命令可为会话设置描述性名称（如 `"oauth迁移"` 或 `"内存泄漏调试"`），便于后续查找。将会话视为分支：不同工作流可拥有独立且持久的上下文环境。

##

​

自动化与规模化扩展

当您能熟练运用单个Claude后，可通过并行会话、非交互模式和扇出模式实现产能倍增。前文均基于单人单会话场景，但Claude Code支持横向扩展。本节将展示如何通过进阶技巧提升工作效率。

###

​

启用非交互模式

在CI流程、预提交钩子或脚本中使用 `claude -p "提示词"`。添加 `--output-format stream-json` 参数可获取流式JSON输出。

通过 `claude -p "您的提示词"` 可在无会话状态下非交互式运行Claude。此模式适用于将Claude集成至CI流水线、预提交钩子等自动化工作流。输出格式支持程序化解析：纯文本、JSON或流式JSON。

    # 单次查询
    claude -p "解释本项目功能"

    # 为脚本提供结构化输出
    claude -p "列出所有API端点" --output-format json

# 实时处理的流式传输
claude -p "分析此日志文件" --output-format stream-json

###

​

运行多个Claude会话

并行运行多个Claude会话可加速开发进程、进行隔离实验或启动复杂工作流。

主要有三种并行会话运行方式：

*   [Claude Code桌面应用](/docs/en/desktop#work-in-parallel-with-sessions)：可视化管理多个本地会话。每个会话都拥有独立的隔离工作树。
*   [网页版Claude Code](/docs/en/claude-code-on-the-web)：在Anthropic安全的云端基础设施中，于隔离的虚拟机内运行。
*   [智能体团队](/docs/en/agent-teams)：通过共享任务、消息传递和团队领导，实现多个会话的自动化协调。

除了并行化工作，多个会话还能支持以质量为核心的工作流。全新的上下文环境能提升代码审查质量，因为Claude不会对其刚编写的代码产生偏向。例如，可采用编写者/审查者模式：

| 会话A（编写者） | 会话B（审查者） |
|---|---|
| `为我们的API端点实现一个速率限制器` | |
| | `审查@src/middleware/rateLimiter.ts中的速率限制器实现。查找边界情况、竞态条件以及与现有中间件模式的一致性。` |
| `这是审查反馈：[会话B的输出]。请处理这些问题。` | |

您也可以对测试采用类似方法：让一个Claude编写测试，然后让另一个编写通过测试的代码。

###

​

跨文件分发任务

循环遍历任务，对每个任务调用`claude -p`。使用`--allowedTools`来限定批量操作的权限范围。

对于大规模迁移或分析任务，您可以将工作分发给多个并行的Claude调用：

1

生成任务列表

让Claude列出所有需要迁移的文件（例如：`列出所有2,000个需要迁移的Python文件`）

2

编写循环遍历列表的脚本

    for file in $(cat files.txt); do
      claude -p "将$file从React迁移到Vue。返回OK或FAIL。" \
        --allowedTools "Edit,Bash(git commit *)"
    done

3

在少量文件上测试，然后大规模运行

根据前2-3个文件出现的问题优化提示词，再对完整数据集执行。`--allowedTools` 参数可限制Claude的操作范围，这在无人值守运行时尤为重要。

您还可以将Claude集成到现有数据处理流程中：

    claude -p "<您的提示词>" --output-format json | 您的命令

开发阶段使用 `--verbose` 参数进行调试，生产环境请关闭该选项。

###

​

使用自动模式实现自主运行

如需在后台安全检查保障下实现不间断执行，请启用[自动模式](/docs/zh/permission-modes#eliminate-prompts-with-auto-mode)。分类器模型会在命令执行前进行审核，阻止权限升级、未知基础设施操作及恶意内容驱动的行为，同时允许常规工作无需确认直接执行。

    claude --permission-mode auto -p "修复所有代码规范错误"

使用 `-p` 参数进行非交互式运行时，若分类器多次拦截操作，自动模式将中止执行（因无用户可回退确认）。具体阈值参见[自动模式回退机制](/docs/zh/permission-modes#when-auto-mode-falls-back)。

##

​

规避常见失败模式

以下常见错误若能及早识别可节省大量时间：

  * **大杂烩会话**：从某项任务开始，中途转向无关提问，又返回原任务。上下文充斥无关信息。

> **解决方案**：无关任务间使用 `/clear` 清空上下文。

  * **重复纠错循环**：Claude操作出错→您纠正→仍错误→再次纠正。上下文被失败尝试污染。

> **解决方案**：两次纠正失败后立即 `/clear`，并整合经验撰写更精准的初始提示。

  * **过度详细的CLAUDE.md**：配置文档过长会导致Claude忽略半数内容，关键规则淹没在冗余信息中。

> **解决方案**：严格精简。若某规则未配置时Claude已能正确处理，请删除该规则或转为钩子函数。

  * **信任验证断层**：Claude生成看似合理的实现方案，却未处理边界情况。

**修正**：始终提供验证（测试、脚本、截图）。若无法验证，则不要交付。

* **无限探索**：你要求Claude“调查”某事却未限定范围。Claude会阅读数百个文件，占满上下文空间。

**修正**：限定调查范围或使用子代理，避免探索过程消耗主上下文。

##

​

培养你的直觉

本指南中的模式并非铁律。它们是普遍有效的起点，但未必适用于所有场景。有时你*应当*允许上下文累积，因为你正深入处理复杂问题且历史记录很有价值；有时你应跳过规划直接让Claude自行探索，因为任务本身具有探索性质；有时模糊的提示恰恰是合适的，因为你希望先观察Claude如何理解问题再加以约束。请留意有效的方法：当Claude产出优秀结果时，注意你的操作细节——提示结构、提供的上下文、所处模式；当Claude遇到困难时，反思原因——是上下文噪音过多？提示过于模糊？任务体量超出单次处理能力？久而久之，你将培养出任何指南都无法涵盖的直觉：懂得何时该具体何时该开放，何时需规划何时可探索，何时清理上下文何时任其累积。
