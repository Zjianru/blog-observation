由Prithvi Rajasekaran撰写，他是我们的Labs团队的成员。

在过去的几个月里，我一直在研究两个相互关联的问题：让Claude生成高质量的前端设计，以及让它在无人干预的情况下构建完整的应用程序。这项工作源于我们之前在前端设计技能和长时运行编码智能体harness方面的努力，在这些工作中，我和我的同事们通过提示工程和harness设计能够将Claude的性能提升到远高于基线的水平——但两者最终都遇到了瓶颈。

为了突破，我寻找了新颖的AI工程方法，这些方法在两个相当不同的领域中保持一致，一个由主观品味定义，另一个由可验证的正确性和可用性定义。从生成对抗网络（GANs）中获得灵感，我设计了一个包含生成器和评估器智能体的多智能体结构。构建一个能够可靠地——并且有品味地——对输出进行评分的评估器，意味着首先开发一套标准，将"这个设计好吗？"这样的主观判断转化为具体的、可评分的内容。

然后，我将这些技术应用于长时运行的自主编码，从我们之前的harness工作中借鉴了两个经验教训：将构建分解为可处理的块，以及使用结构化工件在会话之间传递上下文。最终结果是一个包含规划器、生成器和评估器的三智能体架构——在多小时的自主编码会话中生成丰富的全栈应用程序。

为什么朴素实现不够用

我们之前已经展示过，harness设计对长时运行的智能体编码有效性有重大影响。在一个早期实验中，我们使用初始化器智能体将产品规格分解为任务列表，一个编码智能体一次实现一个特性，然后将工件传递给下一个会话以携带上下文。更广泛的开发者社区已经收敛到类似的见解，诸如"Ralph Wiggum"方法使用钩子或脚本让智能体保持连续迭代循环。

但一些问题仍然存在。对于更复杂的任务，智能体仍然倾向于随着时间推移而失控。在分析这个问题时，我们观察到执行这类任务的智能体有两个常见的失败模式。

第一个是模型在处理长任务时会因上下文窗口填满而失去连贯性（参见我们关于上下文工程的帖子）。一些模型还表现出"上下文焦虑"，即它们在接近认为的上下文限制时开始过早地结束工作。上下文重置——完全清除上下文窗口并启动一个新的智能体，结合携带前一个智能体状态和下一步的结构化交接——可以解决这两个问题。

这与紧凑化不同，在紧凑化中，对话的前面部分被就地总结，以便同一个智能体可以在缩短的历史中继续。紧凑化保持了连续性，但没有给智能体一个干净的状态，这意味着上下文焦虑可能仍然存在。重置提供了一个干净的状态，但代价是交接工件必须有足够的状态让下一个智能体干净地接工作。在我们之前的测试中，我们发现Claude Sonnet 4.5表现出足够强烈的上下文焦虑，仅靠紧凑化不足以实现强大的长任务性能，所以上下文重置成为harness设计的基本要素。这解决了核心问题，但为每个harness运行增加了编排复杂性、令牌开销和延迟。

第二个问题是我们之前没有解决的，即自我评估。当被要求评估他们产生的工作时，智能体倾向于自信地赞扬工作——即使对于人类观察者来说，质量明显平庸。这个问题在设计等主观任务中尤为突出，那里没有等同于可验证软件测试的二元检查。布局是否感觉精致还是普通是一个判断，智能体在评价自己的工作时可靠地偏向正面。

然而，即使在确实有可验证结果的任务上，智能体在完成任务时仍然有时会表现出妨碍其性能的判断失误。将执行工作的智能体与判断工作的智能体分开被证明是解决这个问题的有力杠杆。这种分离本身并不能立即消除那种宽容；评估器仍然是一个倾向于对LLM生成的内容慷慨的LLM。但是调优一个独立的评估器变得怀疑自己工作比让生成器批评自己的工作要可行得多，而且一旦存在外部反馈，生成器就有了具体的东西来迭代。

前端设计：让主观质量可评分

我从前端设计开始实验，那里自我评估问题最为明显。在没有任何干预的情况下，Claude通常会倾向于安全的、可预测的布局，这些布局在技术上是可用的，但在视觉上平凡。

两个见解塑造了我为前端设计构建的harness。首先，虽然美学不能完全简化为分数——而且个人品味总是会有所不同——但它们可以通过编码设计原则和偏好的评分标准来改进。"这个设计美吗？"很难一致地回答，但"这是否符合我们的好设计原则？"给了Claude一个具体的评分依据。其次，通过将前端生成与前端评分分离，我们可以创建一个反馈循环，推动生成器产生更强的输出。

我强调设计质量和原创性而不是工艺和功能性。Claude在工艺和功能性方面默认已经表现很好，因为所需的技术能力往往自然而然地来到模型。但是，在设计和原创性方面，Claude经常产生最多算是平淡的输出。这些标准明确地惩罚了高度通用的"AI垃圾"模式，并且通过更重视设计和原创性，它推动模型承担更多的美学风险。

我使用带有详细分数分解的小样本示例来校准评估器。这确保了评估器的判断与我的偏好一致，并减少了跨迭代的分数漂移。

我使用Claude Agent SDK构建了这个循环，编排很简单。生成器智能体首先根据用户提示创建一个HTML/CSS/JS前端。我给评估器Playwright MCP，让它可以直接与实时页面交互，然后在评分每个标准之前写出详细的批评。实际上，评估器会自己导航页面，截图并仔细研究实现，然后才产生其评估。那个反馈作为下一次迭代的输入返回给生成器。我每个生成运行5到15次迭代，每次迭代通常推动生成器在响应评估器的批评时朝着一个更独特的方向发展。因为评估器主动导航页面而不是对静态截图评分，每个周期都需要真实的挂钟时间。完整运行延伸至四个小时。我还指示生成器在每次评估后做出战略决策：如果分数趋势良好则深化当前方向，或者如果方法不起作用则完全转向不同的美学。

跨运行，评估器的评估在迭代前改进然后趋于平稳，仍有空间剩余。一些生成增量改进。其他的在迭代之间采取了尖锐的美学转变。

标准的措辞以我完全未预料到的方式引导了生成器。包括"最好的设计是博物馆级别的"这样的短语推动设计走向特定的视觉收敛，表明与标准相关的提示直接塑造了输出的特性。

虽然分数通常随着迭代改进，但模式并不总是干净地线性化。后来的实现总体上往往更好，但我经常看到我更喜欢中间迭代而不是最后一次迭代的情况。实现复杂性也倾向于在轮次中增加，生成器在响应评估器的反馈时寻求更雄心勃勃的解决方案。即使在第一次迭代中，输出也明显比没有任何提示的基线好得多，这表明标准和相关语言本身在评估器反馈导致进一步改进之前就将模型引离了通用默认值。

在一个值得注意的例子中，我提示模型为一家荷兰艺术博物馆创建一个网站。到第九次迭代时，它为一个虚构的博物馆产生了一个干净的、深色主题的着陆页。该页面在视觉上精致，但基本上符合我的预期。然后，在第十个周期，它完全放弃了这种方法，并将网站重新想象为一个空间体验：一个用CSS透视渲染的棋盘格地板的3D房间，艺术品以自由形式的位置挂在墙上，以及基于门道的导航在画廊房间之间而不是滚动或点击。这是我之前在单次生成中从未见过的创造性飞跃。

扩展到全栈编码

有了这些发现，我将这种GAN启发的模式应用到了全栈开发。生成器-评估器循环自然映射到软件开发生命周期，在那里代码审查和QA扮演与设计评估器相同的结构角色。

架构

在我们之前的长时运行harness中，我们用初始化器智能体、一次处理一个特性的编码智能体以及会话之间的上下文重置解决了连贯的多会话编码问题。上下文重置是一个关键的解锁：harness使用Sonnet 4.5，它表现出前面提到的"上下文焦虑"倾向。创建一个在上下文重置之间良好工作的harness是保持模型任务的关键。Opus 4.5在很大程度上自己消除了那种行为，所以我能够完全从harness中删除上下文重置。智能体作为跨整个构建的连续会话运行，Claude Agent SDK的自动紧凑化在此过程中处理上下文增长。

对于这项工作，我在原始harness的基础上构建了一个三智能体系统，每个智能体都解决了我在之前的运行中观察到的特定差距。该系统包含以下智能体角色：

规划器：我们之前的长时运行harness要求用户预先提供详细规格。我想自动化那个步骤，所以我创建了一个规划器智能体，接受一个简单的1-4句提示并将其扩展为完整的产品规格。我提示它要大胆地对待范围，专注于产品背景和高层技术设计而不是详细的技术实现。强调这一点是因为担心如果规划器预先指定细粒度的技术细节并出错，规格中的错误会级联到下游实现中。让智能体在要产生的可交付成果上受到约束，然后让它们在工作时弄清楚路径，这似乎更聪明。我还要求规划器寻找将AI功能编织到产品规格中的机会。（参见底部附录中的示例。）

生成器：之前harness中的一次一个特性的方法对范围管理很有效。我在这里应用了类似的模型，指示生成器以冲刺的方式工作，一次从规格中选择一个特性。每个冲刺用React、Vite、FastAPI和SQLite（后来是PostgreSQL）栈实现应用程序，并指示生成器在将工作交给QA之前在每个冲刺结束时自我评估其工作。它也有git用于版本控制。

评估器：之前harness的应用程序看起来令人印象深刻，但当你实际尝试使用它们时仍然有真正的bug。为了捕捉这些，评估器使用Playwright MCP点击运行中的应用程序，就像用户会做的那样，测试UI功能、API端点和数据库状态。然后它根据它发现的bug和一套针对产品规格深度、功能性、视觉设计和代码质量的标准——改编自前端实验——对每个冲刺进行评分。每个标准都有一个硬阈值，如果任何一个低于它，冲刺失败，生成器会收到关于哪里出错的详细反馈。在每个冲刺之前，生成器和评估器协商一个冲刺合同：在任何代码编写之前商定对该工作块的"完成"定义。这之所以存在是因为产品规格故意是高层级的，我想要一个步骤来弥合用户故事和可测试实现之间的差距。生成器提出它将构建什么以及如何验证成功，评估器审查该提案以确保生成器构建的是正确的东西。两者迭代直到达成一致。

通信通过文件处理：一个智能体写一个文件，另一个智能体读取它并在那个文件内或用一个新的文件回应，然后前一个智能体会读取它。生成器然后根据协商的合同构建，然后再将工作交给QA。这保持工作忠于规格而不会过早地过度指定实现。

在评估了solo运行后，我将注意力转向harness运行。这个运行从相同的单句提示开始，但规划器步骤将该提示扩展为一个跨十个冲刺的16个特性规格。它远远超出了solo运行的尝试。除了核心编辑器和播放模式，规格还要求精灵动画系统、行为模板、音效和音乐、AI辅助精灵生成器和关卡设计师，以及带有可分享链接的游戏导出。我给规划器访问我们的前端设计技能，它读取并用来为应用程序创建视觉设计语言作为规格的一部分。对于每个冲刺，生成器和评估器协商一个定义该冲刺具体实现细节的合同，以及将用于验证完成的可测试行为。

该应用程序立即显示出比solo运行更多的精致和流畅。画布使用了整个视口，面板大小合理，界面具有符合规格中设计方向的统一视觉标识。solo运行中的一些笨拙确实保留了下来——工作流程仍然没有清楚表明你应该在尝试填充关卡之前构建精灵和实体，我必须通过摸索来弄清楚。这读作基础模型的产品直觉差距，而不是harness设计要解决的问题，尽管它确实表明了一个地方，在harness内有针对性地迭代可以帮助进一步提高输出质量。

通过编辑器，与solo相比，新运行的优势变得更加明显。精灵编辑器更丰富、功能更完善，工具面板更干净，颜色选择器更好，缩放控制更可用。

因为我要求规划器将其AI功能编织到其规格中，应用程序还带有一个内置的Claude集成，让我可以通过提示生成游戏的不同部分。这大大加快了整个工作流程。

让评估器达到这个水平需要工作。开箱即用，Claude是一个糟糕的QA智能体。在早期运行中，我看到它识别合法问题，然后说服自己决定它们不是什么大问题并批准工作。它也倾向于表面地测试，而不是探测边缘情况，所以更微妙的bug经常悄悄通过。调优循环是阅读评估器的日志，找到其判断与我的判断不同的例子，并更新QA的提示以解决那些问题。经过几轮这种开发循环，评估器的评分方式变得合理。即使那样，harness输出显示了模型QA能力的极限：小布局问题，某些地方感觉不直观的交互，以及评估器没有彻底行使的更深层嵌套功能中的未发现bug。显然还有更多的验证空间可以通过进一步调优来获取。但与solo运行相比，应用程序的核心功能根本无法工作，提升是明显的。

迭代harness

第一组harness结果令人鼓舞，但它也很笨重、慢且昂贵。合理的下一步是找到简化harness而不降低其性能的方法。这部分是常识，部分是一个更一般的原则：harness中的每个组件都编码了关于模型无法单独完成的事情的假设，这些假设值得进行压力测试，因为它们可能不正确，而且随着模型改进会很快过时。我们关于构建有效智能体的博客文章将基本思想框架为"找到最简单的解决方案，只有在需要时才增加复杂性"，这是任何维护智能体harness的人一致显示的模式。

在我的第一个简化尝试中，我大幅削减了harness并尝试了一些创造性的新想法，但我无法复制原始性能。这也变得难以判断harness设计的哪些部分实际上是承重的，以什么方式。基于那个经验，我转向了更有条理的方法，一次移除一个组件并审查它对最终结果的影响。

当我经历这些迭代周期时，我们还发布了Opus 4.6，这为减少harness复杂性提供了进一步的动力。有充分的理由期待4.6比4.5需要更少的脚手架。从我们的发布博客："[Opus 4.6]更仔细地计划，在更长时间内维持智能体任务，可以更可靠地在更大的代码库中操作，并有更好的代码审查和调试技能来捕获自己的错误。"它在长上下文检索方面也有实质性改进。这些都是harness被构建来补充的能力。

移除冲刺结构

我从完全移除冲刺结构开始。冲刺结构有助于将工作分解成块供模型连贯地处理。鉴于Opus 4.6的改进，有充分的理由相信模型可以在没有这种分解的情况下本地处理这项工作。

我保留了规划器和评估器，因为每个继续增加明显价值。没有规划器，生成器范围不足：给定原始提示，它会在没有先制定规格的情况下开始构建，最终创建比规划器做的功能更少的应用程序。

随着冲刺结构的移除，我将评估器转移到运行结束时的一次通过，而不是每个冲刺评分。由于模型更有能力，它改变了评估器对某些运行的承重程度，其有用性取决于任务相对于模型可以可靠地独立完成的事情的位置。在4.5上，边界很近：我们的构建处于生成器独立做好工作的边缘，评估器在整个构建中捕获有意义的问题。在4.6上，模型的原始能力增加了，所以边界向外移动。曾经需要评估器检查才能连贯实现的任务现在经常在生成器独立处理的范围内，对于该范围内的任务，评估器成为不必要的开销。但是对于构建中仍在生成器能力边缘的部分，评估器继续给予真正的提升。

实际的含义是评估器不是一个固定的二进制决策。当任务位于当前模型可靠地独立完成的事情之外时，它是值得成本的。

除了结构性简化，我还添加了提示来改进harness如何将AI功能构建到每个应用程序中，具体是让生成器构建一个能够通过工具驱动应用程序自身功能的适当智能体。这需要真正的迭代，因为相关知识足够新，Claude的训练数据覆盖得很稀疏。但是通过足够的调优，生成器正确地构建了智能体。

接下来是什么

随着模型继续改进，我们可以大致预期它们能够工作更长时间，处理更复杂的任务。在某些情况下，这意味着围绕模型的脚手架随着时间推移变得不那么重要，开发人员可以等待下一个模型并看到某些问题自行解决。另一方面，模型越好，就越有空间开发可以实现模型基线无法做到的复杂任务的harness。

考虑到这一点，这项工作有几个值得延续的经验。在你构建的模型上进行实验，阅读其在现实问题上的痕迹，并调优其性能以达到你期望的结果，这始终是很好的做法。在处理更复杂的任务时，有时有空间通过分解任务并对问题的每个方面应用专门的智能体来获得提升。当新模型落地时，通常是很好的做法是重新检查harness，剥离不再承重的部分，并添加新部分以实现以前可能无法实现的能力。

从这项工作中，我的信念是，有趣的harness组合空间不会随着模型改进而缩小。相反，它在移动，对于AI工程师来说，有趣的工作是继续寻找下一个新的组合。
