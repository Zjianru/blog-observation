# Claude Code 最佳实践

从环境配置到并行会话扩展，获取 Claude Code 最大效能的技巧与模式。

Claude Code 是一个智能体驱动的编程环境。与回答完问题就等待的聊天机器人不同，Claude Code 能够读取您的文件、运行命令、进行修改，并在您观看、重定向或完全离开的情况下自主解决问题。

这改变了您的工作方式。您无需亲自编写代码再让 Claude 审查，而是描述您想要什么，由 Claude 来构思如何构建。Claude 会进行探索、规划和实施。

但这种自主性仍然伴随着学习曲线。Claude 在您需要理解的某些约束条件下工作。

本指南涵盖了在 Anthropic 内部团队以及在不同代码库、语言和环境中使用 Claude Code 的工程师中已被证明有效的模式。关于智能体循环在底层如何工作，请参阅《Claude Code 工作原理》。

大多数最佳实践都基于一个约束：Claude 的上下文窗口填充速度很快，并且随着填充其性能会下降。

Claude 的上下文窗口保存着您的整个对话，包括每条消息、Claude 读取的每个文件以及每个命令输出。然而，这个窗口可能很快被填满。一次调试会话或代码库探索就可能生成并消耗数万个令牌。

这很重要，因为随着上下文填充，LLM 的性能会下降。当上下文窗口即将填满时，Claude 可能会开始"忘记"先前的指令或犯更多错误。上下文窗口是需要管理的最重要资源。要了解会话在实践中如何被填满，请观看一个交互式演练，了解启动时加载了什么以及读取每个文件的成本。使用自定义状态行持续跟踪上下文使用情况，并参阅《减少令牌使用》以获取降低令牌使用量的策略。

## 为 Claude 提供验证其工作的方式

包含测试、截图或预期输出，以便 Claude 能够自我检查。这是您可以做的最高杠杆率的事情。

当 Claude 能够验证自己的工作（例如运行测试、比较截图和验证输出）时，其表现会显著提高。

如果没有明确的成功标准，它可能会产生看起来正确但实际上无法工作的东西。您成为唯一的反馈循环，每个错误都需要您的关注。

| 策略 | 之前 | 之后 |
| :--- | :--- | :--- |
| **提供验证标准** | "实现一个验证电子邮件地址的函数" | "编写一个 validateEmail 函数。示例测试用例：user@example.com 为 true，invalid 为 false，user@.com 为 false。实现后运行测试" |
| **视觉验证 UI 更改** | "让仪表板看起来更好" | "[粘贴截图] 实现此设计。对结果截图并与原图比较。列出差异并修复它们" |
| **解决根本原因，而非症状** | "构建失败了" | "构建失败并出现此错误：[粘贴错误]。修复它并验证构建成功。解决根本原因，不要压制错误" |

UI 更改可以使用 Claude in Chrome 扩展进行验证。它会在您的浏览器中打开新标签页，测试 UI，并迭代直到代码正常工作。

您的验证也可以是测试套件、linter 或检查输出的 Bash 命令。投入精力使您的验证坚如磐石。

## 先探索，再规划，后编码

将研究和规划与实施分开，以避免解决错误的问题。

让 Claude 直接跳转到编码可能会产生解决错误问题的代码。使用"规划模式"将探索与执行分开。

推荐的工作流程分为四个阶段：

1.  **探索**
    进入规划模式。Claude 读取文件并回答问题，但不进行更改。

2.  **规划**
    要求 Claude 创建详细的实施计划。按 Ctrl+G 在文本编辑器中打开计划，以便在 Claude 继续之前直接编辑。

3.  **实施**
    切换回正常模式，让 Claude 进行编码，并根据其计划进行验证。

4.  **提交**
    要求 Claude 提交并附上描述性消息，并创建 PR。

规划模式很有用，但也增加了开销。

对于范围明确且修复很小（例如修复拼写错误、添加日志行或重命名变量）的任务，可以直接要求 Claude 执行。

当您不确定方法、更改涉及多个文件或不熟悉正在修改的代码时，规划最有用。如果您可以用一句话描述差异，则可以跳过规划。

## 在提示中提供具体上下文

您的指令越精确，需要的修正就越少。

Claude 可以推断意图，但它无法读懂您的想法。请引用特定文件、提及约束条件并指向示例模式。

| 策略 | 之前 | 之后 |
| :--- | :--- | :--- |
| **界定任务范围**。指定哪个文件、什么场景以及测试偏好。 | "为 foo.py 添加测试" | "为 foo.py 编写一个测试，覆盖用户注销的边缘情况。避免使用模拟对象。" |
| **指向来源**。引导 Claude 找到可以回答问题的来源。 | "为什么 ExecutionFactory 有如此奇怪的 API？" | "查看 ExecutionFactory 的 git 历史记录，并总结其 API 是如何形成的" |
| **参考现有模式**。引导 Claude 参考代码库中的模式。 | "添加一个日历小部件" | "查看主页上现有小部件的实现方式以理解模式。HotDogWidget.php 是一个很好的例子。遵循该模式实现一个新的日历小部件，让用户可以选择月份，并通过向前/向后分页来选择年份。从头开始构建，除了代码库中已使用的库外，不使用其他库。" |
| **描述症状**。提供症状、可能的位置以及"已修复"的样子。 | "修复登录错误" | "用户报告会话超时后登录失败。检查 src/auth/ 中的身份验证流程，特别是令牌刷新。编写一个重现该问题的失败测试，然后修复它" |

当您正在探索并且可以承受纠正过程时，模糊的提示可能有用。像"您会改进这个文件中的哪些地方？"这样的提示可能会揭示您没有想到要询问的内容。

## 提供丰富内容

使用 `@` 引用文件、粘贴截图/图像或直接管道传输数据。

*   **使用 `@` 引用文件**，而不是描述代码所在位置。Claude 会在回复前读取该文件。
*   **直接粘贴图像**。将图像复制/粘贴或拖放到提示中。
*   **提供文档和 API 参考的 URL**。使用 `/permissions` 将常用域名加入允许列表。
*   **管道传输数据**，运行 `cat error.log | claude` 直接发送文件内容。
*   **让 Claude 获取所需内容**。告诉 Claude 使用 Bash 命令、MCP 工具或读取文件来拉取上下文。

## 配置您的环境

一些设置步骤可以显著提高 Claude Code 在所有会话中的效率。有关扩展功能的完整概述以及何时使用每个功能，请参阅《扩展 Claude Code》。

## 编写有效的 CLAUDE.md 文件

运行 `/init` 命令，根据当前项目结构生成一个初始的 CLAUDE.md 文件，然后随时间逐步完善。

CLAUDE.md 是一个特殊文件，Claude 在每次对话开始时都会读取它。包含 Bash 命令、代码风格和工作流程规则。这为 Claude 提供了无法仅从代码中推断出的持久上下文。

`/init` 命令会分析你的代码库，检测构建系统、测试框架和代码模式，为你提供一个可以完善的良好基础。

CLAUDE.md 文件没有固定的格式要求，但要保持简短且易于人类阅读。例如：

```markdown
# 代码风格

- 使用 ES 模块（import/export）语法，而非 CommonJS（require）
- 尽可能解构导入（例如：`import { foo } from 'bar'`）

# 工作流程

- 完成一系列代码更改后，务必进行类型检查
- 出于性能考虑，优先运行单个测试，而非整个测试套件
```

CLAUDE.md 在每个会话中都会加载，因此只包含广泛适用的内容。对于仅有时相关的领域知识或工作流程，请使用技能（skills）。Claude 会按需加载它们，而不会使每次对话变得臃肿。

保持简洁。对于每一行，问问自己："删除这一行会导致 Claude 犯错吗？" 如果不会，就删掉它。臃肿的 CLAUDE.md 文件会导致 Claude 忽略你的实际指令！

| 应包含 | 应排除 |
| :--- | :--- |
| Claude 无法猜测的 Bash 命令 | Claude 通过阅读代码可以弄清楚的内容 |
| 与默认设置不同的代码风格规则 | Claude 已经知道的标准语言约定 |
| 测试说明和首选的测试运行器 | 详细的 API 文档（改为链接到文档） |
| 仓库规范（分支命名、PR 约定） | 频繁变化的信息 |
| 特定于项目的架构决策 | 冗长的解释或教程 |
| 开发环境特性（必需的环境变量） | 代码库的文件级描述 |
| 常见的陷阱或非显而易见的行为 | 不言自明的实践，如"编写干净的代码" |

如果 Claude 持续做出你不希望的行为，尽管你已有规则禁止它，那么文件可能太长了，规则被淹没了。如果 Claude 问的问题在 CLAUDE.md 中已有答案，那么措辞可能含糊不清。将 CLAUDE.md 视为代码：当出现问题时审查它，定期修剪，并通过观察 Claude 的行为是否真的改变来测试更改。

你可以通过添加强调（例如，"重要"或"你必须"）来调整指令，以提高遵守程度。将 CLAUDE.md 提交到 git 中，以便你的团队可以贡献。该文件的价值会随时间积累。

CLAUDE.md 文件可以使用 `@path/to/import` 语法导入其他文件：

```markdown
项目概述请参阅 @README.md，可用的 npm 命令请参阅 @package.json。

# 附加说明

- Git 工作流程：@docs/git-instructions.md
- 个人覆盖设置：@~/.claude/my-project-instructions.md
```

你可以在以下几个位置放置 CLAUDE.md 文件：

- 主文件夹（`~/.claude/CLAUDE.md`）：适用于所有 Claude 会话
- 项目根目录（`./CLAUDE.md`）：提交到 git 中以便与团队共享
- 项目根目录（`./CLAUDE.local.md`）：个人项目特定笔记；将此文件添加到 `.gitignore` 中，以免与团队共享
- 父目录：对于 monorepos 很有用，其中 `root/CLAUDE.md` 和 `root/foo/CLAUDE.md` 都会自动被拉取
- 子目录：当处理这些目录中的文件时，Claude 会按需拉取子 CLAUDE.md 文件

## 配置权限

使用自动模式让分类器处理批准，使用 `/permissions` 来允许列表特定命令，或使用 `/sandbox` 进行操作系统级隔离。每种方式都能减少中断，同时让你保持控制。

默认情况下，Claude Code 会请求执行可能修改你系统的操作的权限：文件写入、Bash 命令、MCP 工具等。这是安全的，但很繁琐。在第十次批准之后，你就不再是真正在审查了，只是在点击通过。有三种方法可以减少这些中断：

**自动模式**：一个单独的分类器模型审查命令，只阻止看起来有风险的操作：权限提升、未知的基础设施或恶意内容驱动的操作。当你信任任务的大方向，但又不想点击通过每一步时，这是最佳选择

**权限允许列表**：允许你知道安全的特定工具，如 `npm run lint` 或 `git commit`

**沙盒化**：启用操作系统级隔离，限制文件系统和网络访问，允许 Claude 在定义的边界内更自由地工作

阅读更多关于权限模式、权限规则和沙盒化的信息。

## 使用 CLI 工具

告诉 Claude Code 在与外部服务交互时使用 CLI 工具，如 `gh`、`aws`、`gcloud` 和 `sentry-cli`。

CLI 工具是与外部服务交互时最高效的方式。如果你使用 GitHub，请安装 `gh` CLI。Claude 知道如何使用它来创建问题、打开拉取请求和读取评论。没有 `gh`，Claude 仍然可以使用 GitHub API，但未经身份验证的请求经常会遇到速率限制。

Claude 也能有效学习它原本不知道的 CLI 工具。尝试使用类似这样的提示：`使用 'foo-cli-tool --help' 来了解 foo 工具，然后用它来解决 A、B、C 问题`。

## 连接 MCP 服务器

运行 `claude mcp add` 来连接外部工具，如 Notion、Figma 或你的数据库。

通过 MCP 服务器，你可以要求 Claude 从问题跟踪器实现功能、查询数据库、分析监控数据、集成来自 Figma 的设计以及自动化工作流程。

## 设置钩子

使用钩子来处理每次都必须发生且零例外的情况。

钩子在 Claude 工作流程的特定点自动运行脚本。与 CLAUDE.md 指令（是建议性的）不同，钩子是确定性的，并保证操作会发生。

Claude 可以为你编写钩子。尝试这样的提示："编写一个在每次文件编辑后运行 eslint 的钩子" 或 "编写一个阻止写入 migrations 文件夹的钩子"。直接编辑 `.claude/settings.json` 来手动配置钩子，并运行 `/hooks` 来浏览已配置的内容。

## 创建技能

在 `.claude/skills/` 中创建 SKILL.md 文件，为 Claude 提供领域知识和可重用的工作流程。

技能用特定于你的项目、团队或领域的信息扩展 Claude 的知识。Claude 会在相关时自动应用它们，或者你可以直接用 `/skill-name` 调用它们。

通过添加一个包含 SKILL.md 的目录到 `.claude/skills/` 来创建技能：

`.claude/skills/api-conventions/SKILL.md`

```markdown
name: api-conventions
description: 我们服务的 REST API 设计约定

# API 约定

- URL 路径使用 kebab-case
- JSON 属性使用 camelCase
- 列表端点始终包含分页
- 在 URL 路径中版本化 API（/v1/, /v2/）
```

技能也可以定义你可以直接调用的可重复工作流程：

`.claude/skills/fix-issue/SKILL.md`

```markdown
name: fix-issue
description: 修复 GitHub 问题
disable-model-invocation: true

分析并修复 GitHub 问题：$ARGUMENTS。

1. 使用 `gh issue view` 获取问题详情
2. 理解问题中描述的问题
3. 在代码库中搜索相关文件
4. 实施必要的更改来修复问题
5. 编写并运行测试以验证修复
6. 确保代码通过代码检查和类型检查
7. 创建描述性的提交信息
8. 推送并创建 PR
```

运行 `/fix-issue 1234` 来调用它。对于具有副作用且你希望手动触发的工作流程，使用 `disable-model-invocation: true`。

## 创建自定义子代理

在 `.claude/agents/` 中定义专门的助手，Claude 可以将任务委托给它们进行隔离处理。

子代理在自己的上下文中运行，拥有自己的一套允许的工具。它们对于需要读取许多文件或需要专注处理而不干扰主对话的任务非常有用。

`.claude/agents/security-reviewer.md`

```markdown
name: security-reviewer
description: 审查代码中的安全漏洞
tools: Read, Grep, Glob, Bash
model: opus

你是一名高级安全工程师。审查代码中的：

- 注入漏洞（SQL、XSS、命令注入）
- 身份验证和授权缺陷
- 代码中的秘密或凭证
- 不安全的数据处理

提供具体的行引用和建议的修复方法。
```

明确告诉 Claude 使用子代理："使用子代理来审查这段代码的安全问题。"

## 安装插件

运行 `/plugin` 浏览市场。插件无需配置即可添加技能、工具和集成。

插件将技能、钩子、子代理和 MCP 服务器捆绑成一个可从社区和 Anthropic 安装的单元。如果你使用类型化语言，请安装代码智能插件，以便在编辑后为 Claude 提供精确的符号导航和自动错误检测。

关于在技能、子代理、钩子和 MCP 之间进行选择的指导，请参阅扩展 Claude Code。

## 有效沟通

与 Claude Code 的沟通方式会显著影响结果质量。

### 询问代码库问题

向 Claude 提出您会向资深工程师咨询的问题。

当接手新代码库时，使用 Claude Code 进行学习和探索。您可以向 Claude 提出与其他工程师交流时会问的同类问题：

- 日志系统是如何工作的？
- 如何创建新的 API 端点？
- foo.rs 第 134 行的 `async move { ... }` 有什么作用？
- CustomerOnboardingFlowImpl 处理哪些边界情况？
- 为什么这段代码在第 333 行调用 `foo()` 而不是 `bar()`？

以这种方式使用 Claude Code 是一种高效的入职工作流程，能缩短上手时间并减轻其他工程师的负担。无需特殊提示技巧：直接提问即可。

### 让 Claude 采访您

对于大型功能开发，可以先让 Claude 对您进行访谈。从一个简洁的提示开始，并要求 Claude 使用 AskUserQuestion 工具进行采访。

Claude 会询问您可能尚未考虑到的方面，包括技术实现、UI/UX、边界情况和权衡取舍。

> 我想构建 [简要描述]。请使用 AskUserQuestion 工具对我进行详细采访。
> 
> 询问技术实现、UI/UX、边界情况、关注点和权衡取舍。不要问显而易见的问题，深入探讨我可能没考虑到的难点。
> 
> 持续采访直到覆盖所有内容，然后编写完整的规范到 SPEC.md。

规范完成后，开启新会话执行实施。新会话拥有专注于实现的干净上下文，并且您有书面规范可供参考。

## 管理会话

对话具有持久性和可逆性。善用这一特性！

### 尽早并频繁修正方向

一旦发现 Claude 偏离轨道，立即纠正。

最佳结果来自紧密的反馈循环。虽然 Claude 偶尔能在首次尝试时就完美解决问题，但快速纠正通常能更快产生更好的解决方案。

- **Esc 键**：在 Claude 执行中途按 Esc 键停止操作。上下文会保留，因此您可以重新引导。
- **Esc + Esc 或 /rewind**：按两次 Esc 或运行 `/rewind` 打开回退菜单，恢复之前的对话和代码状态，或从选定消息开始总结。
- **"撤销那个"**：让 Claude 撤销其更改。
- **/clear**：在不相关任务之间重置上下文。包含无关上下文的长会话可能降低性能。

如果在同一会话中针对同一问题纠正 Claude 超过两次，说明上下文已充满失败的尝试。运行 `/clear` 并使用结合了经验教训的更具体提示重新开始。干净的会话配合更好的提示，几乎总是优于积累了多次修正的长会话。

### 积极管理上下文

在不相关任务之间运行 `/clear` 重置上下文。

当接近上下文限制时，Claude Code 会自动压缩对话历史，在释放空间的同时保留重要代码和决策。

在长会话期间，Claude 的上下文窗口可能充满无关对话、文件内容和命令。这可能降低性能，有时还会分散 Claude 的注意力。

- 频繁使用 `/clear` 在任务间完全重置上下文窗口
- 自动压缩触发时，Claude 会总结最重要的内容，包括代码模式、文件状态和关键决策
- 如需更多控制，运行 `/compact <指令>`，例如 `/compact 专注于 API 变更`
- 要仅压缩部分对话，使用 Esc + Esc 或 `/rewind`，选择消息检查点，然后选择"从此处总结"。这会压缩该点之后的消息，同时保持较早的上下文完整
- 在 CLAUDE.md 中通过"压缩时始终保留修改文件的完整列表和任何测试命令"等指令自定义压缩行为，确保关键上下文在总结后得以保留

对于不需要保留在上下文中的快速问题，使用 `/btw`。答案会显示在可关闭的覆盖层中，永远不会进入对话历史，因此您可以在不增加上下文的情况下查看细节。

### 使用子代理进行调查

通过"使用子代理调查 X"委托研究任务。它们在独立的上下文中探索，保持您的主对话清洁以专注于实施。

由于上下文是根本性约束，子代理是最强大的可用工具之一。当 Claude 研究代码库时，它会读取大量文件，所有这些都会消耗您的上下文。子代理在独立的上下文窗口中运行并返回摘要：

> 使用子代理调查我们的认证系统如何处理令牌刷新，以及是否有任何现有的 OAuth 实用程序可供复用。

子代理探索代码库，读取相关文件，并返回发现结果，所有这些都不会干扰您的主对话。

您也可以在 Claude 实施后使用子代理进行验证：

> 使用子代理审查此代码的边界情况

### 通过检查点回退

Claude 的每个操作都会创建检查点。您可以将对话、代码或两者恢复到任何先前的检查点。

Claude 会在更改前自动创建检查点。双击 Escape 或运行 `/rewind` 打开回退菜单。您可以仅恢复对话、仅恢复代码、同时恢复两者，或从选定消息开始总结。详见检查点功能说明。

无需精心规划每个步骤，您可以指示 Claude 尝试有风险的操作。如果无效，回退并尝试不同方法。检查点跨会话持久保存，因此即使关闭终端，稍后仍可回退。

检查点仅跟踪 Claude 所做的更改，不跟踪外部进程。这不是 git 的替代品。

### 恢复对话

运行 `claude --continue` 从上次中断处继续，或 `--resume` 从最近会话中选择。

Claude Code 在本地保存对话。当任务跨越多个会话时，您无需重新解释上下文：

```bash
claude --continue    # 恢复最近的对话
claude --resume      # 从最近对话中选择
```

使用 `/rename` 为会话指定描述性名称，如"oauth-migration"或"debugging-memory-leak"，以便后续查找。将会话视为分支：不同的工作流可以拥有独立、持久的上下文。

## 自动化与规模化扩展

一旦你能够熟练运用单个Claude，可以通过并行会话、非交互模式和扇出模式来倍增产出。

到目前为止的所有内容都基于单用户、单Claude、单会话的假设。但Claude Code具备水平扩展能力。本节介绍的技巧将展示如何实现更高的工作效率。

### 运行非交互模式

在CI流程、预提交钩子或脚本中使用 `claude -p "提示词"`。添加 `--output-format stream-json` 参数可获得流式JSON输出。

通过 `claude -p "你的提示词"`，你可以在无会话的情况下非交互式运行Claude。非交互模式让你能够将Claude集成到CI流水线、预提交钩子或任何自动化工作流中。输出格式支持以编程方式解析结果：纯文本、JSON或流式JSON。

```bash
# 一次性查询
claude -p "解释这个项目的功能"

# 脚本使用的结构化输出
claude -p "列出所有API端点" --output-format json

# 实时处理的流式输出
claude -p "分析这个日志文件" --output-format stream-json
```

### 运行多个Claude会话

并行运行多个Claude会话可以加速开发、运行隔离实验或启动复杂工作流。

主要有三种并行会话运行方式：

**Claude Code桌面应用**：可视化管理多个本地会话。每个会话都拥有独立的worktree工作树。

**Claude Code网页版**：在Anthropic的安全云基础设施的隔离虚拟机中运行。

**智能体团队**：通过共享任务、消息传递和团队领导实现多个会话的自动化协调。

除了并行化工作，多会话还能实现质量导向的工作流。全新的上下文环境能改善代码审查效果，因为Claude不会对它刚编写的代码产生偏见。

例如，采用编写者/审查者模式：

| 会话A（编写者） | 会话B（审查者） |
|----------------|----------------|
| 为我们的API端点实现限流器 | |
| | 审查@src/middleware/rateLimiter.ts中的限流器实现。关注边界情况、竞态条件以及与现有中间件模式的一致性 |
| 根据审查反馈：[会话B输出]解决这些问题 | |

测试工作也可以采用类似模式：让一个Claude编写测试，另一个编写通过测试的代码。

### 跨文件扇出处理

通过循环任务调用 `claude -p` 处理每个项目。使用 `--allowedTools` 参数为批量操作限定权限范围。

对于大规模迁移或分析任务，可以将工作分配到多个并行的Claude调用中：

1. **生成任务列表**
   
   让Claude列出所有需要迁移的文件（例如，列出所有2000个需要迁移的Python文件）

2. **编写循环处理脚本**
   
   ```bash
   for file in $(cat files.txt); do
     claude -p "将$file从React迁移到Vue。返回OK或FAIL。" \
       --allowedTools "Edit,Bash(git commit *)"
   done
   ```

3. **先测试少量文件，再规模化运行**

根据前2-3个文件出现的问题优化提示词，然后在完整文件集上运行。`--allowedTools` 标志限制了Claude的操作权限，这在无人值守运行时尤为重要。

你还可以将Claude集成到现有的数据处理流水线中：

```bash
claude -p "<你的提示词>" --output-format json | your_command
```

开发时使用 `--verbose` 参数进行调试，生产环境关闭该选项。

### 使用自动模式自主运行

对于需要连续执行且具备后台安全检查的场景，使用自动模式。分类器模型会在命令执行前进行审查，阻止权限升级、未知基础设施操作和恶意内容驱动的行为，同时允许常规工作无需提示继续执行。

```bash
claude --permission-mode auto -p "修复所有lint错误"
```

对于使用 `-p` 标志的非交互运行，如果分类器多次阻止操作，自动模式会中止执行，因为没有用户可以回退。详见自动模式回退阈值说明。

## 避免常见失败模式

以下是常见错误模式。早期识别能节省大量时间：

**厨房水槽式会话**：从一个任务开始，然后询问Claude不相关的内容，再回到第一个任务。上下文充满无关信息。

*修复方案*：无关任务间使用 `/clear` 命令。

**反复纠正**：Claude执行错误，你纠正它，仍然错误，再次纠正。上下文被失败的尝试污染。

*修复方案*：两次纠正失败后，使用 `/clear` 并编写包含所学经验的更好的初始提示词。

**过度详细的CLAUDE.md**：如果CLAUDE.md过长，Claude会忽略一半内容，因为重要规则淹没在噪音中。

*修复方案*：无情删减。如果Claude在没有某条指令时已能正确执行，删除该指令或将其转换为钩子。

**信任与验证的差距**：Claude生成看似合理但未处理边界情况的实现。

*修复方案*：始终提供验证（测试、脚本、截图）。如果无法验证，不要部署。

**无限探索**：要求Claude"调查"某事但未限定范围。Claude读取数百个文件，填满上下文。

*修复方案*：限定调查范围或使用子智能体，避免探索消耗主上下文。

## 培养直觉能力

本指南中的模式并非一成不变。它们是普遍有效的起点，但可能不适用于所有情况。

有时你应该让上下文累积，因为你正深入处理一个复杂问题，历史记录很有价值。有时你应该跳过规划，让Claude自行探索，因为任务是探索性的。有时模糊的提示词恰恰合适，因为你希望先观察Claude如何理解问题再加以约束。

关注有效的方法。当Claude产生优秀输出时，注意你的操作：提示词结构、提供的上下文、所处的模式。当Claude遇到困难时，思考原因：上下文太杂乱？提示词太模糊？任务太大无法一次完成？

随着时间的推移，你将培养出任何指南都无法涵盖的直觉。你会知道何时该具体、何时该开放，何时该规划、何时该探索，何时该清除上下文、何时该让其累积。
