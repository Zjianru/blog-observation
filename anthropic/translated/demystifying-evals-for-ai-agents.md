介绍

好的评估帮助团队更有信心地交付AI智能体。没有它们，很容易陷入反应循环——只在生产中捕获问题，在那里修复一个失败会产生其他问题。评估使问题和行为变化在影响用户之前可见，它们的价值在智能体的生命周期中会复合。

正如我们在构建有效智能体中所描述的，智能体在多个回合中操作：调用工具、修改状态并根据中间结果进行调整。这些使AI智能体有用的相同能力——自主性、智能和灵活性——也使它们更难评估。

通过我们的内部工作以及与前沿智能体开发客户的合作，我们学会了如何为智能体设计更严格和更有用的评估。以下是跨一系列智能体架构和实际部署用例的经验。

评估的结构

评估（"eval"）是对AI系统的测试：给AI一个输入，然后对其输出应用评分逻辑以衡量成功。在这篇文章中，我们专注于可以开发期间运行而无需真实用户的自动化评估。

单回合评估很简单：一个提示、一个响应和评分逻辑。对于早期的LLM，单回合、非智能体评估是主要评估方法。随着AI能力进步，多回合评估变得越来越常见。

在一个简单的eval中，智能体处理一个提示，评分器检查输出是否符合预期。对于更复杂的多回合eval，一个编码智能体接收工具、一个任务（在这个案例中构建一个MCP服务器）和一个环境，执行一个"智能体循环"（工具调用和推理），并用实现更新环境。然后评分使用单元测试来验证工作的MCP服务器。

智能体评估更加复杂。智能体在多个回合中使用工具，修改环境中的状态并沿途调整——这意味着错误可以传播和复合。前沿模型也可以找到超越静态评估限制的创造性解决方案。例如，Opus 4.5通过发现政策中的一个漏洞解决了关于预订航班的τ2-bench问题。它"失败"了按书面说明的评估，但实际上提出了对用户更好的解决方案。

为什么要构建评估？

当团队第一次开始构建智能体时，他们可以通过手动测试、dogfooding和直觉组合取得惊人的进展。更严格的评估可能看起来像是减慢交付的overhead。但经过早期原型设计阶段，一旦智能体投入生产并开始扩展，没有评估就开始构建工作会开始崩溃。

崩溃点通常出现在用户报告智能体在更改后感觉更糟时，团队"蒙着眼睛飞行"，除了猜测和检查没有办法验证。没有评估，调试是被动的：等待投诉、手动重现、修复bug，并希望没有其他东西回归。团队无法区分真实回归和噪声，无法在发布前自动测试数百个场景的更改，或衡量改进。

我们已经多次看到这种进展。例如，Claude Code始于基于Anthropic员工和外部用户反馈的快速迭代。后来，我们添加了评估——首先是针对简洁性和文件编辑等狭窄领域，然后是针对过度工程化等更复杂行为。这些评估帮助识别问题、指导改进，并集中研究和产品合作。结合生产监控、A/B测试、用户研究等，评估提供了在扩展时继续改进Claude Code的信号。

编写评估在智能体生命周期的任何阶段都有用。早期，评估强制产品团队指定对智能体来说成功意味着什么，后来它们有助于保持一致的质量标准。

Descript的智能体帮助用户编辑视频，所以他们围绕成功编辑工作流程的三个维度构建评估：不要破坏东西、做我要求的、以及做得好。它们从手动评分发展到LLM评分器，标准由产品团队定义并有定期人工校准，现在定期运行两套独立的测试进行质量基准测试和回归测试。Bolt AI团队在已经拥有广泛使用的智能体之后才添加评估。3个月内，他们构建了一个评估系统，运行他们的智能体并用静态分析对输出进行评分，使用浏览器智能体测试应用程序，并采用LLM法官评估指令遵循等行为。

一些团队在开发开始时创建评估；其他人在规模化时添加评估，当评估成为改进智能体的瓶颈时。评估在智能体开发开始时特别有用，可以明确编码预期行为。两个工程师阅读相同的初始规格可能会对AI应该如何处理边缘情况得出不同的解释。评估套件解决了这种模糊性。无论何时创建，评估都有助于加速开发。

评估也塑造了您采用新模型的速度。当更强大的模型出现时，没有评估的团队面临数周的测试，而有评估的竞争对手可以快速确定模型的优势、调优提示，并在几天内升级。

一旦评估存在，您免费获得基线和回归测试：延迟、令牌使用量、每任务成本和错误率可以在静态任务库上跟踪。评估也可以成为产品和研究团队之间最高带宽的沟通渠道，定义研究人员可以优化的指标。显然，评估的好处远远超出跟踪回归和改进。它们复合的价值很容易被错过，因为成本前期可见而收益后期积累。

如何评估AI智能体

我们看到今天部署在规模的几类常见智能体，包括编码智能体、研究智能体、计算机使用智能体和对话智能体。每种类型可能跨广泛的行业部署，但它们可以使用类似技术进行评估。您不需要从头开始发明评估。以下部分描述了经过验证的技术 用于几种智能体类型。将这些方法作为基础，然后扩展到您的领域。

智能体的评分器类型

智能体评估通常结合三种类型的评分器：基于代码的、基于模型的和人类的。每个评分器评估成绩单或结果的某个部分。有效评估设计的关键组成部分是选择正确的评分器。

能力与回归评估

能力或"质量"评估问："这个智能体能做什么好事？"它们应该从低通过率开始，目标是智能体挣扎的任务，给团队一座要攀登的山。

回归评估问："智能体仍然处理它曾经处理的所有任务吗？"应该有接近100%的通过率。它们防止退步，因为分数下降表明有问题需要改进。当团队在能力评估上爬山时，运行回归评估也很重要，以确保更改不会在其他地方造成问题。

在智能体发布和优化后，通过率高的高分能力评估可以"毕业"成为持续运行以捕获任何漂移的回归套件。曾经测量"我们能这样做吗？"的任务然后测量"我们仍然能可靠地做到这一点吗？"

评估编码智能体

编码智能体编写、测试和调试代码，导航代码库并像人类开发人员一样运行命令。现代编码智能体的有效评估通常依赖于明确定义的任务、稳定的测试环境和生成代码的彻底测试。

确定性评分器对编码智能体很自然，因为软件评估通常很直接：代码运行吗？测试通过吗？两个广泛使用的编码智能体基准SWE-bench Verified和Terminal-Bench 遵循这种方法。SWE-bench Verified从流行的Python仓库给智能体GitHub问题，并通过运行测试套件对解决方案进行评分；解决方案只有在修复失败的测试而不破坏现有测试时才能通过。LLM在这一评估上从40%进步到一年内的>80%。Terminal-Bench采取不同路线：它测试端到端技术任务，如从源码构建Linux内核或训练ML模型。

一旦您有了一组用于验证编码任务关键结果的通过或失败测试，通常还需要对成绩单进行评分。例如，基于启发式的代码质量规则可以根据不仅仅是通过测试来评估生成的代码，具有明确标准的基于模型的评分器可以评估智能体如何调用工具或与用户交互等行为。

示例：编码智能体的理论评估

考虑一个智能体必须修复身份验证绕过漏洞的编码任务。如下图所示的说明性YAML文件，可以使用评分器和指标评估此智能体。

task:
  id: "fix-auth-bypass_1"
  desc: "Fix authentication bypass when password field is empty and ..."
  graders:
    - type: deterministic_tests
      required: [test_empty_pw_rejected.py, test_null_pw_rejected.py]
    - type: llm_rubric
      rubric: prompts/code_quality.md
    - type: static_analysis
      commands: [ruff, mypy, bandit]
    - type: state_check
      expect:
        security_logs: {event_type: "auth_blocked"}
    - type: tool_calls
      required:
        - {tool: read_file, params: {path: "src/auth/*"}}
        - {tool: edit_file}
        - {tool: run_tests}
  tracked_metrics:
    - type: transcript
      metrics:
        - n_turns
        - n_toolcalls
        - n_total_tokens
    - type: latency
      metrics:
        - time_to_first_token
        - output_tokens_per_sec
        - time_to_last_token

评估对话智能体

对话智能体在支持、销售或辅导等领域与用户互动。与传统聊天机器人不同，它们保持状态、使用工具并在对话中途采取行动。虽然编码和研究智能体也可能涉及与用户的多个回合互动，但对话智能体呈现了独特的挑战：互动质量本身就是您正在评估的一部分。有效的对话智能体评估通常依赖于可验证的最终状态结果和捕获任务完成和互动质量两者的标准。与大多数其他评估不同，它们经常需要第二个LLM来模拟用户。我们在alignment审计智能体中使用这种方法，通过扩展的对抗性对话对模型进行压力测试。

对话智能体的成功可能是多维的：工单是否解决（状态检查）、是否在<10回合内完成（成绩单约束）、语气是否适当（LLM标准）？两个包含多维性的基准是τ-Bench及其后继者τ2-Bench。这些模拟跨零售支持和航班预订等领域的多次回合互动，其中一个模型扮演用户角色而智能体导航真实场景。

示例：对话智能体的理论评估

考虑一个智能体必须处理沮丧客户的退款的支援任务。

graders:
  - type: llm_rubric
    rubric: prompts/support_quality.md
    assertions:
      - "Agent showed empathy for customer's frustration"
      - "Resolution was clearly explained"
      - "Agent's response grounded in fetch_policy tool results"
  - type: state_check
    expect:
      tickets: {status: resolved}

评估研究智能体

研究智能体收集、综合和分析信息，然后产生答案或报告等输出。与编码智能体不同，单元测试提供二元通过/失败信号，研究质量只能相对于任务来判断。"全面"、"有来源"或"正确"的内容取决于背景：市场扫描、收购尽职调查和科学报告各有不同的标准。

研究评估面临独特挑战：专家可能不同意综合是否全面， ground truth随着参考内容不断变化而转移，更长、更开放的输出为错误创造更多空间。例如，像BrowseComp这样的基准测试AI智能体是否可以在开放网络上大海捞针——问题设计为容易验证但难以解决。

构建研究智能体评估的一种策略是结合评分器类型。 groundedness检查验证索赔是否被检索来源支持，覆盖率检查定义好的答案必须包含的关键事实，来源质量检查确认咨询的来源是权威的，而不仅仅是第一个检索的。对于有客观正确答案的任务（"X公司Q3收入是多少？"），精确匹配有效。LLM可以标记不支持的索赔和覆盖差距，但也可以验证开放式综合的连贯性和完整性。

鉴于研究质量的主观性质，LLM标准应该经常根据专家人工判断进行校准，以有效评估这些智能体。

计算机使用智能体

计算机使用智能体通过与人类相同的界面与软件交互——截图、鼠标点击、键盘输入和滚动——而不是通过API或代码执行。它们可以使用任何带有图形用户界面（GUI）的应用程序，从设计工具到传统企业软件。评估需要在真实或沙盒环境中运行智能体，它可以使用的软件应用程序，并检查它是否达到预期结果。例如，WebArena测试基于浏览器的任务，使用URL和页面状态检查来验证智能体正确导航，以及后端状态验证用于修改数据的任务（确认订单实际已下达，而不仅仅是确认页面出现）。OSWorld将这扩展到完整操作系统控制，评估脚本在任务完成后检查各种工件：文件系统状态、应用程序配置、数据库内容和UI元素属性。

浏览器使用智能体需要在令牌效率性和延迟之间取得平衡。基于DOM的交互执行快速但消耗更多令牌，而基于截图的交互较慢但更节省令牌。例如，当要求Claude总结Wikipedia时，从DOM提取文本更高效。当在Amazon上寻找新的笔记本电脑包时，截屏更高效（因为提取整个DOM是令牌密集型的）。在我们的Claude for Chrome产品中，我们开发了评估来检查智能体是否为每个上下文选择正确的工具。这使我们能够更快、更准确地完成基于浏览器的任务。

如何思考智能体评估中的非决定论

无论智能体类型如何，智能体行为在运行之间存在差异，这使得评估结果比初看时更难解释。每个任务有自己的成功率——也许一个任务90%，另一个50%——在一个eval运行中通过的任务可能在下一个运行中失败。有时，我们想要测量的是（多少次）智能体成功完成任务。

两个指标帮助捕捉这种细微差别：

pass@k测量智能体在k次尝试中至少获得一次正确解决方案的可能性。随着k增加，pass@k分数上升：更多的"射门机会"意味着至少1次成功的几率更高。50%的pass@1分数意味着模型在第一次尝试时在半数任务上成功。在编码中，我们通常最感兴趣智能体第一次尝试找到解决方案——pass@1。在其他情况下，提出许多解决方案只要其中一个有效就有效。

pass^k测量所有k次试验成功的概率。随着k增加，pass^k下降，因为要求更多试验之间的一致性是一个更高的标准。如果您的智能体每次试验成功率为75%，您运行3次试验，所有三次都通过的概率是(0.75)³ ≈ 42%。这个指标对面向客户的智能体尤其重要，用户每次都期望可靠行为。

pass@k和pass^k随着试验增加而分歧。在k=1时，它们相同（都等于每次试验成功率）。到k=10，它们讲述相反的故事：pass@k接近100%而pass^k下降到0%。

两个指标都有用，使用哪个取决于产品要求：pass@k适用于一次成功很重要的工具，pass^k适用于一致性必不可少的智能体。

从零到一：智能体伟大评估的路线图

本节阐述了我们从无评估到可以信任的评估的实用、经过现场测试的建议。将其视为评估驱动的智能体开发的路线图：尽早定义成功，清楚地衡量它，并持续迭代。

收集初始评估数据集的任务

步骤0.尽早开始

我们看到团队延迟构建评估，因为他们认为需要数百个任务。实际上，从真实失败中抽取的20-50个简单任务是一个很好的开始。毕竟，在早期智能体开发中，对系统的每次更改通常都有明确、明显的影响，这种大效应量意味着小样本量就足够了。更成熟的智能体可能需要更大、更难的评估来检测更小的效果，但在开始时最好采取80/20方法。等待越久，评估就越难构建。早期，产品需求自然转化为测试用例。等太久，你就在从实时系统反向工程成功标准。

步骤1.从你已经手动测试的内容开始

从你在开发期间运行的手动检查开始——你在每次发布前验证的行为和最终用户尝试的常见任务。如果你已经在生产中，查看你的bug跟踪器和支持队列。将用户报告的失败转化为测试用例确保你的套件反映实际使用；按用户影响优先排序帮助你在重要的地方投入努力。

步骤2：编写明确的任务和参考解决方案

正确把握任务质量比看起来更难。一个好的任务是两个领域专家会独立得出相同通过/失败裁定的任务。他们自己能通过任务吗？如果不能，任务需要细化。任务规格中的模糊性成为指标中的噪声。相同的适用于基于模型的评分器标准：模糊的rubrics产生不一致的判断。

每个任务应该可以被正确遵循指令的智能体完成。这可能很微妙。例如，审计Terminal-Bench时发现，如果任务要求智能体编写脚本但没有指定文件路径，并且测试假设脚本的特定文件路径，智能体可能由于自己的错误而失败。评分器检查的所有内容应该从任务描述中清楚；智能体不应该因为模糊的规格而失败。对于前沿模型，许多试验中0%通过率（ 即0% pass@100）最常是破碎任务的信号，而不是能力不足的智能体，是检查任务规格和评分器的信号。对于每个任务，创建参考解决方案是有用的：已知通过所有评分器的工作输出。这证明任务是可解决的，并验证评分器配置正确。

步骤3：构建平衡的问题集

测试行为应该发生的情况和不应该发生的情况。一方面评估产生一方面优化。 例如，如果你只测试智能体应该在什么时候搜索，你可能最终得到一个为几乎所有事情搜索的智能体。尝试避免类别不平衡评估。我们在为Claude.ai中的网络搜索构建评估时亲身学到了这一点。挑战是防止模型在不应该搜索时搜索，同时保留它在适当时候进行广泛研究的能力（如查找天气）。团队构建了覆盖两个方向的评估：模型应该搜索的查询（如查找天气）和应该从现有知识回答的查询（如"谁创立了Apple？"）。在触发不足（不应该搜索时搜索）和过度触发（应该搜索时搜索）之间找到正确的平衡是困难的，需要对提示和评估进行多轮完善。随着更多示例问题出现，我们继续添加评估以提高覆盖率。

设计评估harness和评分器

步骤4：使用稳定环境构建健壮的评估harness

评估中智能体的运行方式与生产中使用的智能体大致相同，并且环境本身不会引入进一步噪声是必不可少的。每次试验应该通过从干净环境开始来"隔离"。运行之间不必要的共享状态（剩余文件、缓存数据、资源耗尽）可能导致由于基础设施不稳定而不是智能体性能相关的相关失败。共享状态也可以人为地提高性能。例如，在一些内部评估中，我们观察到Claude通过检查先前试验的git历史在某些任务上获得不公平优势。如果多个不同试验由于相同的环境限制（如有限的CPU内存）而失败，这些试验不是独立的，因为它们受到相同因素影响，评估结果对于测量智能体性能变得不可靠。

步骤5：深思熟虑地设计评分器

如上所述，伟大的评估设计涉及为智能体和任务选择最佳评分器。我们建议尽可能选择确定性评分器，在必要时或为额外灵活性使用LLM评分器，并谨慎使用人类评分器进行额外验证。

有一种常见冲动检查智能体是否遵循非常具体的步骤，如正确顺序的工具调用序列。我们发现这种方法过于严格，结果产生过度脆弱的测试，因为智能体经常找到评估设计者没有预料到的有效方法。所以为了不不必要的惩罚创造力，评估智能体产生的内容通常比评估它采用的路径更好。

对于有多组件的任务，建立部分信用。正确识别问题并验证客户但未能处理退款的支援智能体比立即失败的智能体明显更好。在结果中表示这种成功连续体很重要。

模型评分通常需要仔细迭代以验证准确性。LLM-as-judge评分器应该与人类专家密切校准，以获得人类评分和模型评分之间几乎没有分歧的信心。为了避免幻觉，给LLM一个出路，比如提供指令在信息不足时返回"Unknown"。创建清晰、结构化的标准来评估任务的每个维度，然后使用隔离的LLM-as-judge对每个维度进行评分，而不是使用一个来评分所有维度，也可以提供帮助。一旦系统健壮，偶尔使用人类审查就足够了。

一些评估有微妙的失败模式，即使智能体表现良好也会导致低分，因为智能体由于评分bug、智能体harness约束或模糊性而无法解决任务。即使是复杂的团队也可能错过这些问题。例如，Opus 4.5最初在CORE-Bench上得分42%，直到一位Anthropic研究人员发现多个问题：严格评分在期望"96.124991…"时惩罚"96.12"、模糊的任务规格，以及不可能精确再现的随机任务。在修复bug并使用限制较少的scaffold后，Opus 4.5的分数跃升至95%。同样，METR在他们的时间地平线基准中发现了几配置错误的任务，这些任务要求智能体优化到所述分数阈值，但评分需要超过该阈值。这惩罚了遵循指令的Claude等模型，而忽略所述目标的模型获得更好的分数。仔细检查任务和评分器可以帮助避免这些问题。

使你的评分器能够抵抗绕过或hack。智能体不应该容易"欺骗"评估。任务和评分器应设计为通过需要真正解决问题而不是利用意想不到的漏洞。

长期维护和使用评估

步骤6：检查成绩单

除非你阅读许多试验的成绩单和分数，否则你不会知道你的评分器是否工作良好。在Anthropic，我们投资了查看评估成绩单的工具，我们定期花时间阅读它们。当任务失败时，成绩单告诉你智能体是犯了真正的错误还是你的评分器拒绝了有效解决方案。它还经常 surfaced关于智能体和评估行为的关键细节。

失败应该看起来公平：很清楚智能体哪里错了以及为什么。当分数不上升时，我们需要确信这是由于智能体性能而不是评估。阅读成绩单是你验证评估正在测量真正重要的东西的方式，是智能体开发的关键技能。

步骤7：监控能力评估饱和

100%的评估跟踪回归但不提供改进信号。当智能体通过所有可解决的任务时，评估饱和发生，没有改进空间。例如，SWE-Bench Verified分数今年开始为30%，前沿模型现在在>80%接近饱和。随着评估接近饱和，进步也会放缓，因为只剩下最困难的任务。这可能使结果具有欺骗性，因为大的能力改进看起来像分数的小增长。例如，代码审查初创公司Qodo最初对Opus 4.5印象深刻，因为他们的一次性编码评估没有捕获更长、更复杂任务的收益。作为回应，他们开发了一个新的智能体评估框架，提供了更清晰的进步图景。

作为规则，在我们深入评估细节并阅读一些成绩单之前，我们不重视评估分数。如果评分不公平，任务模糊，有效解决方案被惩罚，或harness约束模型，评估应该修改。

步骤8：通过开放贡献和维护长期保持评估套件健康

评估套件是一个需要持续关注和明确所有权的有生命的工件，才能保持有用。

在Anthropic，我们尝试了各种评估维护方法。证明最有效的是建立专用评估团队来拥有核心基础设施，而领域专家和产品团队贡献大多数评估任务并自己运行评估。

对于AI产品团队，拥有和迭代评估应该像维护单元测试一样常规。团队可能在"早期测试中工作"的AI功能上浪费数周，但未能满足未陈述的期望，一个设计良好的评估本可以早早发现。定义评估任务是压力测试产品需求是否足够具体以开始构建的最佳方式之一。

我们建议实践评估驱动开发：在智能体可以满足之前构建评估来定义计划能力，然后迭代直到智能体表现良好。在内部，我们经常构建今天"足够好"但对模型几个月能做什么的赌注的功能。通过率低的起始能力评估使这可见。当新模型发布时，运行套件快速揭示哪些赌注得到了回报。

离产品需求和用户最近的人最适合定义成功。使用当前模型能力，产品经理、客户成功经理或销售人员可以使用Claude Code贡献评估任务作为PR——让他们！或者，甚至更好，积极启用他们。

创建有效评估的过程。

评估如何适应理解智能体的整体方法

自动化评估可以针对数千个任务运行而无需部署到生产或影响真实用户。但这只是理解智能体性能的多种方式之一。完整图景包括生产监控、用户反馈、A/B测试、手动成绩单审查和系统性人工评估。

理解AI智能体性能的方法概述。

这些方法映射到智能体开发的不同阶段。自动化评估在发布前和CI/CD中特别有用，在每次智能体更改和模型升级时运行，作为防止质量问题的第一道防线。生产监控在发布后检测分布漂移和意外的真实世界失败。A/B测试在有足够流量时验证重大更改。用户反馈和成绩单审查是持续实践以填补空白：不断分类反馈，每周采样成绩单阅读，并在需要时深入研究。保留系统性人工研究用于校准LLM评分器或评估主观输出，在人类共识作为参考标准的地方。

就像安全工程中的瑞士奶酪模型，没有单一评估层捕获每个问题。通过组合多种方法，滑过一层的信息被另一层捕获。

最有效的团队结合这些方法：自动化评估用于快速迭代，生产监控用于ground truth，以及周期性人类审查用于校准。

结论

没有评估的团队陷入反应循环——修复一个失败，创建另一个，无法区分真实回归和噪声。早期投入的团队发现相反的情况：开发加速，因为失败成为测试用例，测试用例防止回归，指标取代猜测。评估给整个团队一座明确的山来攀登，将"智能体感觉更糟"转化为可操作的东西。价值在复合，但只有当你把评估视为核心组成部分而不是事后才想到的。

模式因智能体类型而异，但这里描述的基本原理是恒定的。尽早开始，不要等待完美的套件。从你看到的失败中获取现实任务。定义明确、稳健的成功标准。深思熟虑地设计评分器并结合多种类型。确保问题对模型来说足够难。迭代评估以提高其信噪比。阅读成绩单！

AI智能体评估仍然是一个新兴的、快速发展的领域。随着智能体承担更长的任务、在多智能体系统中协作，并处理越来越主观的工作，我们将需要调整我们的技术。我们会随着学习继续分享最佳实践。

附录：评估框架

几个开源和商业框架可以帮助团队实现智能体评估，而无需从头构建基础设施。正确的选择取决于你的智能体类型、现有堆栈，以及你是否需要离线评估、生产可观察性或两者。Harbor设计用于在容器化环境中运行智能体，基础设施用于跨云提供商运行大规模试验和标准化格式用于定义任务和评分器。像Terminal-Bench 2.0这样的流行基准通过Harbor注册表发布，使得运行已建立的基准以及自定义评估套件变得容易。Braintrust是一个平台，将离线评估与生产可观察性和实验跟踪相结合——适用于在开发期间需要迭代并在生产中监控质量的团队。其`autoevals`库包括用于事实性、相关性和其他常见维度的预构建评分器。LangSmith提供跟踪、离线在线评估和与LangChain生态系统紧密集成的数据集管理。Langfuse作为自托管开源替代方案提供类似功能，适用于有数据居住要求的团队。

Arize提供Phoenix，一个用于LLM跟踪、调试和离线或在线评估的开源平台，以及AX，一个扩展Phoenix用于规模、优化和监控的SaaS产品。许多团队组合多种工具、推出他们自己的评估框架，或者只使用简单的评估脚本作为起点。我们发现，虽然框架可以是加速进度和标准化的有价值方式，但它们只和通过它们运行的评估任务一样好。通常最好快速选择一个适合你工作流的框架，然后将你的精力投入到评估本身，通过迭代高质量测试用例和评分器。
