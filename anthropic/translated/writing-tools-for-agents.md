什么是工具？

在计算中，确定性系统给定相同输入每次都产生相同输出，而非确定性系统（如代理）在相同起始条件下可能产生不同响应。

当我们传统地编写软件时，我们是在确定性系统之间建立契约。例如，函数调用如getWeather("NYC")每次以完全相同的方式获取纽约市的天气。

```

工具是一种新型软件，反映了确定性系统和非确定性代理之间的契约。当用户问"今天我应该带伞吗？"时，代理可能调用天气工具、从一般知识回答，甚至首先提出澄清位置的问题。偶尔，代理可能会产生幻觉，甚至无法掌握如何使用工具。

这意味着在编写代理软件时从根本上重新思考我们的方法：不是像为其他开发者或系统编写函数和API那样编写工具和MCP服务器，我们需要为代理设计它们。

我们的目标是增加代理能够使用工具通过多种成功策略解决各种任务的有效表面面积。幸运的是，根据我们的经验，对代理最"人体工程学"的工具最终也出人意料地直观，人类也容易掌握。

如何编写工具

在本节中，我们描述如何与代理合作来编写和改进您给它们的工具。首先快速启动工具原型并在本地测试。然后运行全面评估以衡量后续更改。与代理一起，您可以重复评估和改进工具的过程，直到您的代理在现实任务上达到强劲性能。

与代理合作

您甚至可以让代理分析您的结果并为您改进工具。只需将评估代理的记录连接起来并粘贴到Claude Code中。Claude是分析记录和同时重构大量工具的专家——例如，确保在做出新更改时工具实现和描述保持一致。

事实上，这篇文章中的大部分建议都来自使用Claude Code重复优化我们的内部工具实现。我们的评估建立在内部工作空间之上，反映了内部工作流程的复杂性，包括真实项目、文档和消息。

我们依赖保留测试集来确保我们不会"过度适应"我们的"训练"评估。这些测试集显示，我们可以提取超越"专家"工具实现的额外性能改进——无论这些工具是由我们的研究人员手动编写还是由Claude自己生成的。

有效工具编写原则

在本节中，我们将我们的学习提炼为一些有效工具编写的指导原则。

为代理选择正确的工具

更多工具并不总是带来更好的结果。我们观察到的一个常见错误是工具仅包装现有软件功能或API端点——无论工具是否适合代理。这是因为代理与传统软件有不同的"启示"——也就是说，它们在使用这些工具时可以采取的潜在行动有不同的方式。

LLM代理具有有限的"上下文"（也就是说，它们一次可以处理的信息量有限），而计算机内存便宜且丰富。考虑在地址簿中搜索联系人的任务。传统软件程序可以高效地一次存储和处理一个联系人列表，在继续之前检查每个联系人。

但是，如果LLM代理使用返回所有联系人的工具，然后必须逐token阅读每个联系人，那是在浪费其有限的上下文空间（想象一下通过逐页从上到下阅读来搜索地址簿中的联系人——也就是说，通过暴力搜索）。更好且更自然的方法（对代理和人类都是）首先是跳到相关页面（也许按字母顺序找到它）。

我们建议构建一些深思熟虑的工具，针对特定的高影响工作流，与您的评估任务匹配，然后再从那里扩展。在地址簿情况下，您可能选择实现search_contacts或message_contact工具，而不是list_contacts工具。

确保您构建的每个工具都有清晰、独特的目的。工具应该让代理以与给定相同底层资源的人类相同的方式细分和解决任务，同时减少本来会被中间输出消耗的上下文。

命名空间您的工具

您的AI代理可能获得对数十个MCP服务器和数百种不同工具的访问权限——包括其他开发者的工具。当工具功能重叠或目的模糊时，代理可能会混淆应该使用哪些。

命名空间（将相关工具分组在公共前缀下）可以帮助在大量工具之间划定边界；MCP客户端有时默认执行此操作。例如，按服务（如asana_search、jira_search）和按资源（如asana_projects_search、asana_users_search）对工具进行命名空间，可以帮助代理在正确的时间选择正确的工具。

```

```

我们发现前缀和后缀命名空间的选择对我们的工具使用评估产生了 nontrivial 的影响。效果因LLM而异，我们鼓励您根据您自己的评估选择命名方案。

代理可能调用错误的工具、以错误的参数调用正确的工具、调用太少的工具，或错误地处理工具响应。通过选择性地实现其名称反映任务自然划分的工具，您同时减少了加载到代理上下文中的工具和工具描述的数量，并将代理计算从代理的上下文转移回工具调用本身。这降低了代理整体犯错的风险。

从工具返回有意义的上下文

同样，工具实现应注意仅向代理返回高信号信息。他们应该优先考虑上下文相关性而不是灵活性，并避免低级技术标识符（例如：uuid、256px_image_url、mime_type）。像name、image_url和file_type这样的字段更有可能直接为代理的下游行动和响应提供信息。

```

```

代理在处理自然语言名称、术语或标识符方面也比处理神秘标识符成功得多。我们发现，仅将任意字母数字UUID解析为更有语义意义和可解释的语言（甚至0索引ID方案）就能显著提高Claude在检索任务中的精确度，减少幻觉。

在某些情况下，代理可能需要同时与自然语言和技术标识符输出交互，如果仅是触发下游工具调用（例如search_user(name='jane') → send_message(id=12345)）。您可以通过在工具中公开简单的response_format枚举参数来启用两者，允许您的代理控制工具返回"详细"还是"简洁"响应（如下图）。

优化工具响应的token效率

优化上下文质量很重要。但优化工具响应返回给代理的上下文数量也是如此。

我们建议为任何可能消耗大量上下文的工具响应实施一些分页、范围选择、过滤和/或截断的组合，并带有合理的默认参数值。对于Claude Code，我们默认将工具响应限制为25,000个tokens。我们预计代理的有效上下文长度会随着时间增长，但对上下文高效工具的需求保持不变。

如果您选择截断响应，请务必用有用的说明引导代理。您可以直接鼓励代理追求更token高效的策略，例如对于知识检索任务进行多次小而有针对性的搜索，而不是一次 broad 搜索。同样，如果工具调用在输入验证期间引发错误，您可以提示设计您的错误响应以清楚地传达具体和可操作的改进，而不是模糊的错误代码或回溯。

提示工程您的工具描述

现在我们来看改进工具的最有效方法之一：提示工程您的工具描述和规范。因为这些被加载到代理的上下文中，它们可以共同引导代理朝向有效的工具调用行为。

在编写工具描述和规范时，想想您会如何向团队中的新员工描述您的工具。考虑您可能隐式带来的上下文——专业查询格式、 niche 术语定义、底层资源之间的关系——并使其明确。通过清晰描述（并用严格数据模型强制）预期输入和输出来避免歧义。特别是，输入参数应该明确命名：不要使用名为user的参数，而是尝试使用名为user_id的参数。

展望未来

要为代理构建有效的工具，我们需要将软件开发实践从可预测的、确定性的模式重新定向到非确定性的模式。

通过我们在本文中描述的迭代、评估驱动的方法，我们已经确定了成功工具的一致模式：有效的工具是深思熟虑且清晰定义的，谨慎使用代理上下文，可以组合在多样化的工作流中，并使代理能够直观地解决现实世界的任务。

在未来，我们期望代理与世界交互的具体机制会发展——从MCP协议的更新到底层LLM本身的升级。通过系统化、评估驱动的方法来改进代理工具，我们可以确保随着代理变得越来越有能力，它们使用的工具将随它们一起发展。
